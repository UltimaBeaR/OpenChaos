# CLAUDE.md — Urban Chaos Modernization Project

## Что это за проект

Модернизация игры **Urban Chaos (1999, MuckyFoot Productions)**.
Оригинальный исходный код находится в `original_game/` — он не изменяется, только читается.
Цель: написать новую версию игры на современном стеке, сохранив оригинальный геймплей 1:1.

## Структура репозитория

```
original_game/                      — оригинальный исходный код (READ ONLY, не трогать)
original_game_knowledge_base/       — детальная документация по оригинальной игре
  KB_INDEX.md                       — индекс KB: что читать и когда, карта подсистем, карта связей
  subsystems/                       — документация по каждой подсистеме
  resource_formats/                 — форматы файлов ресурсов
  overview.md, cut_features.md, ... — прочие файлы KB
original_game_knowledge_base_dense/ — компактная версия всей KB для полноконтекстных задач
  DENSE_SUMMARY.md                  — все подсистемы компактно (числа, формулы, зависимости)
  DENSE_SUMMARY_BUILD_GUIDE.md      — инструкция по пересборке summary
new_game_planning/                  — планы и решения по новой версии игры
new_game/                           — код новой версии игры (создаётся когда начнём писать)
CLAUDE.md                           — этот файл
```

## Мои инструкции (для восстановления контекста)

### При начале нового разговора:
1. Прочитать этот файл (CLAUDE.md)
2. Дальше — зависит от задачи:

| Задача | Что читать |
|--------|------------|
| Любое обращение к KB про игру (вопросы, анализ, планирование) | `original_game_knowledge_base_dense/DENSE_SUMMARY.md` — **читать вместо** subsystem файлов |
| Вопрос явно глубокий и явно про одну конкретную подсистему/формат | `subsystems/X.md` или `resource_formats/X.md` напрямую, DENSE не нужен |
| Планирование новой игры | `new_game_planning/README.md` |

**НЕ читать** subsystem файлы по умолчанию — DENSE_SUMMARY покрывает большинство вопросов.
**Не читать DENSE_SUMMARY проактивно** — только когда есть конкретный вопрос или задача про игру.

**Когда пересобирать DENSE_SUMMARY** (`original_game_knowledge_base_dense/DENSE_SUMMARY.md`):
- После значительных изменений в нескольких KB файлах
- Если DENSE_SUMMARY содержит устаревшие числа или факты
- После добавления новых подсистем в KB
- При мелких правках одного файла — проще обновить нужную секцию вручную
- Инструкция КАК пересобирать: `original_game_knowledge_base_dense/DENSE_SUMMARY_BUILD_GUIDE.md`

### Ключевые правила:
- `original_game/` — только читать. **Единственное исключение:** можно добавлять аналитические комментарии с префиксом `// claude-ai:` для облегчения навигации по коду. Такие комментарии легко отличить от оригинальных и легко найти через grep.
- Коммиты делает пользователь вручную — я никогда не коммичу сам
- Все важные решения и инфу записывать в документацию — контекст разговора теряется между сессиями
- Перед любым анализом кода сначала проверить есть ли уже документация по этой части
- Если задача требует сложного архитектурного reasoning или анализа запутанных взаимосвязей — попросить пользователя переключить на Opus. Сам переключить модель не могу.
- **Всё про этот проект писать внутри проекта** (CLAUDE.md или `original_game_knowledge_base/`) — **НИКОГДА не писать в memory файлы Claude**. Причина: пользователь коммитит файлы проекта и прыгает по веткам — контекст должен восстанавливаться исключительно из репозитория. Memory файлы не коммитятся. Использовать memory ТОЛЬКО для универсальных личных предпочтений пользователя, не привязанных ни к какому проекту.

### Правила оформления документации (`original_game_knowledge_base/`):
- **Размер файлов:** один файл = одна тема, цель ~100-350 строк. Если файл вырос за 400 строк — разбить.
- **README файлы** = только индекс со ссылками, без содержимого.
- **Имена файлов** — без нумерации (не `00_overview.md` — просто `overview.md`).
- **Дублирование** — не допускать: если тема описана в одном файле, другой файл только ссылается на него.
- **Куда что писать:**
  - Решения о переносе фич → `cut_features.md`
  - Архитектура оригинала → `overview.md` (только архитектура, без решений о переносе)
  - Форматы ресурсов → `resource_formats/` (отдельный файл на каждый формат)
  - Подсистемы → `subsystems/` (отдельный файл на каждую подсистему)
  - Компактная версия всей KB → `original_game_knowledge_base_dense/DENSE_SUMMARY.md`
  - Правила работы → этот файл (CLAUDE.md)
  - Архитектурные решения новой игры → `new_game_planning/`
- **После каждого важного решения** — сразу записывать в нужный файл. Не откладывать.

### ВАЖНО: Кодовая база — пре-релизная версия!
`original_game/` — это **НЕ финальный релиз**, а более ранняя пре-релизная версия кода.
- Финальная PS1/PC игра была доделана после заморозки этой кодовой базы
- Некоторые фичи в коде **закомментированы или недоделаны**, но в финальном релизе они **работают**
- Некоторые баги и артефакты рендеринга, видимые при запуске этой версии, в финале исправлены
- **Нельзя считать "закомментированный код = вырезанная фича"** — может быть просто незакоммиченная работа
- Эталон для "что есть в финальной игре" — **обе релизные версии: Steam PC И PS1**
- При сомнениях по поводу конкретной фичи — спрашивать пользователя (играл в финал на PS1, может проверить PC по лонгплеям)

### Язык:
- **Общение с пользователем:** русский
- **Мои заметки и внутренняя документация** (в `original_game_knowledge_base/`, `new_game_planning/`, CLAUDE.md): русский (английские термины и цитаты из кода — ок)
- **Публичная документация проекта** (README.md и подобные): английский (репозиторий открытый)
- **Комментарии в коде новой игры** (`new_game/`): английский (код для международной аудитории)

### Порядок работы:
1. **Фаза 1 (ЗАВЕРШЕНА):** Детальный анализ оригинального кода → `original_game_knowledge_base/`
2. **Фаза 2 (ТЕКУЩАЯ):** Планирование архитектуры новой игры → `new_game_planning/`
3. **Фаза 3:** Написание новой игры (папка будет создана позже)

### Workflow (экономия контекста):
- **Читать минимум файлов:** KB_INDEX.md + нужный subsystem файл покрывают большинство задач.
- **Для архитектурных задач:** DENSE_SUMMARY.md содержит всю KB компактно — не нужно грузить десятки subsystem файлов.
- **⚠️ KB не помещается в контекст целиком!** Суммарно ~40+ файлов по 100-350 строк. Читать только нужные подсистемы по одной. При чтении 5+ файлов KB контекст забивается и начинается компакция — данные теряются. Если нужно много подсистем — использовать DENSE_SUMMARY.md.
- **Пересборка DENSE_SUMMARY — только через subagent'ов!** См. `DENSE_SUMMARY_BUILD_GUIDE.md`. Никогда не читать всю KB в основном контексте для пересборки.
- **Subagents — только для механических задач:** grep, glob, поиск паттернов. Не для основного анализа.
- **Все решения записывать сразу** в нужный файл — контекст разговора теряется между сессиями.

## Принятые решения

### Стратегия
- Переписать с нуля, используя оригинал как документацию и эталон поведения
- Оригинальные ресурсы игры (из `original_game/fallen/Debug/`) переиспользуются как есть
- Поддержка PSX, Dreamcast, Editor — не нужны, выпиливаются
- Экспериментальный/нереализованный код в оригинале — не переносится
- **Переносим ВСЕ фичи из обеих релизных версий (PC + PS1).** Если фича есть в PC но нет в PS1 — переносим. Если конфликт реализаций — берём PC версию. Код с `#if !defined(PSX)` (PC-only) — переносится, если фича работает в финальной PC версии.
- Решения "не переносить" пересмотрены в итерации 7 — все в силе (MINE/THERMODROID/BIKE/hook/sewer/multiplayer — нет в финале ни на PC, ни на PS1)

### Что переносится 1:1
- Физика и коллизии (критично для геймплея)
- Система загрузки ресурсов (форматы файлов)
- Игровая логика и механики

### Что переписывается по-новому
- Графическая подсистема (новое API вместо DirectX)
- Архитектура движка (чёткое разделение движок / игра)
- Система модулей (C++23 modules вместо #include хедеров)
- Система сборки и кросс-платформенность

### Стек (предварительный, уточнить в Фазе 2)
- **Язык:** C++23 с модулями
- **Компилятор:** Clang (не MSVC — нужна кросс-платформенность)
- **Сборка:** CMake
- **Менеджер пакетов:** vcpkg
- **Окно/Ввод:** SDL3 (поддержка геймпада — обязательна на будущее)
- **Графика:** OpenGL 4.6 (пересмотреть в Фазе 2, возможно Vulkan)
- **Звук:** miniaudio или SDL_mixer (кросс-платформенно)
- **Скриптинг:** не нужен (MuckyBasic в оригинале не используется)
- **Платформы:** Windows, Linux, macOS

### Float precision (важно!)
Оригинал использует 32-bit float. В новой версии:
- Тип `float` в C++ = 32-bit IEEE 754 — ок
- **Проблема:** компиляторы могут использовать 80-bit extended precision (x87 FPU) или SIMD
- **Решение:** при конфигурации сборки добавить флаги `-mfpmath=sse -msse2` (GCC/Clang) для принудительного использования SSE, что даёт строго 32-bit результаты
- Это важно для воспроизведения физики оригинала

## Ресурсы игры

Находятся в **подпапках** `original_game/fallen/Debug/` — скопированы из Steam-версии.
Сам корень Debug/ содержит артефакты сборки (.obj, .exe, .dll и т.д.) — не ресурсы.
Папки с ресурсами: `Meshes/`, `Textures/`, `clumps/`, `data/`, `levels/`, `talk2/`, `text/`, `bink/`, `stars/`, `server/`
Также `config.ini` в корне Debug/ — конфиг игры.
Игра с этими ресурсами запускается и работает (проверено).
