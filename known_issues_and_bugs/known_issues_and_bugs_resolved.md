| DualSense adaptive trigger: rapid-tap shots without a click / click-without-shot during cooldown | **Fully fixed 2026-04-17** by a series of related changes. Evolution of the fix: (1) act-bit click detection — fire on the direct hardware 1→0 signal. Removed "shot without click" for slow taps. But for rapid taps the hardware motor has a natural rate limit ~300ms — clicks were lost in favor of the threshold fallback without tactile feedback. (2) A `fire_cooldown_ms` gate to mode=NONE after each shot — this not only removed "click without shot" during the post-fire window, but also **turned out to be the key to bypassing the ~300ms rate limit**: a brief reset of the effect to NONE forces the hardware motor to re-arm faster than its natural self-rearm. The final tuning `pistol.fire_cooldown_ms=100` gives a tactile click on EVERY shot at any tap rate, close to keyboard mashing rate. The important hardware discovery is recorded in [`dualsense_adaptive_trigger_facts.md`](../new_game_devlog/dualsense_libs_reference/dualsense_adaptive_trigger_facts.md) "The motor can be forced to reset by sending mode=NONE for ~100ms". Cooldown is tracked locally in `weapon_feel` (wall-clock) — Person::Timer1 was not used since it is unreliably implemented (decremented only in STATE_MOVEING + reused for unexpected purposes). Details and evolution in [`weapon_haptic_and_adaptive_trigger.md`](../new_game_devlog/weapon_haptic_and_adaptive_trigger.md). |

| non-Windows + DualSense USB: vibration barely works | **Fixed 2026-04-18:** `OUTPUT_REPORT_USB_BYTES` 74 → 48 in [`ds_device.cpp`](../libDualsense/ds_device.cpp) + the corresponding `wire_len` in [`ds_output.cpp`](../libDualsense/ds_output.cpp). Stage 5: on Windows VSync is implemented via `DwmFlush()` (compositor sync) instead of `SetSwapInterval(1)` — functionally identical for the player, but the compositor path doesn't trigger the WDDM throttle. Switched via `sdl3_gl_configure_vsync()` depending on OS + window mode (windowed/fullscreen): windowed on Windows → DwmFlush, fullscreen on Windows / non-Windows → ordinary driver VSync. Fullscreen must be borderless (in exclusive DwmFlush is a no-op), recorded in [stage12.md](stage12.md). ## macOS (resolved)

| Issue | Description |
|----------|----------|
| Very low FPS (feels like software rendering) | Cause: the alpha sort rendered each polygon with a separate GL draw call (350+ calls per frame). Fix: batched alpha sort — accumulating indices + one draw call per PolyPage. Frame time from ~80ms → ~5ms. |
| DualSense: huge input latency (~1 second) | Cause: the macOS IOKit HID backend (via SDL3 hidapi) returns reports from a FIFO queue (oldest first). DualSense over BT sends ~133 reports/sec, the game reads ~30/sec → the queue grew by ~100/sec → after a second we were reading second-old data. On Windows the HID backend behaves differently. Fix: drain the whole queue per frame, use the last report (ds_bridge.cpp `read()`). Also: a dirty check for LED/vibration output (gamepad.cpp), skip PlugAndPlay re-enumeration when a device is connected (ds_bridge.cpp). |
