| Zoo of `+0.5` / `-0.5` pixel compensations in the TL-vertex paths (D3D6 → OpenGL) | In the original D3D6 code there were no `0.5`s at all. It was closed with pointwise `+0.5`s in several places (`PolyPoint2D::SetSC`, `font_atlas`, `sky` stars, `gl_blit_textured_quad`, `ge_blit_to_framed_area`) + one `-0.5` in the `tl_vert.glsl` shader. Every time the bug showed up in a new place — yet another pointwise compensation was added. Currently the MM-path of characters and the outro 3D wireframe are WITHOUT compensation (a half-pixel shift relative to the world, invisible to the user). Architecturally it's a zoo: easy to forget in new code, easy to break by changing one place. A possible universal fix: remove the shader `-0.5` in sync with removing all manual `+0.5`s (mathematically identical to the current state), or: keep the current scheme but document the mandatory `+0.5` for new direct-TL code. Detailed analysis → [`../new_game_devlog/pixel_half_offset_plan.md`](../new_game_devlog/pixel_half_offset_plan.md). **Don't touch while ad-hoc** — too easy to break everything. Do it as a separate iteration with full inventory and tests. | Investigate (separate iteration) |

| Flickering of the fire/smoke order from near-batching (deliberately left for performance) | For fire/bonfires close to the camera, when the camera moves, **flickering** is noticeable — the fire and smoke swap draw order for a couple of frames every few steps. Cause: for the sake of FPS, the fire/smoke has **near-batching by depth layers** enabled — sorting is quantized into coarse layers, this cuts draw calls (near a burning car 332 micro-calls → ~20-30, GPU ≈ −2× on a scene with fire) and **actually solves the FPS drop**. Without batching the fire is drawn correctly but eats FPS. **The solution was deliberately left** — FPS is more important than this artifact. Improvement options: **(1) clustering by source** — all sprites of one fire source (bonfire, burning car, etc.) sorted as a single cluster at the depth of the source itself, not each sprite at its own depth. The source depth is world-anchored → moves smoothly with the camera → no jumps at layer boundaries → no flicker. To account for during implementation: both flame and smoke must be in the cluster. Smoke is particles (`PARTICLE_Draw`), the bonfire flame is immediate sprites; if you cluster only the flame, the smoke particles are sorted separately and creep over the fire. So a particle must know the world position of its source (remember it at spawn), and the sorting depth of the particle is computed from it. Within the cluster the fire should be drawn over the smoke. **(2) Atlas** — all the fire and all the smoke into one texture atlas → everything can be batched in one chunk, the very problem of "2 batches, smoke over fire or vice versa" disappears. **(3)** others are possible. **Where the current system sits in the code:** all in `poly.cpp` — the `s_sprite_layers` table, the `poly_alpha_key` function, the sort comparator inside `POLY_frame_draw`. The system is written generically (for any class of sprites), but apart from fire/smoke it's not used anywhere and probably won't be — if a better solution is found, it can be removed entirely. | Return to after 1.0 |

## Audit: FPS-dependent timers (render vs wall-clock)

| Task | Description | Status |
|--------|----------|--------|
| Run an audit of all time counters/timers for FPS dependency | **A bug class.** The original was render-locked (frame == tick, ~15–22 FPS), so many delays/timers in the code are written as "increment per frame" and were tuned to this fixed FPS. Our render is **unlocked** (200+ FPS) — any such counter that ticks in the render loop, not by the physics tick or wall-clock, "matures" as many times faster as the FPS is higher → delays collapse, animations/windows flash, events fire earlier than real time. **Already caught and fixed one by one:** the tutorial dialog grace ([game.cpp](../new_game/src/game/game.cpp)), menu animations/fades ([gamemenu.cpp](../new_game/src/ui/menus/gamemenu.cpp)), the Mission Completed overlay delay + voice wait (gamemenu.cpp, 2026-06-04), the `PYRO_FIREBOMB` explosion particle spawn ([pyro.cpp](../new_game/src/effects/combat/pyro.cpp)). These are scattered findings — **there was no systematic pass.** **Task:** have an agent go through the whole codebase, find all counters/timers/time accumulators (patterns `+= N` in the render path, `*_counter`, `*_timer`, `*_wait`, `*_delay`, dependencies on `TICK_RATIO`/`TICK_SHIFT` outside the physics tick, etc.), for each determine: (1) does it tick by render frame, by physics tick, or by wall-clock; (2) does it visually/gameplay-wise depend on FPS; (3) does it need to be moved to wall-clock (`sdl3_get_ticks()` / accumulating `millisecs`) or to the physics tick. Compile a prioritized list, fix in a batch. **Approach samples** — the already-completed fixes listed above. | Planned |
