**Не должны воспроизводиться в 1.0.** Все перечисленные краши давно не воспроизводились — возможно уже исправлены попутно с ASan-фиксами. Список сохранён чтобы не забыть и перепроверить при финальном тестировании.

| Проблема | Описание | Статус |
|----------|----------|--------|
| ASSERT при подтягивании на ящик | **Репро:** игрок подтягивается на ящик → ASSERT(0) в `person.cpp` `SUB_STATE_PULL_UP` handler (строки 9481 / 9489 — `if (face == GRAB_FLOOR)` ASSERT, либо `else` ветка при `face == 0`). Триггер не единичный ящик, а класс мест/геометрии. Актуально на 2026-04-19. | Рандомные краши во время геймплея | Значительно реже после ASan-фиксов (90%+ стабильна). Скорее всего несколько разных багов — memory corruption, uninit pointers, out-of-bounds. (2) **Texture cache miss / shader recompile**: после перезагрузки миссий какой-то ресурс выпадает из cache — каждый кадр на upload идёт + GPU stalls. (3) **Particle / dirt / projectile pool leak**: пул не сбрасывается полностью при load следующей миссии, остаётся "висеть" мусор который каждый кадр обрабатывается (см. также бaг "Много гильз на экране при стрельбе из AK47" — потенциально тот же класс leak'а). (4) **GL state divergence**: GL state machine оказывается в субоптимальном состоянии (включён лишний blend/depth path, лишние uniforms заливаются), но переинициализируется на process start. (5) **Какой-то per-frame consumer выходит на slow path** — например `there_is_a_los` для bloom/lens flares начинает делать значительно больше iterations из-за какого-то stale spatial structure (PCOM grid, OB list). **План действий:** (а) поймать момент следующего воспроизведения и сделать **CPU profile** (`perf-diag` skill — perf counter в render path); если CPU bound — понять какой системой; (б) сделать **CPU+GPU frame capture** (RenderDoc / GPUView) в bad state и в good state, diff. (в) добавить аккумуляторы `frame_alloc_count` / `frame_alloc_bytes` per frame чтобы поймать allocation-leak гипотезу. (г) добавить heap memory snapshot перед/после загрузки миссии — сравнить с good state (есть ли расхождение в peak / fragmentation). **Не воспроизводится надёжно** — расследование оппортунистическое: при следующем поимании не закрывать процесс, попробовать диагностику. **⚙️ Инструмент для отладки — сначала сделать per-stage frame breakdown** (задача из [`fps_unlock_issues.md #22`](../new_game_devlog/fps_unlock/fps_unlock_issues.md#22-)): встроенный debug-режим с таймерами на каждую стадию кадра (physics tick, AI/PCOM, AENG_draw, POLY render, post-process, UI, present) и экранным breakdown усреднённым по ~30-60 кадрам. При следующем поимании включить его и сразу видно какая стадия занимает лишние 40+ мс на кадр — без этого инструмента расследование вслепую. Реализация: `g_perf_diag_enabled` global, `perf_stage_begin/end` API, ring buffer avg, текстовая таблица в углу экрана. **Нужно в 1.0** — катастрофическая просадка убивает playability на whatever миссии оно хитнётся. | Исследовать (плавающий, perf) |
