## Принятые решения по стратегии

### Стратегия: итеративный рефакторинг оригинала

Оригинальный код копируется в `new_game/` и итеративно рефакторится по этапам.
К финалу от оригинального кода останется только физика и игровая логика —
всё остальное (рендерер, платформенный код, архитектура) будет заменено.
Оригинал в `original_game/` используется как эталон поведения и документация.

### Что переносится 1:1

- Физика и коллизии — критично для геймплея, изменения дадут другие ощущения
- Загрузка ресурсов — используем те же самые файлы из Steam-версии, форматы не меняем
- Игровая логика и механики

### Что переписывается по-новому

- Графическая подсистема (новое API)
- Архитектура движка (чёткое разделение движок / игра)
- Система модулей (C++20 modules, этап 9)
- Система сборки и кросс-платформенность

### Что не переносится

- Поддержка PSX и Dreamcast (`#ifdef PSX` / `TARGET_DC` блоки)
- Glide renderer (папка `Glide Engine/` в fallen/)
- Software renderer (`sw.cpp`)
- Редакторы карт/анимаций (`Editor/`, `GEdit/`, `LeEdit/`, `SeEdit/`)
- MuckyBasic — не используется в игре (вся миссионная логика через EWAY)
- Экспериментальный контент не вошедший в релиз (MINE, THERMODROID, BIKE, hook, sewer, multiplayer)

### Эталон: обе релизные версии

Переносятся **все фичи из обеих версий** — PC и PS1. Ни одна фича не выбрасывается только потому что её нет на одной из платформ.
Если фича есть только на PS1 — переносим. Если только на PC — переносим.
Если фича есть на обеих но реализована по-разному — берём PC реализацию как приоритетную.

## Архитектурные принципы

1. **Движок отделён от игры** — в оригинале всё перемешано, это исправляем
2. **Модули вместо хедеров** — C++20 modules (этап 9, после рефакторинга архитектуры)
3. **Явные зависимости** — никаких глобальных состояний без крайней необходимости
4. **Читаемость** — код понятен без комментариев, комментарии объясняют "почему"
5. **Кросс-платформенность** — никакого platform-specific кода в игровой логике

### Физика и логика не привязаны к FPS

Game loop строится на фиксированном timestep: физика и игровая логика обновляются
с фиксированным шагом, рендеринг — с любой частотой кадров.
Скорость рендеринга не влияет на скорость игры.

Оригинал жёстко зафиксирован на 30 FPS через spin-loop busy-wait (логика на 15 FPS),
с `TICK_RATIO` для компенсации просадок на медленных машинах.
Результат: на современном железе игра искусственно ограничена 30 FPS.
В новой версии — явный fixed-timestep loop: логика обновляется с фиксированным шагом,
рендеринг — с любой частотой, без busy-wait.

### Строгое разделение рендерера и game state

В оригинале рендерер мутирует game state (`drawxtra.cpp` изменяет ammo,
создаёт PYRO/SPARK эффекты прямо в draw pass). В новой версии — строгое разделение:
рендерер только **читает** state, никогда не пишет.
Конкретные места для рефакторинга при переходе: `drawxtra.cpp`, `PYRO_draw_armageddon()`.

## Технологический стек

### Язык: C++20

C++20 — основной стандарт. Modules (C++20) — на этапе 9 при рефакторинге архитектуры.

**Почему C++20, не C++23:** C++20 полностью поддерживается Clang 20 и покрывает все наши
потребности (concepts, ranges, coroutines, modules). C++23 мало что добавляет для игрового кода,
при менее стабильной поддержке. Modules на C++20 (не C++23) — стандарт modules один и тот же,
в C++23 только мелкие уточнения.

**Почему:** физика и логика портируются механически без концептуального перевода.
Лучшая нативная игровая экосистема. Modules уберут хедер-хаос (300+ файлов с #include).

**Не брать:** Rust (другая парадигма, болезненный порт C-логики), Zig (нестабильный), C#.

### Компилятор и сборка: Clang + CMake + vcpkg

**Почему Clang:** кросс-платформенный, лучшая поддержка C++20 modules среди компиляторов.
MSVC — привязка к Windows, не берём.

**Текущий статус (Этап 5.1):** `clang-cl` (Clang 20+) + CMake + Ninja. Переход с vcxproj завершён на Этапе 3.
C++20 стандарт указан в CMake (`CMAKE_CXX_STANDARD 20`). CMake проверяет что Clang >= 20.

#### Флаги компилятора (Release) — обоснование

| Флаг | Причина |
|------|---------|
| `-fno-strict-aliasing` | Оригинальный код активно использует type punning через указатели: `*(float*)&someInt`, `(char*)somePtr` и т.д. Clang с `-O2` имеет право оптимизировать такие места на основе strict aliasing rules, что может давать некорректные результаты. Флаг отключает этот класс оптимизаций. |
| `-fno-inline-functions` | Без этого флага Clang `-O2` ломает рендеринг body parts персонажей: конечности хаотично прыгают вокруг торса. Воспроизводится только в Release (`-O2`), в Debug (`-O0`) и при `-O1` не проявляется. Точная причина не установлена — предположительно Clang инлайнит `FIGURE_draw_prim_tween_person_only_just_set_matrix` в цикл по body parts и что-то ломает в порядке записи/чтения матриц через глобальный `g_matWorld`. TODO: расследовать при работе над рендерером. |

#### Флаги компилятора (оба конфига) — обоснование

| Флаг/настройка | Причина |
|----------------|---------|
| `StructMemberAlignment=1Byte` (`/Zp1`) | Оригинальный код использует плотную упаковку структур. Без этого паддинг нарушит совместимость с бинарными форматами ресурсов (`.gob`, `.ilf` и др.) и сетевыми пакетами. |

### Платформы: Windows, Linux, macOS — с первого дня

### Окно и ввод: SDL3

Нативная поддержка геймпада (важно для будущего). Кросс-платформенно.

### Float precision

**Проблема:** компиляторы могут использовать 80-bit extended FPU precision,
что даст другие результаты чем оригинал (32-bit float).

**Решение:** флаги компилятора `-mfpmath=sse -msse2` (принудительно SSE = строго 32-bit).
Добавить в CMakeLists.txt. Критично для физики — она портируется 1:1.

### Анимационная система: без библиотек

Оригинал использует vertex morphing (не скелетную анимацию): линейная интерполяция
между двумя наборами вершин (tween ∈ [0,256]). Реализуется вручную в шейдере или CPU.

### Скриптинг: не нужен

MuckyBasic в игре не используется. Вся миссионная логика реализована
через EWAY — data-driven система EventPoints (42 условия, 52 действия, бинарный .ucm формат).
Нужна только реализация EWAY движка на C++.

## Графика — открытый вопрос (решить перед Этапом 2)

Платформы: Windows, Linux, macOS — все три.

| API | Windows | Linux | macOS |
|-----|---------|-------|-------|
| OpenGL 4.6 | ✓ | ✓ | ✗ (deprecated, max 4.1) |
| OpenGL 4.1 | ✓ | ✓ | ✓ (deprecated, но работает) |
| Vulkan | ✓ | ✓ | через MoltenVK |

### OpenGL 4.1

**За:** меньше boilerplate, быстрее стартовать. 4.1 достаточно для воспроизведения
оригинала (Painter's Algorithm, vertex lighting, compute не нужен). Работает на macOS сейчас.

**Против:** deprecated на macOS с 2018 — Apple может удалить в любой момент.
Нет compute shaders. OpenGL в maintenance mode с 2017.

### Vulkan + MoltenVK

**За:** официальный преемник OpenGL от Khronos. Единый API на всех платформах
(MoltenVK транслирует в Metal). Future-proof. Используется в продакшене (Steam, Dota 2 и др.).
Если захочется улучшить графику — не нужно переписывать рендерер.

**Против:** значительно больше boilerplate. Сложнее в разработке и отладке.

## Звук — открытый вопрос (решить перед Этапом 3)

Оригинал использует Miles Sound System с 3D позиционным звуком (`MFX_play_xyz`).

### OpenAL Soft (предпочтительный)

Нативный 3D позиционный звук — прямой аналог MSS32. Зрелый, хорошо проверен в геймдеве.
Apple deprecated свой OpenAL, но OpenAL Soft (от kcat) решает это.

### miniaudio

Header-only, тривиальная интеграция. Но 3D spatializer менее зрелый чем OpenAL.
