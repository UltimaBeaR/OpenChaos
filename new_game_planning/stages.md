# Этапы разработки (Фаза 3)

## Этап 1 (ЗАВЕРШЁН) — Форматирование

Запуск авто-форматтера для приведения стилистики (табы/пробелы, переносы и т.д.) к единообразию.
После этого форматирование не трогается вручную до конца этапа 2.

**Критерий успеха:** код компилируется и работает идентично до форматирования.

## Этап 2 (ЗАВЕРШЁН) — Удаление мёртвого кода

**Цель:** максимально очистить код от ненужного мусора без изменения чего-либо — ни логики, ни имён, ни структуры. На выходе тот же код, только без балласта.

**Главное правило этапа: код переносится 1 в 1, без каких-либо изменений.**

Конкретно — запрещено:
- Переименовывать переменные, функции, типы, файлы
- Менять структуру файлов и папок
- Рефакторить что-либо (даже очевидное)
- Менять логику, порядок операций, алгоритмы

Разрешено:
- **Удалять** следующее и только следующее:
  - `#ifdef PSX` / `TARGET_DC` / `VERSION_GLIDE` / `EDITOR` ветки и весь код внутри
  - Папки редакторов: `Editor/`, `GEdit/`, `LeEdit/`, `SeEdit/` — **в последнюю очередь** (требуют анализа, GEdit.h включается в 43+ файлов без guard)
  - `Glide Engine/` — 3dfx Glide рендерер
  - `MuckyBasic/`, `thrust/` — не используются в игре
  - `PSXENG/` — PSX-специфичный рендеринг
  - `Command.cpp` — legacy, заменён PCOM в финале
  - `cam.cpp` — мёртвый (`#ifdef DOG_POO`), активна только `fc.cpp`
  - `#if 0` блоки — мёртвый закомментированный код
  - Любой другой код который **однозначно задокументирован** в базе знаний как неиспользуемый, экспериментальный, не вошедший в релиз или выключенный. Источники:
    - `original_game_knowledge_base/preprocessor_flags.md` — полная таблица всех `#ifdef` флагов с пометкой "нужен для PC" / "нет"
    - `original_game_knowledge_base/cut_features.md` — фичи не вошедшие в финал
    - `original_game_knowledge_base_dense/DENSE_SUMMARY.md` — быстрый обзор, упоминает заглушки и неиспользуемый код; при работе с конкретной подсистемой дополнительно читать полный subsystem файл из `original_game_knowledge_base/subsystems/`
- **⚠️ Система инклюдов в проекте кривая — никогда не предполагай, всегда проверяй:**
  - Инклюды нередко тянут лишнее неочевидным образом — файл может включать то, что по названию никак не предположишь
  - Флоу импортов не соответствует "каноническому" — нельзя угадать по имени файла что он включает и кто его включает
  - Перед любым действием с файлом/папкой — grep по всему проекту, не полагаться на интуицию

- **⚠️ ВАЖНО перед удалением любой папки или файла:** проверить `#include` из всего проекта — возможно, из удаляемого файла что-то включается без защитного `#ifdef`. Алгоритм:
  1. Найти все `#include` на файлы из удаляемой папки/файла **без** обёртки удаляемого `#ifdef`
  2. Если такие есть — раскопать что используется и зачем
  3. Если нужно для рантайма — вынести этот код, удалив из файла всё лишнее
  4. Только после этого удалять папку/файл целиком
- **Минимальные правки для компиляции** — только если удаление кода сломало синтаксис окружающего кода. Допустимо: убрать осиротевший `else`, закрыть незакрытую скобку, убрать `#endif` без `#ifdef`. Недопустимо: переписывать логику, менять структуру, добавлять что-то новое.
- **Комментарии оригинала** — удалять или оставлять на усмотрение
- **Свои комментарии** — добавлять без ограничений
- **⚠️ Отслеживание перемещений файлов — ОБЯЗАТЕЛЬНО:** когда файл копируется/перемещается из одного места в другое (например, хедер из Editor/ в Headers/):
  1. Добавить в начало файла: `// uc_orig_file: fallen/Editor/Headers/OrigName.h`
  2. Если перемещается только отдельная сущность (не весь файл) — рядом с объявлением: `// uc_orig: OldName (fallen/файл.cpp)`
  3. Записать в `new_game_planning/entity_mapping.md` в таблицу "Перемещения файлов"
  - Это нужно для понимания связей нового кода с оригиналом в будущих этапах
  - Подробнее про систему соответствий — см. Этап 4.3 (таблица entity_mapping.md заведена досрочно)

**Что сохраняется:** всё остальное без исключений — код трогается только в списке выше.

**Важные правила для агентов:**
- Код задокументирован в KB как неиспользуемый/экспериментальный/не в финале → удалять.
- Код выглядит мёртвым, но в KB ничего про него нет → не трогать, спросить пользователя.
- При любой неясности ("удалять это или нет?", "это минимальная правка или уже рефакторинг?") — спрашивать, не угадывать.

**Критерий успеха:** игра компилируется и запускается, можно пройти уровень.

   **Исключения — не трогать:**

### Пункт 2 — Удаление посторонних файлов и папок

Цель: убрать из репозитория всё что заведомо не относится к коду игры и не понадобится
в дальнейшей разработке (инсталляторы, дистрибутивы, сборочный мусор и т.п.).

**Примеры кандидатов на удаление:** `dxinstall/`, старые `.exe`/`.lib` дистрибутивы,
инсталляционные скрипты, временные файлы сборки, дубликаты ресурсов и т.п.

**Правила:**

- **Никогда не удалять без анализа.** Каждый файл/папка-кандидат сначала изучается.
- **Формат отчёта по каждому кандидату** (предоставляется пользователю перед удалением):
  - Что это за файл/папка и для чего предназначена
  - Используется ли из кода (includes, vcxproj, линковка)
  - Риски удаления
  - Вопрос: удаляем или нет?
- **Исключение:** если файл/папка очевидно нужна (исходный код игры, ресурсы, документация) — не спрашивать, пропустить.
- Решение принимает **пользователь**, не Клод.

## Правила работы на Этапе 2 (обязательно соблюдать!)

**Итерационность — главное правило работы:**
- Каждую итерацию пользователь запрашивает отдельно. Клод **не запускает следующую итерацию сам** — ждёт команды.
- После каждой итерации пользователь компилирует (Клод тоже компилирует самостоятельно после каждого изменения — чтобы убедиться что ничего не сломалось), делает smoke-тест, и если всё ок — коммитит. Только тогда следующая итерация.
- **КРИТИЧНО:** после любого автоматического изменения файлов (скрипт, массовый поиск-замена и т.д.) — СТОП. Не делать больше ничего. Дождаться компиляции от пользователя. Не фиксить "complex cases", не трогать файлы, не анализировать — ничего. Компиляция сначала.
- Каждая итерация = один логический кластер удалений (например: одна категория флагов, одна папка, один тип мёртвого кода)
- **Размер итерации:** примерно одна из следующих работ:
  - Удаление одной папки (например, Glide Engine + обновление vcxproj)
  - Удаление `#if 0` блоков по всему проекту
  - Удаление одного флага (например, только `#ifdef PSX`) по всему проекту
  - Удаление bike.cpp + всех `#ifdef BIKE` блоков
  - НЕ всё это сразу в одной итерации!

**Читать KB перед работой:**
- Перед началом каждой итерации читать соответствующий subsystem файл и/или раздел DENSE_SUMMARY

## Этап 3 — Переход на Clang ✅ ЗАВЕРШЁН (Шаг 1)

**Цель:** сменить компилятор с MSVC на Clang.

**Выполнено (Шаг 1 — ClangCL в vcxproj):**
- `PlatformToolset` переключён на `ClangCL` в обоих конфигурациях
- Исправлены ошибки компиляции Clang-специфичные:
  - `figure.cpp`: `def##name` → `def name` в макросе `ALIGNED_STATIC_ARRAY` (invalid token paste)
  - `Drive.cpp`: `const char*` → `(char*)` каст для строкового литерала
  - `GDisplay.cpp`, `frontend.cpp`: `MFFileHandle > 0` → `!= nullptr`
  - `Controls.cpp`: corrupted `'£'` (U+FFFD) → `(UBYTE)0xa3`
  - `frontend.cpp`: `{-1,...}` в UBYTE-поле → `{(UBYTE)-1,...}`
- Исправлен баг vcxproj: убраны 100 мусорных строк `/D /NODEFAULTLIB`, добавлены SDL2.lib/OpenAL32.lib в Debug
- Release флаги: `-O2 -fno-inline-functions -fno-strict-aliasing`
  - `-fno-inline-functions`: без него Clang `-O2` ломает рендеринг body parts персонажей
    (инлайнинг `FIGURE_draw_prim_tween_person_only_just_set_matrix` в цикл — причина неизвестна, TODO)
  - `-fno-strict-aliasing`: превентивно, код активно использует pointer type punning

**Шаг 2 (TODO):** vcxproj → CMakeLists.txt

**Критерий Шага 1:** проект компилируется ClangCL, Debug и Release запускаются, рендеринг корректный. ✅

## Этап 3.5 — Актуализация базы знаний

**Цель:** привести `original_game_knowledge_base/` и `DENSE_SUMMARY` в соответствие с реальностью, исправив ошибки и неточности, выявленные за время работы на Этапе 2.

**Контекст:** за время Этапа 2 (удаление мёртвого кода) было обнаружено множество расхождений между KB и реальным состоянием кода. Например: MFStdLib задокументирована как неиспользуемая, но на деле активно используется в игре. Подобных фактов накопилось достаточно, чтобы требовалась отдельная ревизия KB.

**Источник информации:** папка `new_game_devlog/` — там записывались все важные находки, решения и неожиданные факты. Особенно много информации в `new_game_devlog/stage2_log.md`. Нужно полностью прочитать все файлы девлога и извлечь оттуда факты, противоречащие текущей KB или дополняющие её.

**Что делать:**
1. Прочитать все файлы в `new_game_devlog/` целиком
2. Выписать все факты, которые расходятся с текущей KB или отсутствуют в ней
3. Обновить соответствующие файлы в `original_game_knowledge_base/subsystems/`, `preprocessor_flags.md`, `cut_features.md` и т.д.
4. **Проверка внутренней согласованности KB:** имея всю KB загруженной — проверить факты на явные несостыковки и ошибки (противоречия между файлами, устаревшие утверждения, неверные описания механик и т.д.). Если ошибка очевидна и можно исправить сразу — исправить. Если требуется дополнительный анализ по оригинальному коду — выписать в отдельный список вопросов для следующей итерации.
5. **Итерация точечного анализа:** по каждому вопросу из списка (п.4) провести отдельный анализ по кодовой базе `original_game/`, без необходимости грузить всю KB заново. Один вопрос — одна мини-итерация: изучить код, исправить KB.
6. Пересобрать `DENSE_SUMMARY` по инструкции из `DENSE_SUMMARY_BUILD_GUIDE.md`

**⚠️ Рекомендация по выполнению:** этот этап лучше выполнять на **Opus модели** (контекстное окно 1М токенов — в 5 раз больше Sonnet). Стратегия: загрузить в контекст **полную** версию KB (`original_game_knowledge_base/` — все subsystem файлы, не DENSE_SUMMARY) + весь девлог (`new_game_devlog/`) + при необходимости другие данные (например, `preprocessor_flags.md`, `cut_features.md`). Имея всё это "в голове" одновременно — править KB, видя и расхождения, и контекст. На Sonnet это физически невозможно — полная KB не помещается в 200К контекст.

**Критерий:** KB не содержит известных неточностей, все находки Этапа 2 отражены.

## Этап 4 — Модули, реструктуризация, переименования

**Цель:** современная структура кодовой базы, читаемые имена, C++23 modules.

Три шага внутри этапа (последовательно):

### Шаг 4.1 — C++23 modules

`#include` хедеры → C++23 modules. Никаких `.h` файлов.
Ожидаемы проблемы с макросами (не экспортируются из модулей) — фиксить по ходу.

### Шаг 4.2 — Реструктуризация файлов

Полная переработка структуры папок и разбивка файлов.
Цель: по структуре файлов понятно где что искать. Никаких монструозных файлов (Game.cpp 53K строк и т.д.).

Структура → `tech_and_architecture.md` (раздел "Структура проекта").

### Шаг 4.3 — Переименование сущностей

Переименование переменных, функций, типов в более удачные имена.

**Лог соответствий — обязателен:** при каждом переименовании вести файл
`new_game_planning/entity_mapping.md` в формате:

| Новое имя | Старое имя (оригинал) | Файл оригинала |
|-----------|-----------------------|----------------|
| ...       | ...                   | ...            |

Это референс для поиска информации о новой сущности в базе знаний по старому коду.

Дополнительно: рядом с объявлением переименованной сущности добавлять комментарий:
```cpp
// uc_orig: OldName (original_file.cpp)
```

Префикс `uc_orig:` уникален — легко грепается, не пересечётся с другими комментариями.

**Критерий этапа:** кодовая база компилируется с модулями, структура файлов отражает архитектуру, тесты (если уже есть) проходят.
