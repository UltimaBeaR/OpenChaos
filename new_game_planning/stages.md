# Этапы разработки (Фаза 3)

## Этап 1 (ТЕКУЩИЙ) — Форматирование

Запуск авто-форматтера для приведения стилистики (табы/пробелы, переносы и т.д.) к единообразию.
После этого форматирование не трогается вручную до конца этапа 2.

**Критерий успеха:** код компилируется и работает идентично до форматирования.

---

## Этап 2 — Удаление мёртвого кода

**Цель:** максимально очистить код от ненужного мусора без изменения чего-либо — ни логики, ни имён, ни структуры. На выходе тот же код, только без балласта.

**Главное правило этапа: код переносится 1 в 1, без каких-либо изменений.**

Конкретно — запрещено:
- Переименовывать переменные, функции, типы, файлы
- Менять структуру файлов и папок
- Рефакторить что-либо (даже очевидное)
- Менять логику, порядок операций, алгоритмы

Разрешено:
- **Удалять** следующее и только следующее:
  - `#ifdef PSX` / `TARGET_DC` / `VERSION_GLIDE` / `EDITOR` ветки и весь код внутри
  - Папки редакторов: `Editor/`, `GEdit/`, `LeEdit/`, `SeEdit/`
  - `Glide Engine/` — 3dfx Glide рендерер
  - `MuckyBasic/`, `thrust/` — не используются в игре
  - `PSXENG/` — PSX-специфичный рендеринг
  - `Command.cpp` — legacy, заменён PCOM в финале
  - `cam.cpp` — мёртвый (`#ifdef DOG_POO`), активна только `fc.cpp`
  - `#if 0` блоки — мёртвый закомментированный код
  - Любой другой код который **однозначно задокументирован** в базе знаний как неиспользуемый, экспериментальный, не вошедший в релиз или выключенный. Источники:
    - `original_game_knowledge_base/preprocessor_flags.md` — полная таблица всех `#ifdef` флагов с пометкой "нужен для PC" / "нет"
    - `original_game_knowledge_base/cut_features.md` — фичи не вошедшие в финал
    - `original_game_knowledge_base_dense/DENSE_SUMMARY.md` — быстрый обзор, упоминает заглушки и неиспользуемый код; при работе с конкретной подсистемой дополнительно читать полный subsystem файл из `original_game_knowledge_base/subsystems/`
- **Минимальные правки для компиляции** — только если удаление кода сломало синтаксис окружающего кода. Допустимо: убрать осиротевший `else`, закрыть незакрытую скобку, убрать `#endif` без `#ifdef`. Недопустимо: переписывать логику, менять структуру, добавлять что-то новое.
- **Комментарии оригинала** — удалять или оставлять на усмотрение
- **Свои комментарии** — добавлять без ограничений

**Что сохраняется:** всё остальное без исключений — код трогается только в списке выше.

**Важные правила для агентов:**
- Код задокументирован в KB как неиспользуемый/экспериментальный/не в финале → удалять.
- Код выглядит мёртвым, но в KB ничего про него нет → не трогать, спросить пользователя.
- При любой неясности ("удалять это или нет?", "это минимальная правка или уже рефакторинг?") — спрашивать, не угадывать.

**Критерий успеха:** игра компилируется и запускается, можно пройти уровень.

## Этап 3 — Переход на Clang

**Цель:** сменить компилятор с MSVC на Clang.

- Настроить сборку через Clang + CMake
- Фиксить MSVC-специфичные вещи которые Clang не примет: `__int64`, `__declspec`, нестандартные расширения и т.д.
- Поддержка модулей в MSVC хуже — этот этап обязателен перед переходом на C++23 modules.

**Критерий:** проект компилируется Clang'ом и игра запускается.

## Этап 4 — Модули, реструктуризация, переименования

**Цель:** современная структура кодовой базы, читаемые имена, C++23 modules.

Три шага внутри этапа (последовательно):

### Шаг 4.1 — C++23 modules

`#include` хедеры → C++23 modules. Никаких `.h` файлов.
Ожидаемы проблемы с макросами (не экспортируются из модулей) — фиксить по ходу.

### Шаг 4.2 — Реструктуризация файлов

Полная переработка структуры папок и разбивка файлов.
Цель: по структуре файлов понятно где что искать. Никаких монструозных файлов (Game.cpp 53K строк и т.д.).

Структура → `tech_and_architecture.md` (раздел "Структура проекта").

### Шаг 4.3 — Переименование сущностей

Переименование переменных, функций, типов в более удачные имена.

**Лог соответствий — обязателен:** при каждом переименовании вести файл
`new_game_planning/entity_mapping.md` в формате:

| Новое имя | Старое имя (оригинал) | Файл оригинала |
|-----------|-----------------------|----------------|
| ...       | ...                   | ...            |

Это референс для поиска информации о новой сущности в базе знаний по старому коду.

Дополнительно: рядом с объявлением переименованной сущности добавлять комментарий:
```cpp
// uc_orig: OldName (original_file.cpp)
```

Префикс `uc_orig:` уникален — легко грепается, не пересечётся с другими комментариями.

**Критерий этапа:** кодовая база компилируется с модулями, структура файлов отражает архитектуру, тесты (если уже есть) проходят.

## Этап 7 — Новый рендерер

**Цель:** заменить DirectX рендерер на OpenGL/Vulkan через абстракцию.

**Стратегия:**
- Ввести интерфейс `IRenderer` на границе DDEngine → DDManager
- DDEngine (bucket sort, pages, polygon submission) остаётся
- DDManager заменяется реализацией `IRenderer` на OpenGL или Vulkan
- Параллельно: перенос мутаций стейта из рендерера в update фазу
  (конкретные места: `drawxtra.cpp`, `PYRO_draw_armageddon()`)

**Выбор API:** OpenGL 4.1 vs Vulkan+MoltenVK — решить перед стартом этапа.
Подробно → `tech_and_architecture.md`

**Критерий:** тестовая инфраструктура из этапа 6 проходит без регрессий.

## Этап 8 — Кросс-платформа

**Цель:** убрать Windows-специфичный код.

- `WinMain` → SDL3 main
- Windows API вызовы → SDL3 / std::
- `DIManager.cpp` (DirectInput) → SDL3 input
- Звук: MSS32 → OpenAL Soft (или miniaudio)

**Критерий:** компилируется и работает на Linux и macOS.
