## Визуальные проблемы

| Проблема | Описание | Статус |
|----------|----------|--------|
| Экран брифинга не скрывается при загрузке миссии (вместо loading screen) | При загрузке миссии briefing screen остаётся нарисованным поверх — а loading screen **рисуется сзади и не виден**. В retail виден правильный loading screen при загрузке миссии, не briefing. В D3D-версии PieroZ этого бага нет — значит регрессия появилась при переходе на OpenGL или позже. **Воспроизводится стабильно**, не плавающий. | Исследовать |
| Радар (HUD) рисуется поверх loading screen при выходе из миссии | При выходе из миссии (на loading screen перед возвратом во frontend) **радар продолжает рисоваться поверх экрана загрузки**. Замечено 2026-05-15. Раньше этого не наблюдалось — вероятно регрессия. Тот же класс что баг «briefing поверх loading screen» — HUD-элемент не скрывается на mission-end transition. Скорее всего radar draw call где-то не gated за состоянием выхода из миссии. Проверить как HUD скрывается на mission-complete vs mission-exit. | Исследовать |
| HUD-таймер не пропадает на mission-complete экране | По завершению миссии с машинкой (тайм-челлендж) на mission-complete экране (полноэкранная штука с текстом "миссия завершена") **продолжает рисоваться элемент HUD'а с таймером** (например `00:46`), хотя остальной HUD корректно скрыт. **В PieroZ-версии весь HUD пропадает** — значит регрессия в нашей кодовой базе. Замечено 2026-05-02. Скорее всего timer drawing call где-то не gated за `mission_complete`-флагом, в отличие от остальных HUD-элементов. **Возможно уже было пофикшено** в одной из последних итераций — **первым делом перепроверить, действительно ли воспроизводится сейчас**; если нет — переносить в resolved. | Перепроверить |
| Короткоживущие визуальные эффекты на высоком FPS: муззл-флэш тусклый, мерцание звёзд почти не видно | **Общий класс багов** — эффекты живущие 1 рендер-кадр становятся тусклыми/незаметными когда FPS растёт. На 25 FPS (оригинальный lock) 1 кадр = ~40 мс — нормальное визуальное «вспышка», глаз воспринимает как яркую. На 100+ FPS 1 кадр = ~10 мс — либо «мигнул и исчез», либо в часть кадров монитора попадает другое состояние → усреднение → визуально нет эффекта. **Регрессия из коммита fps-unlock**. **Известные проявления:** **(1, приоритет для 1.0) Муззл-флэш** — пламя из ствола рисуется на 1 кадр. Воспроизведение: любая миссия с пистолетом/AK — вспышка еле видна; залочить на 25 FPS — возвращается. Сейчас триггерится через `DT_FLAG_GUNFLASH` на DrawTween, ставится и съедается на следующем кадре в `figure_draw_held_item`. **(2, less critical) Мерцание звёзд** — звёзды на ночном небе должны заметно мигать (проверено на релизной PC-версии). У нас мерцания почти нет. **Фикс (общий, направление):** anchor любого короткоживущего визуального триггера к **игровому времени** (милисекунды), а не к рендер-кадрам. Для муззл-флэша: ввести таймер на персонаже (или global per-character muzzle-flash counter), декрементируется по game time, держит флаг активным ~40-50 мс. Для звёзд: пересмотреть `sky.cpp` — переключение состояния звезды по game time с фиксированной частотой (например 12-25 Hz). **Приоритет: муззл-флэш обязательно (виден в любой стрельбе); звёзды если успеем в том же фиксе. Если не успеем — звёзды в post-1.0.** | Исследовать |
| CRT-шейдер (эмуляция монитора) не зависит от физического размера экрана | Есть шейдерный эффект CRT (эмуляция ЭЛТ-монитора) — сам по себе хороший. Но сейчас он завязан на **виртуальный** размер экрана, а должен — на **физический** (реальное разрешение / плотность пикселей дисплея). На дисплеях с разной плотностью пикселей (Steam Deck vs внешний 4K vs обычный 1080p) эффект выглядит по-разному масштабированным. **Сделать:** пробросить физический размер/DPI в CRT-шейдер и нормировать параметры эффекта по нему. **Fallback если не получится быстро:** отключить CRT-шейдер в релизе (он опциональный визуальный эффект, отсутствие не блокер). До фикса/решения шейдер не должен попасть в релиз в нынешнем виде. | Исследовать (или отключить) |

## Управление

| Проблема | Описание | Статус |
|----------|----------|--------|
| Бесконечный скролл меню (stick drift) | В главном меню иногда начинается непрерывное прокручивание пунктов вниз, как будто зажата клавиша. Нет deadzone для стиков геймпада в обработке меню. **Стабильно воспроизводится на Xbox контроллере** (сильный drift) — и в главном меню, и на экране выбора миссий (карта). На DualSense встречалось один раз. В геймплее проблемы нет — значит deadzone применяется только к игровому инпуту, но не к меню/карте. Референс при исследовании: в release_changes `NOISE_TOLERANCE` в `interfac.cpp` был изменён с 8192 (~25%) на 1024 (~3%) — возможно это значение используется и для меню; проверить не применяя к игровому вводу. | Исследовать |
| Геймпад: R3 не работает на свежеподключённом DualSense пока не нажмёшь Enter с клавиатуры | **Репро (2026-05-20, RTA):** загрузил миссию RTA → нажал Enter (смена оружия с клавиатуры) → подключил DualSense по Bluetooth → ввёл чит на оружие и взял его → нажал **R3 на геймпаде для смены оружия — не сработало**. Затем нажал Enter с клавиатуры — оружие сменилось. После этого R3 на геймпаде **начал работать**. То есть R3 был "глухим" до тех пор, пока **один раз** не сработала смена оружия с клавиатуры. **Класс:** связан с двумя другими записями про **рассинхрон input state при подключении/отключении геймпада в середине геймплея**. Здесь конкретно: при mid-game BT-connect геймпада какой-то контекст input'а (snapshot инвентаря для `CONTROLS_rot_inventory`, либо флаг "active device", либо edge-detect состояние R3) не инициализируется. Первый `CONTROLS_new_inventory`-вызов с клавиатуры пересчитывает state и "будит" gamepad-путь. **Гипотеза по чинить:** на event подключения нового HID-устройства принудительно сбрасывать input snapshots (inventory rotation, last-edge per-button state) — аналогично disconnect-сценарию. Типичный сценарий "сел играть с клавиатуры, потом подключил геймпад", в котором первое нажатие R3 не реагирует. | Исследовать |
| Геймпад: R3 не прокручивает инвентарь при подборе оружия с открытым popup | **Репро (2026-04-15):** игрок нажимает R3 → открывается popup выбора оружия. Пока popup открыт, игрок подбирает новое оружие с земли. Новое оружие появляется в popup, но **выбрать его R3-вращением нельзя** — ротация крутит только по старому списку (до подбора). Закрыть popup и открыть заново — новое оружие выбирается нормально. **Workaround:** расфокус окна + фокус обратно снимает залипание (аналогично багу с курком DualSense). **Причина (установлена):** снапшот списка оружий в `CONTROLS_rot_inventory` не пересчитывается когда инвентарь меняется во время открытого popup, и rising-edge R3 крутит по устаревшему списку. **Фикс направление:** пересчитывать snapshot при изменении инвентаря или закрывать popup на подборе оружия. Сценарий нечастый, но при попадании выглядит как поломанная фича. | Исследовать (есть репро) |
| Переключение DualSense-стик ↔ клавиатурный вперёд — перс залипает на «полушаге» | **Репро:** стик «вперёд» на DualSense → клавиатурный «вперёд» → снова стик вперёд на DualSense → снова клавиатурный вперёд → повторить несколько раз. В какой-то момент при очередном нажатии клавиатурного «вперёд» персонаж **не бежит**, а «медленно идёт» с **анимацией передвижения застывшей на полу-кадре**, без реального движения. Гипотеза: на переключении активного input device где-то остаётся флаг «движение активно» или счётчик скорости, и новое клавиатурное нажатие попадает в полусостояние (анимация включилась как walking frame, но velocity не применяется). | Исследовать |
| Outro не закрывается крестиком окна | Финальный outro (после прохождения игры) закрывается только по ESC; клик в крестик окна (X) игнорируется. Стандартное закрытие окна должно работать всегда. (См. также общую идею в [stage12.md](stage12.md): сделать единый и понятный обработчик X — желательно мгновенный выход независимо от состояния игры — главное меню / outro / геймплей.) | Исследовать |

| Переработка управления (геймпад + клавиатура + опции) | **Проектирование и детали:** [stage13_gamepad.md](stage13_gamepad.md). **Главная фича-задача 1.0 (кроме баг-фиксов).** **(A) Геймпад (левый стик):** сейчас невозможно (1) перекат в сторону, (2) прыжок назад, (3) движение пешком назад. Плюс иногда самопроизвольный side-jump (после климба). Когда персонаж прижался к стене (◯/circle hold) — на левом стике движение вдоль стены не работает, только d-pad/стрелки. Также при недожатом стике + кнопке прыжка срабатывает уклонение / прыжок назад **в странную сторону** (не туда, куда задавал стик — замечено 2026-05-22). **Решение:** ввести **L2 как модификатор «tactical»** — `L2 + стик в сторону` = перекат в нужную сторону (работает и в обычном режиме и в рукопашке); `L2 + стик назад` = движение пешком назад; `L2 + стик назад + ✕` = прыжок назад почти с места. Убрать автоматические перекаты при «движение + jump». L2 у перса сейчас свободна (управление машиной — отдельный input context); на Xbox L2=LT нужен однозначный порог срабатывания. **(A2) Полный перенос функционала D-pad на левый стик** — сейчас ряд действий (перекаты, ходьба назад, спуск задом по лестнице) работает только через D-pad и не срабатывает со стика. Цель: убрать необходимость использования D-pad для движения. D-pad перепрофилировать на смену оружия и т.п. — стандарт современных игр. **(A3) Кувырок вместо прыжка в сторону (action tree priorities):** при беге + резкий поворот стиком + прыжок персонаж делает кувырок (flip) вместо прыжка в сторону. Особенно мешает при активном паркуре. Пересмотреть приоритеты/пороги в action tree чтобы прыжок в сторону срабатывал предсказуемо, а кувырок — только при явном намерении. **(B) Клавиатура:** дефолты на **WASD**. Реализовать **движение камерой мышкой** (аналогично правому стику геймпада) — **только камера**; стрельба/прицеливание мышью **НЕ делаем в 1.0** (см. post-1.0 «Mouse shooting»). Сделать **движение как на стике** (направление в world-space, не tank). **(C) Настройки (Options):** добавить **переключалку «Tank controls» ↔ «Stick-like»** — общая для клавы и геймпада, при переключении применяется сразу к обоим устройствам. **(D) НЕ делаем в 1.0:** (1) развязка кнопки «Action» на отдельные действия (сейчас multi-action) — оригинальное поведение оставляем, играбельно; (2) стрельба/прицеливание мышью; (3) Steam Input API (единый API для всех контроллеров через Steam); (4) контекстные подсказки кнопок везде в игре. **(E) Финальная функциональная валидация (отдельно, ДО смоук-теста):** убедиться что **возможности клавиатуры и геймпада взаимно покрывают друг друга** — всё что можно сделать с клавы, можно сделать с геймпада, и наоборот. Оба способа игры — валидные. Также: **DualSense и Xbox по функционалу не должны отличаться** (одни и те же действия доступны на обоих). **Геймпадный чит** (комбинации стиков/кнопок) тоже должен быть доступен с клавы — другим механизмом (последовательность клавиш или комбинация), т.к. на клаве по аналогии не сработает. Эта валидация — **не часть смоук-теста** (тот будет на DualSense, как финальное прохождение игры); это отдельный чек ПЕРЕД смоук-тестом, по сути часть переработки управления. | Спроектировать и реализовать |
| Зависание клавиатурного ввода в меню при отключении DualSense по Bluetooth | **Репро:** играем с подключённым DualSense по Bluetooth → выключаем геймпад (либо BT-передатчик на геймпаде, либо BT на ПК / уводим геймпад из радиуса). Сразу после этого клавиатурный ввод в **главном меню** и в **меню паузы** начинает глючить. **Симптомы:** *главное меню* — бесконечное переключение пунктов вниз/вверх как будто залипла клавиша; клавиатура на ввод не реагирует (Enter/Esc не работают). *Меню паузы* — клавиатура полностью не реагирует. **Гипотеза:** при пропадании HID-устройства из enumeration где-то остаётся "залипшее" состояние стика/d-pad с предыдущего фрейма (последние values не сбрасываются на zero / center), и меню-навигация интерпретирует их как непрерывное нажатие. Плюс возможно keyboard input блокируется логикой "active device = gamepad" — последнее активное устройство остаётся gamepad даже когда оно физически исчезло. **Чинить:** на disconnect-event геймпада (1) явно занулять все stored stick/dpad values для этого устройства; (2) переключать "active device" на keyboard если активный был только что отключённый pad; (3) разблокировать keyboard input в меню. Типичный сценарий — игрок выключает геймпад чтобы перейти на клавиатуру, и при этом UI ломается. | Исследовать |

## Геймплей

| Проблема | Описание | Статус |
|----------|----------|--------|
| Много гильз на экране при стрельбе из AK47 (утечка — не удаляются) | При длинной очереди из AK47 на экране накапливается большое количество летящих/лежащих гильз (brass casings). Каждый выстрел AK47 эжектит **3 гильзы** за шот (`person.cpp actually_fire_gun`: `DIRT_create_brass` вызывается трижды для AK47 против 1 раза для пистолета) — соответственно плотность гильз в 3× больше чем у других оружий. **2026-04-19 подтверждение:** гильзы **никогда не исчезают** — за длинный игровой период их накапливается огромное количество, потенциальная причина тормозов. **Решение:** ограничить одновременное количество гильз на сцене (например ~200) — при превышении уничтожать самые старые (FIFO). Связано с багом «Подтормаживания при автоматической стрельбе NPC» (в post-1.0) — там NPC стреляет быстрее обычного и гильз становится катастрофически много. | Реализовать (FIFO лимит) |
| Машина сама тормозит при езде | Репро (2026-04-17, миссия 1): во время езды на машине она произвольно начинает тормозить, хотя игрок не жмёт тормоз. Рандом — условия пока непонятны.
