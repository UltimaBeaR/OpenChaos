# Таблица соответствий: оригинал → новый код

Этот файл ведётся с Этапа 2 (заранее, план был Этап 4.3).

**Зачем:** при поиске информации о сущности в KB по новому коду — здесь можно найти
как она называлась и где лежала в оригинале.

**Правило:** при любом перемещении файла, копировании хедера из другой папки или
переименовании сущности — обязательно добавить запись сюда И комментарий в коде:

| Случай | Комментарий в коде | Куда добавлять |
|--------|--------------------|----------------|
| Весь файл перемещён/скопирован | `// uc_orig_file: fallen/путь/к/оригиналу` | В начало файла |
| Отдельная сущность перемещена или переименована | `// uc_orig: OldName (fallen/файл.cpp)` | Рядом с объявлением |
| Сущность существовала в двух местах одновременно — дубликат удалён/очищен | `// uc_also_in: fallen/путь/к/файлу.h` | На канонической версии |

Пути — всегда относительно корня оригинальной игры (начиная с `fallen/`).
Все три варианта грепаются через `uc_orig` или `uc_also_in`; `uc_orig_file` — только файловые перемещения.

**`uc_also_in`** — используется когда одна и та же сущность существовала в двух местах одновременно
(часто из-за collision инклюд-гвардов в оригинале). Ставится на канонической версии,
указывает на место где дублирующее определение было **удалено или переименовано**. Пример:
```c
// uc_also_in: fallen/Headers/Engine.h (устаревшая копия удалена, DDEngine версия — каноническая)
typedef struct { ... } Camera;
```
