## Этап 4.0 — Перенос сущностей ✅ ЗАВЕРШЁН

Папки `src/old/` (текущий код) + `src/new/` (новая структура). Сущности переносились из `old/`
в `new/` итерационно, по одному логическому кластеру за раз. Код всегда компилируется.

Что сделано:
- Основной объём сущностей перенесён в новую иерархическую структуру
- На каждой сущности: `// uc_orig: Name (fallen/path/file.cpp)` — ссылка на оригинальный код
- Неиспользуемые сущности удалены (~2.5 МБ кода)
- `src/old/` удалена, `src/new/` распакована напрямую в `src/`
- Точка входа: `src/main.cpp` (тонкий враппер) → `HOST_run()` в `engine/graphics/graphics_api/host.cpp`

Что ещё осталось (этап не завершён):
- `src/fallen/` и `src/MFStdLib/` — старые заголовки вынесены сюда как временное решение,
  но это неправильно: все типы, константы, объявления из них должны быть частью новой
  структуры `src/`. Пока 319 `// Temporary:` инклудов ссылаются на эти папки.
- Все `// Temporary:` инклуды должны быть разрешены: типы перенесены в нормальные хедеры,
  после чего `src/fallen/` и `src/MFStdLib/` удаляются

**Критерий завершения 4.0:** `src/` содержит только папки новой структуры, никаких
`fallen/`, `MFStdLib/`, никаких `// Temporary:` инклудов.

## Этап 4.1 — Удаление мёртвого кода 🔄 ТЕКУЩИЙ

**Цель:** убрать весь неиспользуемый код который остался после переноса.

**Лог:** [`new_game_devlog/stage4_1_log.md`](../new_game_devlog/stage4_1_log.md)

### Правила этапа

- **Только удаления.** Не менять существующий код, не переименовывать, не реструктурировать.
  Допустимо: убрать осиротевшую ветку `if`/`else` после удаления внутренностей.
- **ASSERT'ы оставлять** — они документируют логику, это не мусор.
- **Код 1:1 с оригиналом** — удаляем только provably dead (не вызывается, `if(0)`, unreachable).
- **Билд после каждого существенного куска правок** — `make build-release`, проверка компиляции.
- **Рантайм-проверка + коммит** — после каждого крупного куска изменений (и на границах смены шагов)
  просить пользователя проверить что игра работает в рантайме и закоммитить.
- **Спрашивать пользователя перед удалением** — показать что найдено, получить подтверждение.
- Мёртвый пре-релизный код (фичи которые могут работать в финале) — удалять.
  Восстановим из `original_game/` когда дойдём до реализации.
- **Пустые файлы удалять** — если после удаления мёртвого кода файл больше не содержит
  определений, удалить файл целиком (+ из CMakeLists.txt, инклудов). Не оставлять болванки.

### Шаги

**Шаг 1 — Файлы целиком (сверху вниз, крупные куски)**
1. Анализ файлов по названиям + сверка с KB — какие подсистемы/файлы скорее всего не в финале
2. Показать пользователю список подозрительных → решение "используется / нет / не знаю"
3. Для помеченных "нет" — верификация грепом: все сущности из файла нигде не вызываются?
4. Удаление целых файлов + правка CMakeLists.txt + инклудов → билд

**Шаг 2 — Compiler warnings (инструментальный поиск)**
1. Временно включить в CMakeLists.txt:
   `-Wunused-variable`, `-Wunused-but-set-variable`, `-Wunused-function`, `-Wunreachable-code`
2. Собрать, собрать warnings, почистить по ним (ASSERT'ы не трогать)
3. Убрать флаги (слишком шумные для постоянного использования)
4. Билд

**Шаг 3 — Гранулярно: неиспользуемые сущности**
- Для каждого .cpp: извлечь все сущности (функции, переменные, типы, макросы)
- Grep по всему `src/` — если сущность нигде не используется → удалить
- Safety: grep как bare identifier (ловить function pointers), проверить макросы и dispatch tables
- По подсистемам от менее связанных к более связанным → билд после каждой

**Шаг 4 — Мёртвый код внутри функций**
- `if(0)` блоки (~10 штук) — удалить
- `/* */` закомментированные тела: если функция вызывается → оставить пустой stub,
  удалить закомментированный код. Если не вызывается → удалить целиком.
- Unreachable код после `return;` — удалить unreachable часть, ASSERT'ы оставить

**Шаг 5 — Orphaned declarations в хедерах**
- Найти в `.h` объявления без определений в `.cpp` (после удалений на шагах 3-4)
- Удалить orphaned declarations, extern variables, unused macros → билд

### Попутно

- Остатки `#ifdef` от Stage 2 (6 штук найдено: OLD_CAM, USE_A3D, PSX, TARGET_DC, _DEBUG,
  MUST_DOUBLE_CLICK) — вычистить по ходу работы с конкретными файлами
- Дубликат `bat.cpp` + `bat_globals.cpp` в CMakeLists.txt (строки 217-218 и 273-274) — убрать

**Результат:** только используемый код, нет «висящих» файлов, функций, переменных.

## Этап 4.2 — Доработка структуры ⏳

**Цель:** уточнить структуру на уровне папок и внутри файлов, без смены архитектуры.

Задачи:
- Ручная инспекция: что лежит не в своей папке → переместить
- Разбить файлы где смешаны сущности разного назначения
- Переместить отдельные сущности из одного файла в другой если там им логичнее
- Объединить файлы где это оправдано
- Обновление `#include` путей и `CMakeLists.txt` при перемещениях
- **Полный аудит соответствия правилам Этапа 4** — пройтись по всем файлам и проверить:
  - Глобальные переменные не сидят в основных `.cpp` файлах (должны быть в `_globals`)
  - `uc_orig` комментарии есть на всех сущностях
  - Нет нарушений DAG (включения вверх по слоям)
  - Инклуды оформлены по правилам
  - Известно что правило про глобалки нарушалось в ряде мест несмотря на строгие инструкции — найти и исправить

Известные неправильные расположения (найдено в процессе 4.0):
- `engine/graphics/graphics_api/wind_procs.h/.cpp` — Win32 WndProc, не относится к graphics API, переместить в `platform/`

Возможные дополнительные задачи (пока не решено):
- Выделить типы в отдельные файлы (`_types.h`) — отдельно от функций
- Выделить константы в отдельные файлы — отдельно от остального
- Решение по каждому из пунктов принимается в процессе осмотра

**Важно:** никакой смены реализации, никаких переименований — только перекомпоновка.
Архитектура и код остаются 1:1, меняется только где физически лежит каждая сущность.

Масштаб: значительно проще чем 4.0 — логику не трогаем вообще.

**Результат:** структура папок и файлов точно отражает назначение каждой сущности,
все файлы соответствуют правилам Этапа 4.
