# Этап 4 — Реструктуризация кодовой базы 🔄 ТЕКУЩИЙ

**Цель:** современная иерархическая структура файлов, сущности сгруппированы по смыслу,
каждая сущность трассируется к оригиналу через `uc_orig` комментарий.

**⚠️ Читать перед работой:** [`stage4_rules.md`](stage4_rules.md) — полные правила, критерии
что куда класть, формат комментариев, инструмент entity mapping.

## Подход

Папки `src/old/` (текущий код) + `src/new/` (новая структура). Сущности переносятся из `old/`
в `new/` итерационно, по одному логическому кластеру за раз. Код всегда компилируется.

## Что делается

- Сущности (функции, классы, типы, переменные) переносятся в новую иерархическую структуру папок
- На каждой сущности: `// uc_orig: Name (fallen/path/file.cpp)` — ссылка на оригинальный код
- Все глобалы — в файлах с постфиксом `_globals`
- Неиспользуемые сущности — удаляются, не переносятся
- Логика и имена сущностей — не меняются
- Маппинг ведётся в `entity_mapping.json` через `tools/entity_map.py`

## C++23 modules

После завершения переноса, отдельным шагом. Причина: сначала нужна готовая структура
для анализа DAG зависимостей.

## Критерий этапа

Все сущности из `old/` перенесены в `new/` (или удалены), `old/` пуст,
код компилируется и работает идентично.
