## Суть подхода

Создаём рядом с `src/` две папки:

```
new_game/src/
├── old/    ← весь текущий код переезжает сюда (fallen/ + MFStdLib/)
└── new/    ← новая структура, .h/.cpp с include guards
```

Из `old/` переносим сущности (функции, классы, типы, переменные) в файлы в `new/`.
`old/` подключает новые хедеры из `new/` вместо старых — всегда компилируемое состояние.
Точка входа (`Main.cpp`) остаётся в `old/` до самого конца.

**C++23 modules — пока не делать.** Обычные `.h/.cpp` с include guards. Конвертация в modules — отдельный этап после завершения структуры. После завершения переноса будет осмысленный DAG зависимостей по мелким файлам — вот тогда и конвертировать.

## Порядок действий при старте Этапа 4

0. **Анализ DAG зависимостей по старой структуре** — сделать до любых перемещений файлов.
   Цель: понять реальную картину зависимостей, скорректировать порядок миграции если нужно.
   - Пробежаться по `#include` во всех файлах `src/fallen/` + `src/MFStdLib/`
   - Найти файлы с минимумом зависимостей на другие файлы проекта (листья DAG) — они идут первыми
   - Найти неожиданные зависимости (файл который тянет что-то совсем неочевидное)
   - Скорректировать порядок миграции в разделе ниже если реальная картина расходится с планом
   - Результат: подтверждённый или обновлённый порядок миграции

1. Создать `tools/entity_map.py` + пустой `new_game_planning/entity_mapping.json`
2. Переместить `src/fallen/` и `src/MFStdLib/` в `src/old/`, обновить CMakeLists.txt — убедиться что компилируется
3. Добавить в CMakeLists.txt оба include directory:
   - `src/new/` — чтобы `old/` мог писать `#include "actors/core/thing.h"`
   - `src/old/` — чтобы `new/` мог включать ещё не перенесённые хедеры через `#include "fallen/Headers/Map.h"` (временная связь, уйдёт по мере миграции)
4. Начать перенос с `core/` — там нет зависимостей вверх

**⚠️ Не создавать пустые папки заранее** — папки создаются только когда в них появляется первый файл. Пустые папки не сохраняются в git и только захламляют.

## Порядок миграции (bottom-up по DAG зависимостей)

Мигрировать в таком порядке — каждый следующий слой зависит только от уже перенесённых:

```
1. core/                    — ни от чего не зависит
2. engine/io/               — зависит только от core
3. engine/graphics/         — зависит от core, io
4. engine/animation/        — зависит от core, graphics
5. engine/lighting/         — зависит от core, graphics
6. engine/effects/          — зависит от core, graphics
7. engine/physics/          — зависит от core
8. engine/audio/            — зависит от core, io
9. engine/input/            — зависит от core
10. assets/                 — зависит от engine/io/
11. world/                  — зависит от engine/*, assets
12. actors/                 — зависит от engine/*, world, assets
13. effects/ (game level)   — зависит от engine/effects/, actors
14. ai/                     — зависит от actors, world
15. missions/               — зависит от actors, world, ai, assets
16. ui/                     — зависит от actors, engine/graphics
```

Нарушение порядка — зависимость от ещё-не-перенесённого модуля. Если так вышло — либо перенести зависимость сначала, либо временно оставить нужное в `old/` и вернуться позже.

## Правила переноса

### Типовой поток миграции одного файла

1. Создать `new/path/file.h` + `new/path/file.cpp` с сущностями и `uc_orig` комментариями
2. Удалить перенесённые сущности из `old/` файла
3. Добавить `#include "path/file.h"` в соответствующий `old/` хедер
4. Скомпилировать — убедиться что всё работает
5. Добавить записи через `python tools/entity_map.py add`

### Разрешено
- Переносить сущности из `old/` в файлы `new/`
- Удалять неиспользуемые сущности и очевидный мусор — не переносить их
- Удалять ВСЕ оригинальные комментарии и писать новые (см. правила ниже)
- Добавлять `// uc_orig:` комментарии (обязательно на каждой сущности — см. ниже)

### Запрещено
- Изменять логику кода
- Переименовывать сущности (единственное исключение — конфликт имён, см. ниже)
- Изменять сигнатуры функций
- Объединять или разбивать функции

### Конфликты имён
Когда сущность с internal linkage (static / анонимный namespace внутри .cpp) становится публичной и конфликтует с другой — добавить префикс из имени файла-источника:

```
Формат:  <имя_файла_без_расширения>_<оригинальное_имя>

Примеры:
  thing.cpp:    static void helper()  →  thing_helper()
  vehicle.cpp:  static void helper()  →  vehicle_helper()
```

Добавлять только при реальном конфликте, не превентивно. Записать в entity mapping с `"conflict": true`.

### Пограничные сущности
Если сущность смешивает несколько областей и непонятно куда — кладём туда где наиболее правильно по ощущению. Tech debt, фиксируем комментарием рядом.

### Правила написания комментариев
- **Язык: английский.** Все комментарии в коде (`new/`) — только на английском.
- Удалять все оригинальные комментарии — они остаются в `old/` и `original_game/` как референс.
- Писать новые **для обычного разработчика** который видит код впервые. Не криптоязык терминов из KB — понятно что происходит.
- Комментарий объясняет "что делает" и "зачем" если неочевидно. Очевидный код не комментировать.
- Без префиксов `// claude-ai:` — в `new/` все комментарии и так написаны Claude.

### Include guards
Формат: `ПУТЬ_ОТ_NEW_ЗАГЛАВНЫМИ_H` — отражает путь файла от корня `new/`.

```cpp
// Файл: new/core/types.h
#ifndef CORE_TYPES_H
#define CORE_TYPES_H
...
#endif // CORE_TYPES_H

// Файл: new/engine/graphics/pipeline/bucket.h
#ifndef ENGINE_GRAPHICS_PIPELINE_BUCKET_H
#define ENGINE_GRAPHICS_PIPELINE_BUCKET_H
...
#endif // ENGINE_GRAPHICS_PIPELINE_BUCKET_H
```

### Опустевшие файлы в `old/`
Когда из файла перенесено всё содержимое — удалить файл. В лог не писать — при необходимости это находится grep'ом по `uc_orig` комментариям в `new/`.

## Правила для глобальных переменных

**Все глобальные переменные — только в файлах с постфиксом `_globals`** (`thing_globals.h`, `map_globals.h` и т.д.). Сразу видно по структуре файлов где живёт глобальный стейт.

Файл не обязательно должен быть "близнецом" другого файла — группировать как логически удобно. Правило одно: глобальная переменная → файл `*_globals.h`.

**Порядок инициализации:**
1. Тривиальный инициализатор (`= 0`, `= nullptr`, литерал) → переносить свободно
2. Нетривиальный инициализатор (вызов функции, конструктор с side effects) → переносить только вместе с зависимостями. Порядок объявления — как в оригинале.
3. Взаимозависимые глобалы → в одном файле, порядок — как в оригинале
4. Сомнительный случай → оставить в `old/`, экспортировать через `extern`, описать ситуацию и спросить

## Комментарий `uc_orig` — обязателен на каждой сущности

На **каждой** перенесённой сущности — комментарий прямо перед объявлением. Сущность = **любое именованное определение**: функция, класс, структура, тип, переменная, `#define` макрос (включая internal/helper макросы, используемые только внутри одного файла).

**⚠️⚠️⚠️ КОММЕНТАРИЙ СТАВИТСЯ И В `.h` И В `.cpp` ⚠️⚠️⚠️**

Если сущность имеет **декларацию в `.h`** и **определение в `.cpp`** — комментарий `uc_orig` ставится **в обоих местах**. Нет исключений. Определение функции в `.cpp` без `uc_orig` — это ошибка.

```cpp
// === file.h ===
// uc_orig: Thing_alloc (fallen/Source/Thing.cpp)
void Thing_alloc(int type);

// === file.cpp ===
// uc_orig: Thing_alloc (fallen/Source/Thing.cpp)
void Thing_alloc(int type) { ... }
```

Ещё примеры:

```cpp
// uc_orig: THING (fallen/Headers/Thing.h)
struct Thing { ... };

// uc_orig: g_thing_count (fallen/Source/Thing.cpp)
int g_thing_count = 0;
```

**Формат:** `// uc_orig: OriginalName (fallen/путь/к/файлу.ext)`

- Имя — как в `original_game/` (не в `old/`)
- Путь — от корня оригинальной игры (`fallen/...`)
- На отдельной строке перед объявлением/определением

**⚠️ БЕЗ ИСКЛЮЧЕНИЙ.** Каждая сущность перенесённая из оригинала получает `uc_orig` — включая:
- Тривиальные и общеизвестные (`TRUE`, `FALSE`, `min`, `max`)
- Internal/helper макросы внутри .cpp файлов (`#define LookUp ...`, `#define xchg ...`)
- Static функции, file-scope переменные, любые другие "вспомогательные" определения
- **Определения функций в `.cpp`** — даже если декларация в `.h` уже помечена

Неважно насколько "внутренней", "очевидной" или "вспомогательной" выглядит сущность — если она перенесена из оригинала, она получает `uc_orig` и запись в entity mapping. Никаких решений "это слишком мелкое/внутреннее, не буду помечать".

**Зачем:** при работе над рендерером/AI/эффектами в будущих этапах — быстро найти аналог в `original_game/` прямо из текущей строки. Особенно важно для фич под `#ifdef` которые удалили — они только в оригинале.

## Entity Mapping — инструмент

**Файл:** `new_game_planning/entity_mapping.json`
**Скрипт:** `tools/entity_map.py`

> **LEGACY:** `new_game_planning/entity_mapping.md` — старый файл с Этапа 2 (human-readable, не используется как инструмент).
> Данные из него **уже перенесены** в JSON (stage 2 записи). `.md` оставлен как исторический документ.

### Типы записей (--kind)

| kind | Что означает | `file` поле | `stage` |
|------|-------------|-------------|---------|
| `file` | Файловый переезд: файл в `old/` пришёл с другого пути в `original_game/` | путь в `old/` | 2 |
| `function` | Функция перенесена из `old/` в `new/` | путь в `new/` | 4 |
| `struct` | Структура перенесена | путь в `new/` | 4 |
| `class` | Класс перенесён | путь в `new/` | 4 |
| `variable` | Глобальная переменная перенесена | путь в `new/` | 4 |
| `type` | Typedef/псевдоним типа перенесён | путь в `new/` | 4 |
| `macro` | #define макрос перенесён | путь в `new/` | 4 |

Поле `orig_file` — **всегда путь в `original_game/`**, не в `old/` и не в `new/`.
Правило простое: путь должен существовать в `original_game/`, а не в `new_game/`.

### Workflow: перед каждым add — сначала find

```
python tools/entity_map.py find --name ENTITY_NAME
```

**Результат `find` → что делать:**

- **Нет результатов** → записи нет. Используй путь файла в `original_game/` как `--orig-file`.
- **Есть `kind: file` запись (stage 2)** → файл в `old/` пришёл с другого пути в `original_game/`.
  Используй `orig_file` из этой записи как `--orig-file` для новых entity-записей.
  Например: `anim.h` пришёл из `fallen/Editor/Headers/Anim.h` → все сущности из него
  получают `--orig-file "fallen/Editor/Headers/Anim.h"`, не `fallen/Headers/anim.h`.
- **Есть entity запись (stage 4)** → уже перенесено. Не добавлять дубликат.

### Формат JSON

```json
{
  "entries": [
    {
      "name": "Thing_alloc",
      "file": "new/actors/core/thing.h",
      "orig_name": "Thing_alloc",
      "orig_file": "fallen/Source/Thing.cpp",
      "kind": "function",
      "conflict": false,
      "stage": 4,
      "date": "2026-03-20"
    }
  ]
}
```

### Команды `entity_map.py`

```
python tools/entity_map.py add    --name NAME --file FILE --orig-name ORIG --orig-file ORIG_FILE --kind KIND [--conflict] [--stage N]
python tools/entity_map.py find   --name NAME            # поиск по name и orig_name (substring)
python tools/entity_map.py list   [--kind KIND] [--stage N]
python tools/entity_map.py stats
python tools/entity_map.py rename --name OLD --new-name NEW   # при переименовании в будущем
```

## Критерии: что куда относить

### Движок (`engine/`) — критерий

> **"КАК"** — алгоритм, механизм, сервис. Низкоуровневый насколько возможно.
> Не знает о персонажах, предметах, механиках UC. Переиспользуем в другой игре.

Граница на примере эффектов:
- `engine/effects/` — PSys инфраструктура, billboard rendering → **движок**
- `effects/` (game) — fire.cpp, pyro.cpp, конкретный огонь UC → **игра**

### Графический движок (`engine/graphics/`) — критерий

> **"КАК отрисовать геометрию и текстуры."**
> Знает о вершинах, полигонах, текстурах. Не знает что такое "коп" или "огонь".

### Graphics API (`engine/graphics/graphics_api/`) — критерий

> Тончайшая обёртка над DirectX.
> При замене DirectX на OpenGL/Vulkan — меняется **только этот слой**.

### File I/O (`engine/io/`) — критерий

> Сырой файловый I/O: открыть файл, прочитать байты, буферизация.
> Не знает о форматах файлов. Если функция просто читает байты с диска → она здесь.

### Asset loaders (`assets/`) — критерий

> UC-специфичные парсеры форматов: знает о структуре .gob, .ilf, .ucm, анимационных файлов.
> Использует `engine/io/` для чтения, сам занимается только разбором формата.
> Все места где код знает о конкретном бинарном формате UC — только здесь.
> Удобно: чтобы найти всё связанное с форматами ассетов — grep по `assets/`.

### Игра (всё остальное) — критерий

> **"ЧТО делает X в Urban Chaos."**
> Реализует механики через комбинирование функций движка.

### Порядок зависимостей (только вниз)
```
core → io → graphics_api → graphics → engine/* → world/actors → ai → missions/ui
```
Зависимость снаружи `engine/` напрямую в `engine/graphics/` — видимое нарушение архитектуры.

## Правила итерационности

1. **Размер итерации — по размеру файла, цель ~130 итераций всего:**

   | Размер файла | Правило | Примеры |
   |---|---|---|
   | Маленький (<300 строк) | Батч 3–5 тематически близких файлов | balloon+bat+pigeon, barrel+dike+drip |
   | Средний (300–3000 строк) | Один .cpp + его .h | vehicle.cpp, cop.cpp, fire.cpp |
   | Большой (>3000 строк) | Нарезать по ~2500 строк | game.cpp, person.cpp, thing.cpp |

   Ориентир, не жёсткое правило. Однотипные константы/дефайны — брать больше за раз.

2. **Компиляция** — Claude компилирует самостоятельно (`make build-release`) после каждой итерации. Пользователя не беспокоить — только сообщать результат.

3. **Smoke test** — пользователь запускает игру каждые 10–20 итераций. Claude не просит чаще.

4. **Самопроверка по диффу перед коммитом** — `git diff` + `git status`, каждый файл. Проверять:
   - **Корректность кода:** логика 1:1, нет dangling if/else, ничего не сломано в рантайме
   - **⚠️ Полнота `uc_orig` комментариев — ГЛАВНАЯ ПРОВЕРКА:**
     1. Открыть **каждый** новый/изменённый файл в `new/`
     2. Пройтись по **каждой** сущности (функция, тип, переменная, макрос, структура)
     3. Убедиться что **перед каждой** стоит `// uc_orig:` с правильным именем и путём

**Не грузить KB проактивно** — для большинства переносов она не нужна.

- DENSE_SUMMARY при переносе не грузить — только для архитектурных вопросов

## Предупреждения

- **Инклюды в `old/` нелинейные** — перед переносом любой сущности grep кто её использует. Нельзя угадывать по имени файла.
- **Dangling if/else** — при удалении строки проверять что она не единственное тело `if` без скобок.
- **`/Zp1`** (struct member alignment) — не менять, критично для бинарных форматов ресурсов.
- **`-fno-inline-functions`** — не убирать до выяснения бага с body parts персонажей.
- Сложный архитектурный анализ → просить переключить на Opus модель.
