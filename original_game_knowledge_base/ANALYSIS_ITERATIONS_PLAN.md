# План дальнейших итераций анализа

Согласован с пользователем. Порядок итераций фиксирован.
Каждая итерация = 1+ сессий (зависит от объёма и контекстного окна ~200K токенов).

---

## Итерация 1: Очистка KB

**Цель:** убрать шум, мёртвую документацию, разделить факты и решения.

### Задачи:
1. **Удалить `muckybasic.md`** (409 строк) — скриптовый язык не используется в игре
2. **Создать `analysis_scope.md`** — единственный файл в KB со ссылками на решения:
   - Системы не анализируемые потому что переписываем с нуля (рендеринг, DirectX враппер, оконная система)
   - Код не в релизе (мотоцикл/BIKE, канализация/sewers, гарпун)
   - Код не используемый в игре (MuckyBasic скрипты)
   - Платформо-специфичный ненужный код (PSX рендеринг, Dreamcast, Editor) — кроме PSX controls
   - Для каждого пункта — причина почему не анализируем
   - **Учесть `cut_features.md`:** там уже есть списки вырезанных фич, решения о переносе и неиспользуемые объекты. Часть этой инфы пересекается с `analysis_scope.md`. При создании — решить: либо объединить в один файл (если получится компактно), либо чётко разграничить: `cut_features.md` = факты о статусе фич в оригинале (что вырезано, что закомментировано), `analysis_scope.md` = наши решения что анализировать/не анализировать и почему. Убрать дублирование, ссылки друг на друга где нужно. Также вычистить из `cut_features.md` решения по новой игре (секция "Решения о принятых изменениях" — перенести в `new_game_planning/`)
3. **Вычистить решения по новой игре из всех KB файлов** — фразы типа "в новой игре заменим на miniaudio", "для OpenGL делить на 64.0f" и т.п. Перенести в `new_game_planning/`.
4. **Вычистить SESSION_START.md** — убрать решения, оставить только факты об оригинале + ссылки на `analysis_scope.md` для контекста

### Ожидаемый результат:
- KB = чистый справочник по оригинальной игре
- Единственная точка контакта с решениями = `analysis_scope.md` (компактный)
- Минус ~500+ строк мусора

## Итерация 2: Аудит аннотаций в коде (`// claude-ai:`)

**Цель:** все комментарии `// claude-ai:` в оригинальном коде корректны и на своих местах.

### Задачи:
1. **grep все `// claude-ai:` комментарии** — собрать полный список
2. **Для каждого комментария проверить:**
   - Стоит ли в правильном месте (относится к коду ниже/выше него)
   - Матчится ли с текущей документацией в KB
   - Нет ли фактических ошибок
3. **Исправить или удалить** некорректные комментарии
4. **Отметить расхождения** с KB → входные данные для итерации 3

### Технический подход:
- Субагенты: grep для сбора всех аннотаций, группировка по файлам
- Основной контекст: точечная проверка каждой аннотации vs код вокруг неё
- Разбить на батчи по файлам если много

## Итерация 3: Консистентность KB

**Цель:** устранить противоречия, дублирование и рассинхроны между KB файлами.

### Задачи:
1. **Перекрёстная проверка фактов** — один и тот же факт упомянут в нескольких файлах? Совпадают ли значения?
2. **Устранить дублирование** — факт должен быть описан в одном месте, остальные ссылаются
3. **Проверить ссылки** между файлами — не битые ли, актуальны ли
4. **SESSION_START.md vs subsystem файлы** — факты в SESSION_START совпадают с полными описаниями?

### Разбивка по подсистемам (по приоритету):
- Физика + коллизии (критично)
- AI/PCOM
- Персонажи/анимации
- Навигация
- Остальные

## Итерация 4: Верификация фактов по коду

**Цель:** каждая цифра, формула и логический вывод в KB проверена по реальному коду.

### Задачи:
1. **Damage values** — все значения урона (пистолет=70, шотган=300-dist, AK47=100/40, fall damage формула и т.д.)
2. **Physics constants** — gravity, damping, friction формулы, velocity limits
3. **Struct sizes** — sizeof каждой структуры (PrimFace3=28, PrimFace4=34, ED_Light=20 и т.д.)
4. **Формулы** — hit chance, vehicle friction bit-shift, fall damage threshold и т.д.
5. **Логические выводы** — "мёртвый код", "не используется", "всегда TRUE" — перепроверить каждый
6. **Таймеры и пороги** — fire rates, respawn timers, detection radii

### Технический подход:
- Для каждого факта: grep/read по коду → сравнить с KB → исправить если расхождение
- Субагенты для механического grep, основной контекст для анализа
- Разбить по подсистемам, начать с физики (самое критичное для 1:1 переноса)
- **ВАЖНО:** при 200K контекста — одна подсистема за сессию максимум

## Итерация 5: Скрытая игровая логика в editor/renderer файлах

**Цель:** найти game state mutations в файлах которые выглядят как "не игровые".

### Задачи:
1. **Сканировать editor файлы** — grep по паттернам мутации game state (Thing, Person, health, score, flags, etc.)
2. **Сканировать renderer файлы** — аналогично (drawxtra.cpp уже показал этот паттерн)
3. **Сканировать platform-specific файлы** — WIN95/DDEngine/ на предмет логики
4. **Документировать найденное** — добавить в соответствующие KB файлы

### Паттерны для поиска:
- Присваивание в game state переменные из файлов рендеринга/редактора
- Вызовы PCOM/AI/physics функций из неожиданных мест
- `#if !defined(PSX)` блоки с логикой (не только рендерингом)

## Итерация 8: Реорганизация KB

**Цель:** структурировать KB для минимального контекста при работе над конкретной подсистемой.

### Задачи:
1. **Разбить крупные файлы** — ai.md (832 строки!), missions.md (644), physics.md (531) и т.д. Цель: ~100-350 строк на файл
2. **Устранить перекрёстные зависимости** — файл про физику не должен содержать детали AI, и наоборот. Общие точки контакта — через ссылки.
3. **Обновить SESSION_START.md** — компактный индекс, ссылки на файлы, карта зависимостей
4. **Протестировать:** для типичной задачи "реализовать подсистему X" — сколько файлов и токенов нужно загрузить?

### Целевая метрика:
- SESSION_START.md < 15K токенов
- Любая подсистема полностью: SESSION_START + 1-3 файла < 40-50K токенов
- Вся KB целиком < 150K токенов (после очистки)

## Итерация 9: Dense Summary + инструкция по пересборке

**Цель:** создать компактную версию всей KB для задач требующих полного контекста (архитектурные решения, поиск скрытых зависимостей, ответы на сложные вопросы по всей игре).

### Задачи:
1. **Создать `DENSE_SUMMARY.md`** — сверхкомпактная версия всей KB:
   - Все подсистемы с ключевыми цифрами, формулами, константами
   - Межсистемные связи и неочевидные зависимости
   - Критичные факты которые легко забыть
   - **НЕ** развёрнутые описания — только суть
   - Целевой размер: ~30-40K токенов (влезает в контекст с запасом на работу)
2. **Вычистить SESSION_START.md** — убрать из него всё что дублируется с DENSE_SUMMARY. SESSION_START остаётся как лёгкий индекс + инструкция старта сессии, DENSE_SUMMARY — как полный контекст.
3. **Создать `DENSE_SUMMARY_BUILD_GUIDE.md`** — пошаговая инструкция как пересобрать dense summary с нуля:
   - Какие файлы KB читать и в каком порядке
   - Что из каждого файла извлекать в summary
   - Формат и структура summary
   - Целевой размер и как проверить что не вышли за лимит
   - Когда пересобирать (после значимых изменений в KB)

### Когда грузить DENSE_SUMMARY:
- Архитектурные обсуждения и принятие решений
- Вопросы затрагивающие несколько подсистем
- Поиск скрытых зависимостей
- Анализ всей KB целиком (аудит, консистентность)

### Когда НЕ грузить (достаточно SESSION_START + файлы подсистемы):
- Работа над конкретной подсистемой
- Написание кода по конкретной фиче

## Общие правила для всех итераций

- **Одна итерация = одна или несколько сессий** (не растягивать сессию до 90%+ контекста)
- **Субагенты — для механики** (grep, glob, подсчёты), не для анализа
- **Результаты записывать сразу** в KB, не накапливать
- **SESSION_START.md обновлять** в конце каждой итерации (прогресс, новые находки)
- **Пользователь явно говорит когда начинать** — не начинать анализ без команды
