## Активные баги

### 2a. Регресс инпута комбо-ударов — открыт

**Симптом:** при комбо ударов нажатия как будто плохо регистрируются — последовательные нажатия могут не срабатывать. До недавних изменений (в render_interp / fps_unlock в целом) такого не было.

**Гипотеза:** скорее всего НЕ render-interp баг — это input/state handling. Возможно где-то логика проверяет animation state вместо input event'а, и из-за изменённого темпа физики/анимации временные окна для комбо нарушены. Или обработка input в state handler'ах где-то завязана на какой-то рендер-сторонний параметр.

**Что нужно:** воспроизвести с известным набором нажатий, проверить регистрируются ли input event'ы (input_actions), сравнить с initial_working_d3d_build/last_working_d3d_build версиями для понимания регресса.

**Приоритет:** средний — мешает играть в комбатах.

### 5. Подёргивание близких к камере объектов (jitter камеры в interp) — открыт

**Симптом:** объекты на расстоянии 1-2 метра от камеры **подёргиваются туда-сюда** — мелкое back-and-forth движение. Дальние объекты — не видно (возможно есть тоже, но pixel-смещение слишком мелкое чтобы заметить). Хорошо заметно на низкой физике (debug-клавиша 1, ~5 Hz и особенно если довести до 1 Hz). Проявляется и в обычном геймплее, и в катсценах. Замечено 2026-05-02. Без интерполяции (toggle 3) — не подтверждено воспроизводится ли тоже.

**Симптом такой**, как будто **угол камеры** интерполируется некорректно — мелко качается между значениями вместо плавного sweep'а. Близкие к камере объекты сильнее реагируют на screen-space смещение от angular jitter (пиксельное смещение пропорционально расстоянию × tan(jitter_angle), для близких объектов малый угол даёт заметный screen-space сдвиг).

**Возможные направления расследования:**
1. **Integer truncation в `lerp_i32` / `lerp_angle_cam`** — на длинных интервалах между physics-ticks накапливаемые ошибки округления видны как jitter.
2. **`double → float` конверсия `g_render_alpha = float(physics_acc_ms / phys_step_ms)`** теряет precision.
3. **Что-то перезаписывает `FC_cam[0]` или EWAY_cam-fields между ctor и render** — даёт несинхронность с object lerps.
4. **Angle wrap-around** (`lerp_angle_cam`) даёт non-monotonic результат на edge-case prev/curr (≈±π/2 mod 2π).
5. **Фракционная позиция камеры в физ-пространстве** — если physics держит camera pos в integer sub-units и tick-to-tick разность 1-2 sub-units, lerp с alpha=0..1 даёт continuous fractional values но floor/cast обратно к integer на каком-то этапе render path даёт discrete jumps.

**Также записано в** [`new_game_planning/known_issues_and_bugs.md`](../../../new_game_planning/known_issues_and_bugs.md) под "Подёргивание близких к камере объектов" — главный список 1.0-blocker'ов.

**Приоритет:** средний — заметно при внимательном взгляде, не критично для геймплея.

## Известные ограничения (не баги — отсутствие покрытия)

### 3. Партиклы рывками

**Симптом:** искры, glitter, дым могут двигаться зернисто на 60 Hz рендере.

**Причина:** PARTICLE pool — отдельная структура от Things, движется через `TICK_RATIO` на physics-tick. Не интерполируется.

**Приоритет:** низкий. Из-за короткого lifetime партиклов и flicker'а — обычно не бросается в глаза. Отложено до подтверждения что заметно.

### 4. Короткий путь lerp угла при быстром вращении — направление может перепутаться

**Симптом (предполагаемый, требует проверки на практике):** при очень быстром вращении объекта lerp угла может крутить **в обратную сторону** или дёргаться. Заметно на сценах где объект раскручен на большой угловой скорости — например, конусы которые задели машиной и они отлетают, быстро крутясь вокруг своей оси.

**Причина:** `lerp_angle_2048` (DIRT pool) и `lerp_angle_cam` (камера) используют **short-path** интерполяцию — выбирают кратчайший путь по окружности (≤180° / ≤1024 единиц). Если за один physics-тик объект реально провернулся на больше чем 180°, кратчайший путь между snap.prev и snap.curr идёт **в противоположном направлении** от настоящего вращения.

Пример: за тик угол изменился с 0 до 1500 (физика провернула на 1500 единиц по часовой). Short-path считает: 1500 → 1500-2048 = -548 — то есть «548 единиц против часовой». Lerp на рендере крутит против часовой, хотя физически объект крутится по часовой. Между двумя последовательными такими тиками рендер показывает «дёргается туда-сюда».

**Когда проявляется:**
- Угловая скорость > 1024 ед/тик = >180° за 50ms = >3600°/сек = >10 оборотов в секунду
- Это **очень быстрое** вращение, бывает только при резких физ-импульсах (отлетающие объекты, перевёрнутые машины)

**Решение для CLASS_VEHICLE — закрыто через `Vehicle->VelR` direction hint** (`lerp_angle_2048_directed`). Знак VelR определяет на какой стороне круга идёт `prev → curr`, обходя short-path heuristic. Vehicle снимает Angle/Tilt/Roll своим snapshot веткой (см. [`architecture.md`](architecture.md), [`coverage.md`](coverage.md)).

**Решение для CLASS_PERSON / CLASS_BAT (pose snapshot) — закрыто архитектурно.** Body-rotation теперь идёт через quat slerp в pose composer (`CQuaternion::BuildTween`) per-bone, не через short-path angle lerp. Slerp работает по 4D-quaternion sphere и не имеет 180°-cliff'а.

**Для других классов проблема остаётся открытой** в той части где быстро крутящиеся объекты используют DIRT pool (`lerp_angle_2048` short-path). Кандидаты:
- DIRT-объекты (катящиеся бочки, отлетающие банки) — могут быстро крутиться при взрыве. Если станет видимо, нужен angular velocity hint в физике этих классов либо selective skip.

**Возможные решения:**
- **Хранить знак угловой скорости рядом со снапшотом.** Аналогично Vehicle, но требует чтобы класс физически хранил angular velocity где-то в Genus-структуре.
- **Просто отключить угловую интерполяцию для быстрых объектов** (skip apply на текущем frame пока скорость > threshold). Лучше дёрганье на 20Hz чем визуальный реверс направления.

**Приоритет:** низкий до подтверждения на практике. Быстрое вращение — короткий эффект (объект быстро остановится), большинство игроков может не заметить.
