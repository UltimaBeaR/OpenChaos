# ⚠️ ОСНОВНОЙ ПРИНЦИП ЗАДАЧИ FPS UNLOCK

> **Визуальные эффекты и звук — привязаны к реальному времени (wall clock).**
> **Вся физика, геймплей, таймеры миссий — привязаны к physics-тикам.**
> **Ничто — к номеру рендер-кадра или `lock_frame_rate(g_render_fps_cap)`.**

**Базовая частота физики: 20 Hz** (`g_physics_hz = UC_PHYSICS_DESIGN_HZ = 20`,
`THING_TICK_BASE_MS = 1000 / UC_PHYSICS_DESIGN_HZ = 50ms` в `thing.cpp`).
В оригинале была пара значений `NORMAL_TICK_TOCK`: глобальный `1000/15 = 66.67ms`
из `Headers/Game.h` (использовался только psystem.cpp как scaling base) и локальный
override в `Thing.cpp` `1000/20 = 50ms` (там TICK_RATIO реально считается).
Сейчас `NORMAL_TICK_TOCK` удалён — каждая подсистема имеет явную базу:
`THING_TICK_BASE_MS` (50ms) в thing.cpp и `PSYSTEM_TICK_BASE_MS` (66.67ms) в psystem.cpp.
Это дизайновая частота оригинала — при ней таймер миссий идёт 1:1 с реальным временем.
PS1 и PC-релиз работали на ~30 Hz и ~22 Hz соответственно, не достигая 1:1.
Не менять без веской причины.

**Визуальная каденция: 30 Hz** (`UC_VISUAL_CADENCE_HZ = 30`, `UC_VISUAL_CADENCE_TICK_MS = 33.33ms`).
PS1 hardware lock 30 Hz / PC config default 30 Hz — под этим темпом MuckyFoot
тестировали wall-clock эффекты (drip, puddle, mist, rain bucket-strobe) и
GAME_TURN-gated визуалы (siren flash, wheel rotation, spark fence gate, etc).
Эта частота отвязана от physics-tick через `VISUAL_TURN` (визуальный счётчик
тикающий 30/sec wall-clock независимо от physics rate).

**Particle scaling: 15 Hz** (`UC_PARTICLE_SCALING_HZ = 15`) — scaling artefact
psystem.cpp, wall-clock-correct через local_ratio compensation. Не design rate.

Подробный разбор всех трёх частот → [`original_tick_rates/findings.md`](original_tick_rates/findings.md).
Per-effect mapping → [`original_tick_rates/summary.md`](original_tick_rates/summary.md).

Если ты пишешь или ревьюишь код в рамках этой задачи — **этот принцип не подлежит обсуждению**.
Любое решение которое его нарушает — баг, не фича.

### Что к чему привязано

| Категория | Привязка |
|---|---|
| Визуальные эффекты (огонь, дождь, туман, анимации объектов) | Wall-clock (`g_frame_dt_ms`, `sdl3_get_ticks()`) |
| Звук | Wall-clock (стрим OpenAL) |
| Физика, коллизии, AI, движение персонажей | Physics-тик (фиксированный шаг, accumulator) |
| Геймплейные таймеры (миссии, кулдауны) | Physics-тик |
| Рендер-кадр | Только сам рендер (ничего больше) |

Граница простая: **если эффект чисто визуальный — wall-clock**. Если влияет на
состояние игрового мира — **physics-тик**.

---

## Что это значит конкретно

**Рендер** — привязан к delta time (wall clock). Не к «сколько кадров уже нарисовали»,
не к индексу render-цикла, не к тому успел или не успел железо в дедлайн.

**Физика** — привязана к фиксированному шагу через accumulator. Wall-clock дельта
накапливается, физика тикает дискретными шагами. Если рендер не успевает — physics
всё равно догоняет нужное число тиков.

**Видео, звук, анимации, AI, таймеры, любое отображение времени** — только к wall-clock
или к physics-tick (который сам идёт по wall-clock). Никогда — к render-frame counter.

**Если железо пролагало** и кадр не отрисовался — следующий рендер показывает
состояние мира, синхронное прошедшему **wall-clock** времени. Мир не «отстаёт»
из-за того что рендер не успел.

---

## `lock_frame_rate(g_render_fps_cap)` — render frame cap (default 0 = unlimited)

На значение `g_render_fps_cap` **НЕ ИМЕЕТ ПРАВА** опираться какая-либо логика —
ни геймплейная, ни видеоплеер, ни анимации, ни звук, ни AI.

`lock_frame_rate` делает **только** искусственное `sleep` между swap'ами для отладки.
Если поставить `g_render_fps_cap = 5`, корректное поведение:

- рендер раз в 200 мс,
- видео играется в реалтайме (пропускает кадры),
- физика тикает в реалтайме (accumulator продолжает работать),
- звук играется в реалтайме (стрим OpenAL),
- AI решения принимаются в реалтайме (по physics-tick).

Никакого «всё замедлилось». Только рендер реже сэмплирует — пропускает
промежуточные визуальные состояния, а сам мир идёт как обычно.

Если меняешь `g_render_fps_cap` и поведение чего-либо кроме самой частоты
рендера — баг. Чинить поведение, не маскировать.

---

## Что нельзя

- ❌ «Через X кадров рендера произойдёт Y» — баг.
- ❌ «Длительность анимации = N render-кадров» — баг.
- ❌ «AI cooldown декрементируется per render frame» — баг.
- ❌ Видеоплеер ждёт между декодированными кадрами вместо drop-when-behind — баг.
- ❌ Завязывать любую длительность / период / задержку на «сколько раз отработал
  render loop» — баг.
- ❌ Пользоваться `g_render_fps_cap` для расчётов чего-либо кроме busy-wait —
  баг.

## Что можно

- ✓ «Через X миллисекунд произойдёт Y» (sdl3_get_ticks-based) — норм.
- ✓ «На physics-tick K произойдёт Y» — норм (тики идут фиксированным шагом по
  wall-clock).
- ✓ Render берёт интерполированное состояние между physics-tick K и K+1 при
  `alpha = acc / phys_step` — норм (post-1.0).
- ✓ Drop video-frame если PTS уже в прошлом относительно wall-clock — норм
  (правильный sync).
- ✓ `lock_frame_rate(g_render_fps_cap)` в путях презента — норм (это
  и есть отладочный busy-wait).

## Если возникает соблазн нарушить

Перечитать этот файл. Спросить пользователя. Не «срезать угол» через render-frame

техдолг, который потом будет ловиться багами на нестандартных частотах.

## Таймеры геймплея: наша скорость быстрее оригинала (1:1 с реальным временем)

`THING_TICK_BASE_MS = 1000 / UC_PHYSICS_DESIGN_HZ = 50ms` — дизайновая базовая
единица, объявлена в `thing.cpp` (там где TICK_RATIO реально считается).
В оригинале здесь стоял локальный `#undef NORMAL_TICK_TOCK + #define 1000/20`
поверх глобального `NORMAL_TICK_TOCK = 1000/15 = 66.67ms` из `Headers/Game.h`.
Сейчас глобальный `NORMAL_TICK_TOCK` удалён, каждая подсистема имеет явную
базу: `THING_TICK_BASE_MS` (50ms) и `PSYSTEM_TICK_BASE_MS` (66.67ms).

При 20 Hz: tick_diff = 50ms → `TICK_RATIO = (50 * 256) / 50 = 256`.
Формула `5 * 256 >> 8 = 5` → 5 × 20 тиков/сек = **100 единиц/сек = 1:1 с реальным временем**.

Оригинал (PS1, ~30 Hz): tick_diff ≈ 33ms → `TICK_RATIO = (33 * 256) / 50 ≈ 168`.
Формула `5 * 168 >> 8 = 3` (integer truncation) → 3 × 30 = **90 единиц/сек ≈ 90% скорости**.
PC retail (~22 Hz): аналогично ≈ 88% скорости. Ни одна из оригинальных версий не давала 1:1.

**Правило:** physics Hz = `UC_PHYSICS_DESIGN_HZ` = 20 — дизайновая частота,
не менять без веской причины. Фиксить только нестабильность (зависимость от
нагрузки, платформы, кадра).

## Паттерн: bucket-strobe для «60-герцовых» визуальных эффектов

> **Применимо к эффектам, у которых оригинальный визуал — это серия резких
> «вспышек» на 30 Hz, и плавность 60+ Hz делает их хуже, а не лучше.**

Не каждый визуальный эффект выигрывает от плавного wall-clock рендера. У
некоторых оригинальных эффектов вся выразительность — в **резкой смене**
состояния каждые 33 мс. Если такой эффект сделать «плавным» (persistence,
fade, motion-blur, lerp между состояниями), он перестаёт выглядеть как
оригинал — становится «мутным», «слабым», «не таким резким».

Канонический пример — **дождь**. Оригинал: 128 случайных капель ровно один
кадр, потом сменяются на 128 других позиций. Это и есть его эстетика.
Persistence + motion + fade на 60 Hz делает дождь визуально **другим
эффектом** (плавным, а не stroboscopic), который пользователь воспринимает
как баг.

**Решение:** bucket-strobe.

```
bucket_size_ms = UC_VISUAL_CADENCE_TICK_MS  // 33.33 ms = original 30 Hz visual cadence
cur_bucket = wall_clock_ms / bucket_size_ms

if (cur_bucket != last_bucket) {
    // Полностью регенерируем состояние эффекта (как один кадр оригинала).
    regenerate_effect_state();
    last_bucket = cur_bucket;
}

// Каждый render-кадр перерисовываем СТАТИЧНОЕ состояние эффекта
// без motion / fade / interpolation. Между bucket boundary state
// не меняется вообще.
draw_effect_state();
```

Что получаем:
- На 30 FPS bucket меняется каждый рендер-кадр ⇒ эффект **идентичен
  оригиналу**, кадр за кадром.
- На 60/120/240 FPS state стоит несколько кадров неизменным, потом разом
  переключается — глаз видит **тот же 30 Hz strobe** что в оригинале,
  без зависимости от render rate.
- Плотность, интенсивность, размер sprite'ов, рандомизация — всё
  рассчитывается ровно по тем формулам что и в оригинале на 30 FPS.

Принцип: если плавность на high-FPS меняет восприятие эффекта — **не
делай плавность**. Делай strobe-bucket по UC_VISUAL_CADENCE_TICK_MS, и эффект
будет 1:1 как оригинал на любой частоте рендера.

**Когда применять:** короткоживущие случайно-регенерируемые ансамбли
(дождь, снег, искры, пыль), стробоскопические эффекты (мигалки сирен,
вспышки выстрелов, неон), оригинал-калиброванные spawn-ансамбли.

**Когда НЕ применять:** долгоживущие объекты с реальной траекторией
(пули, гранаты, машины, NPC), интерполируемые позиции игровых
сущностей, видеоплеер, звук, AI решения, mission timer и любая логика
с реальной wall-clock семантикой — для них работает обычный wall-clock
рендер с интерполяцией физики (см. выше).

См. реализацию для дождя: `AENG_draw_rain` в
[`engine/graphics/pipeline/aeng.cpp`](../../new_game/src/engine/graphics/pipeline/aeng.cpp).
