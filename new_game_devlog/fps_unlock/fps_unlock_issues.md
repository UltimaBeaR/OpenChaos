## 14. Визуальный статтер при переключении render cap

> **Схема клавиш изменилась.** Запись писалась под старую схему (клавиша 5 = тогл
> 30↔240 FPS). Сейчас аналогичный тогл — клавиша **2** (unlimited↔30). Поведение
> переключения cap'а суть та же, проблема может проявляться так же.

**Симптом:** при переключении render cap (с unlimited на capped или обратно)
иногда появляются тормоза — визуально как 10-15 FPS, счётчик при этом показывает
правильное значение. Воспроизводится редко, не стабильно. Vsync выключен.

**Ключевое наблюдение (старая схема):** повторное нажатие тогла фиксит проблему.
Причина — не OS scheduler, а внутренний стейт игры который накапливается при
unlimited FPS и сбрасывается при переключении обратно.

**Гипотезы:**
- `physics_acc_ms` накопил большое значение при unlimited FPS (до переключения);
  первые кадры при низком cap'е делают несколько physics-тиков за один рендер-кадр,
  утяжеляя кадр. Кратковременно, но может быть заметно.
- Какой-то render-side накопленный стейт (буфер команд GPU, деградировавший
  pipeline state) который сбрасывается при смене режима.

**Куда копать:** добавить логирование `physics_acc_ms` в момент переключения cap'а
— проверить не превышает ли оно `phys_step_ms * N` для нескольких тиков. Связано с
issue [#17](#17-накопление-physics-тиков-при-просадке-производительности-spiral-of-death).

**Актуальность:** только в debug-режиме. В релизе механизма нет. Низкий приоритет.

## 15. Отзывчивость управления зависит от physics Hz

> **Частично улучшено (2026-04-29).** Скорость поворота `max_angle` поднята с 128 до 192
> (TICK_RATIO-масштабируемое) — даёт 3840 единиц/сек поворота = то что ощущалось в
> оригинале. Остаточная задержка до 50ms на смену направления — фундаментальный предел
> 20 Hz физики, требует render interpolation для полного решения.
>
> **Решится** переработкой управления — см. [`../input_system/current_plan.md`](../input_system/current_plan.md)
> (модуль `input_frame` на render-tick'е, sticky press для физики уберут потерю edge'ей).

**Симптом:** при physics 20 Hz управление ощущается менее отзывчивым — нажатие кнопки
регистрируется только на следующем physics-тике, до которого может быть до 50ms.

Особенно заметно на **левом стике при беге** — резкая смена направления, персонаж
поворачивается медленнее чем ожидается. Проявляется как «накапливающийся» эффект
поворота при вращении стиком по кругу.

**Первопричина (зафиксирована):** `max_angle = 128` в `player_turn_left_right_analogue()`
не масштабировался с physics Hz. При 20 Hz давал 2560 единиц/сек поворота.
Поднято до 192 + добавлено TICK_RATIO-масштабирование → 3840 единиц/сек.
При какой Hz оригинал тестировался неизвестно, но ощущение стало лучше (подтверждено).

**Остаток:** ввод опрашивается раз в 50ms (physics-тик). Между тиками нажатия видны
только как последнее состояние. При быстром вращении стика — aliasing.

**Фикс остатка (направление):** render interpolation снимет визуальную рывковость.
Полный фикс latency — вынести опрос ввода в render-цикл с накоплением между тиками.

## 17. Накопление physics-тиков при просадке производительности (spiral of death)

> **⚠️ Гипотетическая проблема, требует проверки.** Возможно частично уже защищено
> существующим clamp'ом frame_dt до 200ms. Подумать над дополнительной защитой.

**Симптом (предполагаемый):** если компьютер уходит в низкопроизводительный режим
(термальный throttling, фоновая задача забрала CPU, переход на батарею у ноутбука),
рендер-кадры становятся длинными. `physics_acc_ms` за один кадр накапливается на
несколько physics-шагов. На следующем кадре `while (acc >= step)` крутится несколько
итераций — каждая по 50ms физики плюс рендер. Если железо тормозит и каждый тик стоит
дорого, цикл может дать **spiral of death**: тики добавляют больше времени к accumulator
чем съедают, кадр становится ещё длиннее, на следующем кадре ещё больше тиков, и т.д.

**Что уже есть (защита):** в [`game.cpp:889`](../../new_game/src/game/game.cpp#L889) —
`if (frame_dt_ms > 200.0) frame_dt_ms = 200.0;` — clamp delta-time на 200ms. Это даёт
максимум 4 physics-тика за один кадр (200/50). Если рендер залип на >200ms, мы **теряем
реальное время** в физике (физика отстаёт от wall-clock), но spiral of death не возникает.

**Что может быть недостаточно:**
- Защита через clamp dt — косвенная. Прямого ограничения «не более N тиков за итерацию
  game loop» нет.
- При дальнейшем падении производительности (frame > 500ms) физика просто отстаёт —
  это игроку видно как «игра идёт в slow-motion». Возможно лучше другое поведение.

**Что попробовать (когда дойдут руки):**
- Явный лимит итераций: `while (acc >= step && tick_count < MAX_TICKS_PER_FRAME) { tick; }`
  с `MAX_TICKS_PER_FRAME = 3..5`. Если accumulator не успел разгрузиться — сбросить остаток
  (`acc = 0`) или оставить (тики просто запоздают).
- Hard reset accumulator при катастрофическом отставании: `if (acc > step * N) acc = step;`
  — лучше потерять время чем уйти в spiral.

**Альтернатива:** возможно проблема надумана и существующего clamp'а 200ms достаточно
для realistic просадок производительности. На целевом железе (Steam Deck, интегрированная
графика) — проверить через искусственную задержку (например `SDL_Delay(300)` в render loop)
и посмотреть как себя ведёт игра.

**Место:** [`game.cpp` physics loop](../../new_game/src/game/game.cpp) — строки 999-1083
(while-loop `physics_acc_ms >= phys_step_ms`).

**Приоритет:** низкий до тех пор пока не воспроизведено. Подумать как защититься
безопасно (без поломки gameplay) — возможно достаточно текущего поведения.

## 16. Ревизия таймингов старых фиксов после смены physics на 20 Hz

> **⚠️ Нужно до 1.0.** При переходе с 30 Hz на 20 Hz все тайминги которые подбирались
> под 30 тиков/сек нужно перепроверить и пересчитать под 20.
>
> **Статус (частично закрыто):** введение `VISUAL_TURN` (30 Hz wall-clock счётчик)
> и миграция всех визуальных `GAME_TURN`-gated сайтов на него закрыло **визуальную**
> часть проблемы — мерцания, анимации фаз, sound variation indices теперь работают
> на оригинальной 30 Hz cadence. Остаются **per-tick gameplay таймеры** (например
> `Timer1`/`Timer2` в person.cpp), они тикают 1.5× медленнее PS1 — это требует
> отдельной ревизии и не закрыто.

**Симптом:** ряд фиксов анимационных и визуальных багов содержит пороги, задержки или
счётчики которые подбирались эмпирически при 30 Hz физики. При 20 Hz эти значения
дадут другой результат — эффект будет слишком медленным (в 1.5× раза) или поведение
изменится.

**Что искать:** константы вида `timer > 30`, `counter += 1` (где 1 подбиралось под
30 тиков/сек), пороги срабатывания в тиках, задержки в тиках — всё что не масштабируется
через `TICK_RATIO`. Особенно опасны маленькие целочисленные значения (1, 2, 3) — они
не компенсируются `TICK_RATIO`.

**Где искать:** `prerelease_fixes.md` — список всех выполненных фиксов пре-релизных
багов. Каждый фикс где упоминается тайминг, счётчик или порог — кандидат на проверку.

**Фикс:** для каждого затронутого места пересчитать константу под 20 Hz, либо
перевести на `TICK_RATIO`-масштабируемый вариант чтобы значение не зависело от Hz.

## 22. [TASK] Режим диагностики перформанса отрисовки (per-stage frame breakdown)

**Зачем:** перф просаживается в **определённых местах** игры (некоторые сцены / ситуации
/ эффекты), но без инструментария непонятно **что именно** тормозит. Нужен способ
видеть в реальном времени какая стадия игрового цикла занимает больше всего
времени, чтобы локализовать узкие места.

**Что нужно:** под отдельным debug-флагом — режим где:

- Игровой цикл инструментирован per-stage таймерами (physics tick, AI/PCOM, AENG_draw,
  POLY render, post-process, UI, present и т.д. — точные стадии надо собрать).
- Каждая стадия меряет своё время выполнения (wall-clock).
- На экране в реалтайме рисуется breakdown — **усреднённые по N кадрам** (как FPS-счётчик)
  значения: сколько мс / сколько % от общего времени кадра занимает каждая стадия.
- Усреднение нужно чтобы цифры не дёргались каждый кадр — рабочая величина окна
  усреднения подбирается по ощущению (~30-60 кадров обычно).

**Что это даёт:**
- Видно сразу что именно тормозит (например: "AENG_draw 65%, остальное ~35%" → лезть
  в render path).
- Можно бегать по сценам и видеть как breakdown меняется (миссия с дождём → particles
  пик; катсцена → animation/figure пик).
- Дешевле чем подключать внешний профайлер каждый раз.
- Помогает калибровать перф под целевое железо (Steam Deck, integrated GPU) — видно
  где экономить.

**Где взять stages:**
- Главный цикл — game.cpp main loop, физика-тик, render-фрейм.
- Render — AENG_draw_screen и его пасы (city / character / poly / postprocess).
- Sub-рендер — POLY queues, шейдеры, lighting, FBO blit / composition.
- Можно начать грубо (3-5 крупных стадий) и углублять при надобности.

**Реализация (грубо):**
- `g_perf_diag_enabled` global, тогл клавишей (например после bangunsnotgames, или
  через console command).
- `perf_stage_begin(name) / perf_stage_end(name)` API в `engine/debug/perf_diag/`.
- Внутри — ring buffer на N кадров, считает avg/min/max за окно.
- При активном флаге рисуется текстовая таблица в углу экрана (text-rendering skill).

**Связь с perf-diag skill:** в `.claude/skills/perf-diag/SKILL.md` — справочник по
graphics performance diagnostics, инструменты для **временного** инструментирования.
Этот issue — про **постоянный встроенный** debug-режим, чтобы пользователь мог
включить и посмотреть без изменения кода. Скилл — для глубоких расследований; этот
режим — для повседневного быстрого чека "где тормозит сейчас".

**Приоритет:** средний. Не блокер 1.0, но желателен ДО релиза — чтобы оптимизировать
перф финально перед выходом, имея на руках инструмент для локализации проблем.
