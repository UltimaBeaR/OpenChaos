## 1. Главное открытие (по коду, не по KB)

Текущая система персонажей — **уже скелетная**, не «болванки без скелета».
Проверено в `figure.cpp` / `polypage.cpp`:

- **Кости** — каждая часть тела = слот матрицы; в коде так и зовётся
  *skeleton slot* (`FIGURE_draw_prim_tween`, `part_number`).
- **Иерархия** — родитель→ребёнок (`parent_base_mat`/`parent_curr_mat`/
  `parent_*_pos`), прямая кинематика.
- **Покостная анимация** — кейфреймы на кость (`GameKeyFrameElement`:
  offset + rotation), межкадровый слёрп кватернионами
  (`CQuaternion::BuildTween`, `figure_morph_matrix`/`figure_morph_offset`).
- **Покостный кэш поз уже есть** — `render_interp_get_cached_pose`,
  `BoneInterpTransform`.
- **Привязка вершина→кость уже в данных** — у каждой вершины индекс кости
  (`bMatIndex`, байт по смещению 12), читается в `ge_draw_multi_matrix`.

Не хватает ровно **одного**: весов (мульти-кость на вершину). Сейчас
каждая вершина жёстко привязана к **одной** кости (rigid skinning) —
поэтому щели на стыках (шея/голова) при движении частей тела.

**Следствие для плана:** авто-риггер в итерации 1 **НЕ нужен** — привязка
уже есть в данных. Авто-риг (генерация новых весов) — только в итерации
сглаживания.

## 2. Где настоящая тяжесть (не риггинг)

Сейчас в матрицы костей **запечена проекция** (World×Projection вместе).
Отсюда костыли-компенсации в `ge_draw_multi_matrix`:

- спец-формула z `pTLVert.z = 1 - POLY_ZCLIP_PLANE / z`
  (`BUGFIX-OC-ZBUF-MISMATCH` — совпадение z-пространства с POLY-путём);
- per-vertex туман считается отдельно из view-space Z
  (`g_mm_fog_view_z`, `POLY_FADEOUT_*`), т.к. в вершины приходит уже
  спроецированное;
- per-vertex half-Lambert освещение на CPU (dot(normal, bone-local light)
  → 64-рамп, `fNormScale = 251`).

Ядро работы и главный риск — **расцепить** это: VBO = модельные вершины
(статичные), матрицы костей = модель→вид (без проекции), а
проекция/освещение/туман — в шейдере, с воспроизведением вида **1:1**.
Риггинг тут ни при чём.

## 3. Цель

Постоянный VBO модели персонажа (создаётся один раз, не пересоздаётся
каждый кадр; переиспользуется между полностью одинаковыми моделями).
Каждый кадр меняются только матрицы костей. Скиннинг + трансформ + свет +
туман — в вершинном шейдере. Вид — пиксель-в-пиксель как сейчас.
Архитектура с первого дня поддерживает мульти-костные веса (потом меняем
только данные весов, не конвейер).

## 4. Этапность

### Итерация 1 — конвейер скиннинга, вид 1:1

Объединяет «жёсткую привязку» и «постоянный VBO + GPU-скиннинг» в одну
сборку (раздельно смысла не имеют).

- **Постоянный VBO в модельном пространстве**: создаётся один раз
  (загрузка / первое использование), переиспользуется между одинаковыми
  моделями. Ключ — id prim-object / модели.
- **Формат вершины — сразу под веса** (см. §5). Итерация 1 пишет
  `[существующая кость, вес 1.0, пусто, пусто, пусто]`. Шейдер делает
  взвешенную сумму; при одном влиянии результат **идентичен** жёсткой
  привязке.
- **Палитра матриц костей** — на инстанс, на кадр (UBO/юниформы).
  Источник — существующий покостный кэш поз (`render_interp`).
  Анимацию/FK/tween **не переписываем**, переиспользуем.
- **Шейдер воспроизводит 1:1**: модель→вид по палитре костей, затем
  чистая проекция; half-Lambert по 64-рамп с `fNormScale 251`; туман по
  view-Z. Убираются костыли z/fog (становятся честной математикой в
  шейдере).
- **Критерий приёмки:** визуально неотличимо от текущего билда на
  контрольных сценах (RTA, катсцены, ближний план Дарси, ночь/туман,
  прозрачные персонажи) **И** геометрия не перезаливается каждый кадр
  **И** трансформ на GPU.

### Итерация 2 — эксперимент со сглаживанием швов

- Меняются **только данные весов**. Конвейер не трогаем (он уже
  мульти-костный с итерации 1).
- Bind-поза: явной T-позы нет → берём подходящий **нейтральный кадр** как
  референс (в нейтральной позе щели минимальны; вылезают при движении —
  значит сглаживать надо переходы в суставах).
- Авто-риг: синтез весов для **граничных вершин в суставах**. Алгоритм
  подбираем итеративно, оцениваем глазами. Система готова — это
  data-only.

## 5. Формат данных вершины (постоянный VBO)

Всё нужное уже есть в исходных данных модели:

- позиция (модельное пространство)
- нормаль (модельное пространство; `GEVertex.nx/ny/nz`)
- UV
- до 4 индексов кости + 4 веса
  - итерация 1: `idx0 = bMatIndex, w0 = 1.0`, остальные нули
  - итерация 2: те же поля, заполняются авто-ригом — **формат не
    меняется**

Решить (открытый вопрос): максимум костей на персонажа → размер палитры
(UBO). Посчитать по фактической иерархии частей тела.

## 6. Чек-лист «1:1» (риски итерации 1)

Самый визуально-чувствительный код в игре, требование 1:1 жёсткое:

- **z-пространство** — воспроизвести то, что давал
  `BUGFIX-OC-ZBUF-MISMATCH`, иначе персонажи начнут конфликтовать по
  глубине со стенами/машинами (это был отдельный тяжёлый баг — см.
  [stage5_bug_zbuffer_mismatch.md](stage5_bug_zbuffer_mismatch.md)).
- **Освещение** — half-Lambert `cos*0.5+0.5` → 64-рамп, `fNormScale 251`,
  tint-таблица (`MM_pcFadeTableTint`). Нормаль трансформируется костью.
  Ориентир — [stage12_character_lighting_investigation.md](../new_game_planning/stage12_character_lighting_investigation.md).
- **Туман** — ⚠️ НЕ per-vertex. CPU считает **плоский на персонажа**
  фактор из одной скалярной `g_mm_fog_view_z` (view-Z центра, константа
  на весь MM-вызов), формула `POLY_FADEOUT_*`, кладёт в `specular.a`.
  Для 1:1 шейдер воспроизводит ровно это (та же скаляр-величина, та же
  формула), а НЕ градиент по глубине вершины.
- **Текстурные варианты по скиллу** — jacket/face (`TEXTURE_PAGE_FLAG_*`,
  `figure.cpp`): геометрия одна, текстура на инстанс. Меньше draw-вызовов,
  чем Этап 1, это **не даст** — отдельный рычаг (атлас/массив текстур),
  вне этого плана.
- **Прозрачность/сортировка** — `FIGURE_alpha`, порядок прозрачных.
- **Тени/отражения персонажей — путь НЕ подтверждён, перепроверить.**
  Ранняя заметка «тени = отдельный путь» опиралась на эмпирику A/B 1A
  (2026-05-19: «тело ломалось на GPU-пути, тени — нет»). Та поломка тела
  была **багом 1A, который уже починен** (см. §12) — значит аргумент
  «тени не ломались» больше ничего не доказывает: путь теней при
  GPU-скиннинге снова **открытый вопрос**, требует отдельной проверки
  (глазами при тумблере K или поиском в коде где строится геометрия тени
  силуэта). Фактически сейчас тени/отражения временно идут по **старому
  (CPU) пути** и работают; их перевод на скелетную систему — отдельная
  работа (Майлстон 1E).
- **near-clip** — `POLY_ZCLIP_PLANE = 1/512`; чистая проекция в шейдере
  должна дать ту же глубину/precision (см. недавний фикс depth-bias).
- **Winding/стрип** — even/odd флип сохранить (как в Этапе 1).

## 7. Что переиспользуем (не переписываем)

- Анимация костей: FK-иерархия, кейфреймы, кватернионный tween — как есть.
- Покостный кэш поз `render_interp` — источник палитры костей для GPU.
- Этап 1 (батчинг) — остаётся; GPU-скиннинг его естественно поглощает
  (один draw на модель/текстуру, но геометрия теперь статична).

## 8. Что НЕ делаем в этом плане

- Авто-риг алгоритм (итерация 1) — привязка уже в данных.
- Атлас/массив текстур ради меньшего числа draw-вызовов — отдельный
  рычаг, ортогонален.
- Гладкие веса — итерация 2, после подтверждения что конвейер работает.
- Vulkan — отдельная далёкая история.

## 9. Открытые вопросы

- Bind-поза для итерации 2 — какой нейтральный кадр.
- Макс. костей/персонаж → размер UBO палитры.
- Ключ переиспользования VBO между одинаковыми моделями (prim-object id?).
- Тени/отражения персонажей: порт вместе или временно старый путь.
- Скиннованные нормали для освещения — трансформ нормали костью в шейдере

## 11. Исполнительная декомпозиция (зафиксировано по коду)

### Интеграционный дизайн (определён разбором кода)

- **Источник палитры костей — `render_interp_get_cached_pose(Thing*)`**
  → `BoneInterpTransform[POSE_MAX_BONES]` (15 костей: pelvis + 14), уже
  **чистые мировые** pos+rot. Де-бейкать `MM_pMatrix` (там запечено
  World×Projection×Viewport) **не нужно**.
- **Граница нового пути:** используется когда (a) есть валидная кэш-поза
  **и** (b) A/B debug-тумблер = «новый». Иначе — старый
  `ge_draw_multi_matrix`. Фолбэк уже естественный: `compute_pose`
  возвращает false → старый путь.
- **Постоянный VBO:** модельные (bone-local) вершины из исходного
  `TomsPrimObject` (`pD3DVertices`/материалы/стрип-индексы) —
  раскладываются один раз. `boneIndex = bMatIndex` (как читает старый
  путь). Нормаль — из тех же данных.
- **Формат вершины (сразу под веса):** pos(vec3, модель), normal(vec3,
  модель), uv(vec2), 4×boneIndex + 4×weight. Итерация 1: `idx0=bMatIndex,
  w0=1`, остальное 0 → при одном влиянии результат идентичен жёсткой
  привязке.
- **Шейдер:** skin (Σ weight·(boneRot·pos+bonePos)) → view → proj →
  viewport (воспроизвести `g_matProjection` + viewport-математику из
  `MM_pMatrix[0]`-сборки и `tl_vert.glsl`); нормаль скиннится boneRot;
  свет — half-Lambert по 64-рамп (`fNormScale 251`, tint-таблица); туман
  — плоский на персонажа из `g_mm_fog_view_z` (`POLY_FADEOUT_*`), НЕ
  per-vertex (см. §6).
- **Палитра в шейдер:** 15× (rot mat3 + pos vec3) юниформом/UBO, на
  инстанс на кадр.

### Майлстоны (каждый = сборка + твоя A/B визуальная сверка)

- **1A — трансформ+z, цвет проброшен.** Новый шейдер+путь за тумблером;
  GPU-скиннинг позиции, чистая проекция/viewport; **per-vertex цвет пока
  берём из старого CPU-расчёта** (лайтинг/туман идентичны *по
  построению*). VBO пока строим per-frame (изолируем корректность
  трансформа/z от вопроса времени жизни буфера). Гейт: силуэт/позиция/
  **перекрытия со стенами** 1:1 (тумблером).
- **1B — лайтинг в шейдер.** half-Lambert+рамп+tint в шейдере. Гейт:
  тона 1:1 (день/ночь/tint).
- **1C — туман в шейдер.** Плоский на персонажа из `g_mm_fog_view_z`
  (НЕ per-vertex — так делает оригинал; см. §6). Гейт: ночь/туман 1:1.
- **1D — постоянный + общий VBO.** Раскладка модели один раз,
  переиспользование между одинаковыми моделями; per-frame только палитра
  костей. Гейт: перф вверх + всё ещё 1:1. После подписания — новый путь
  дефолт, старый/тумблер удаляем.
- **1E — тени/отражения персонажей.** До 1D идут по старому пути; портим
  на новый (тот же шейдер) и сверяем. Решить: в 1D или отдельно.
  - ⚠️ **Про путь теней (коррекция, 2026-05-19, поздняя):** ранний вывод
    «тени = отдельный путь» был сделан из эмпирики «тело ломалось на
    GPU-пути, тени — нет». Поломка тела оказалась **багом 1A, который
    починен** (§12), поэтому «тени не ломались» больше ничего не
    доказывает — путь теней при GPU-скиннинге снова **открытый вопрос**.
    Сейчас тени/отражения временно идут по **старому (CPU) пути** и
    работают. На Майлстоне 1E: сначала **разобрать реальный shadow-путь в
    коде** (где строится геометрия тени силуэта), не предполагая «тот же
    шейдер автоматически», и только потом портировать.

### Что НЕ в итерации 1

Авто-риг/веса (это итерация 2), атлас текстур (отдельный рычаг),
удаление старого пути (только после полного 1:1-подписания).

### 2026-05-19 — Переписан отбор детальных теней + 6 слотов

Общий хелпер `AENG_select_shadow_persons` (aeng.cpp, namespace) —
используется и `AENG_draw_city`, и `AENG_draw_warehouse` (логика
идентична, дубляж убран). Заменил оригинальный «4 ближайших к камере».

**Приоритет = depthPri × sidePri** (`SHADOW_MIX=2`):
- **depthPri (глубина):** `r` = проекция перса на ось камера→Дарси
  (`r=0` плоскость камеры, `r=1` глубина Дарси). Сплайн: камера
  `CAM_FLOOR` → Дарси `DARCI_LEVEL` → пик `1.0` на `PEAK_R` (чуть за
  Дарси) → пологое плечо (`PLATEAU_R`,`SHOULDER_DROP`) → длинный
  экспо-хвост (`FALL_TAU`).
- **sidePri (бок):** экранный X через `POLY_transform` (передавать
  `WorldPos>>8` — масштаб `POLY_cam_*`!), `|xn|` 0=центр,1=край;
  **FOV-инвариантно** (проекция движка несёт FOV/zoom/lens/aspect).
  Горб: центр `1.0` → колено `SIDE_KNEE`/`SIDE_KNEE_LEVEL` → длинный
  экспо-хвост `SIDE_TAU` к краям (без жёсткого нуля).
- **Фокус (Дарси)** — резерв слота (`score=-INF`, не вытесняется).
- **Гистерезис:** `s_prev_shadow_sel` (набор прошлого кадра) →
  скидка `SHADOW_STICKY_MARGIN` (анти-дребезг при вращении камеры).
- **Тай-брейк:** primary квантуется `SHADOW_TIE_BUCKET`, внутри
  бакета ближе к Дарси выигрывает (`dDarci>>SHADOW_TIE_DSHIFT`).
- Все коэффициенты — именованные `const` в начале хелпера, подобраны
  по ощущению (была визуализация `0.00=0.00*0.00` над персами + лог
  по K — **удалены** после настройки).

**Кол-во слотов → `config.h` `OC_MAX_DETAILED_SHADOWS` (=6).**
`AENG_NUM_SHADOWS` = этот конст. Страница тени `TEXTURE_SHADOW_SIZE`
64→128, пакинг квадратной сеткой `cols=SIZE/AA` (4×4),
`offset=(i%cols, i/cols)*AA`. ASSERT'ы обобщены. Перф: каждый слот =
отдельный GPU-силуэт-пасс/кадр — 6 vs 4 = +50% где ≥6 человек.

`AENG_draw_warehouse` зеркалит город: shadow_person вынесен из
`if(AENG_shadows_on)`, безусловный oval-проход для не-детальных.
Остаток: `OVAL_project_onto_mapsquare` пропускает `PAP_FLAG_HIDDEN`
→ в помещении blob не проецируется (оригинальное ограничение
MuckyFoot; отдельный пункт в known_issues).

### 2026-05-19 — Унификация пути позы (в работе)

Решение: **единственный источник позы скелета** — снимок интерполяции.
Никаких легаси-пересчётов (`calc_sub_objects_position` / `GetCMatrix` /
`build_tween_matrix`) как параллельного пути нигде.

- **Ядро (`render_interp_compute_pose`):** снимок позы **всегда доступен**.
  Если ещё не снят (спавн / смена меша / телепорт / после загрузки) —
  ленивый захват прямо в момент запроса (`capture_pose`, первый захват =
  prev=curr=текущая поза, без интерполяции этот кадр). Убраны early-return
  по `g_render_interp_enabled` и `ri_cfg::INTERP_THING_WORLD_POSE`.
- **Debug-тумблер интерполяции** (`g_render_interp_enabled`) теперь
  выключает **только сглаживание между тиками** (alpha=1 → берётся curr),
  а не саму позу. Снимки снимаются каждый тик независимо от тумблера.
- **Этап «безопасный» (этот шаг):** ядро + тень (`smap_bone_world`) +
  9 рантайм-фолбэков (свет тела/отражения, гранаты, прицел, хелсбары,
  OVAL, факел) — везде удалена ветка легаси-пересчёта.
- **Этап figure.cpp (СДЕЛАН, подтверждён в игре по одной функции):**

**Ход расследования (и одна моя ошибка):**

- Промежуточный фикс: подавление flush по `MM_FLUSH_VERT_THRESHOLD`
  при сборке кэша (материал → один меш). Корректен, оставлен, но не
  первопричина.
- Эксперимент «собрать меш и сразу уничтожить каждый кадр» → картинка
  **корректна** ⇒ GL `ge_skin_mesh_create/draw` исправен, проблема в
  **переиспользовании** меша между кадрами.
- ⚠️ **Ошибочный промежуточный вывод:** первый зонд считал хэш по
  ВСЕЙ вершине, включая `color` (а это CPU-освещение, меняется каждый
  кадр). Хэш «менялся» → я ошибочно заключил «вершины позируются
  каждый кадр, 1D невозможен». **Неверно.** Исходный скелетный анализ
  (другой чат) был прав.
- **Исправленный зонд** (хэш только позиции/нормали/кости/UV, без
  цвета), запущенный КАЖДЫЙ кадр: **54 слота, 0 изменений геометрии**
  даже при беготне близко/далеко. → геометрия **реально статична**,
  кэшировать её можно.

**Первопричина (подтверждена):** классическая порча VAO через
element-буфер. `ge_skin_mesh_create` привязывал `GL_ELEMENT_ARRAY_BUFFER`
**до** привязки своего VAO, а `ge_skin_mesh_draw` оставлял VAO меша
привязанным после отрисовки. Когда позже создавался меш другой модели,
его `glBindBuffer(ELEMENT…)` срабатывал при ещё привязанном VAO
предыдущего меша и **затирал тому индексный буфер**. Меш рисовался с
чужими индексами → часть треугольников пропадала, но видимые
анимировались правильно (vbo-привязка не страдала). «Пересборка каждый
кадр» это маскировала (буфер самовосстанавливался ежекадрово).

**Фикс:** в `ge_skin_mesh_create` привязывать свой VAO ПЕРЕД любым
`glBindBuffer(GL_ELEMENT_ARRAY_BUFFER)`; в `ge_skin_mesh_draw` снимать
привязку VAO (`glBindVertexArray(0)`) после отрисовки.

**Состояние кода:** 1D полностью включён, фикс на месте, весь
диагностический код удалён, сборка exit 0.

**1. 1E — тени персонажей (РАЗОБРАНО 2026-05-19, решение принято).**

Реальный путь найден — это система **SMAP** (`smap.cpp`), отдельная от

- `SMAP_person` (`smap.cpp:381`): CPU-скиннинг — каждая вершина каждой
  части тела гоняется через матрицы костей (тот же общий кэш позы
  `render_interp_get_cached_pose`, что у тела) и проецируется на
  «плоскость солнца» (`SMAP_init`, dir 147/-148/-147).
- `SMAP_point_finished` (UV-фит силуэта в битмап) → `SMAP_add_prim_
  triangles`/`SMAP_tri_add` → `AA_draw`: софт-сглаженная растеризация
  силуэта в буфер 32×32 (`AENG_aa_buffer`), копируется в общую
  shadow-текстуру 64×64 (до 4 паковано, под-ректы хардкод).
- `SMAP_project_onto_poly` + `AENG_add_projected_fadeout_shadow_poly`
  (`aeng.cpp:658/708`): клип квадов земли по плоскости тени, проекция UV
  на пол, submit на `POLY_PAGE_SHADOW` с затуханием по дистанции.

Убираем CPU-косто = **шаги скиннинг+софт-растер** (на каждого ближнего
перса/кадр). Проекция на пол (`SMAP_project_onto_poly` + `AENG_add_
projected_*`) остаётся как есть — ей всё равно, чем заполнена текстура.

**Подход (одобрен пользователем):** рендерить силуэт на GPU нашим уже
готовым скиннингом (персистентный VBO + skin-шейдер) в маленький
офскрин-FBO, ортопроекция вдоль направления солнца, выход = плоский
силуэт (без света/тумана), отдать в ту же shadow-текстуру в тот же
под-рект. Шаги клипа/проекции/затухания НЕ трогаем.

**Критерий приёмки (решение пользователя):** НЕ побайтово. Тень — мягкое
ощущение, в будущем планируется замена на полноценный shadow map. Цель:
«похоже», без упора в пиксели/сглаживание.

**Рамка ортопроекции (решение пользователя + выбор за мной):**
- Жёсткое требование: тень **никогда не должна обрезаться** (рисуется в
  квадратной текстуре; недолёт рамки = срез силуэта = заметный артефакт).
- CPU-проход по **вершинам** запрещён (это и есть то, что убираем).
  Проход по **костям** допустим.
- Решение: рамку считать из позиций костей (уже в кэше позы) + запас на
  вылет конечностей за пределы bbox костей (запас от масштаба
  персонажа). Лучше тень чуть мельче/мягче, чем срез. Лёгкий дрейф
  проекции допустим, грубое расхождение — нет.
- **Про размер запаса и перф (уточнено с пользователем):** силуэт
  рендерится в фиксированный буфер 32×32 (как в оригинале). Размер
  запаса на стоимость GPU НЕ влияет: фрагментная часть ограничена
  32×32 при любом запасе (больше запас → меньше закрашено → даже
  чуть дешевле), вершинная часть = число вершин, от запаса не зависит.
  Единственная цена большого запаса — **качество** (силуэт занимает
  меньше из 32 текселей → мельче/мыльнее). Поэтому запас брать
  **минимально достаточный** (гарантия отсутствия среза при любой
  анимации) — ради резкости тени, НЕ ради перфа.
- Запас оформить именованной константой с обоснованием.

**3. Тотальное удаление всего старого кода персонажей.** Только ПОСЛЕ
п.1–2 (иначе тени/отражения сломаются — они на CPU-рисовании).
Удалять **физически и полностью**, не оставляя мёртвых веток/функций.
Учесть (обсуждено с пользователем):
- **Гарантия полноты:** для именованных сущностей (функции, глобалы) —
  удалять *сам символ*, тогда **компилятор** ловит каждое оставшееся
  использование (сборка не линкуется, пока не вычищено). Проверять не
  «нет слова error», а что прошла **линковка** (правило CLAUDE.md).
- Перед удалением — `grep` по **всему репозиторию** на каждый символ:
  полный список использований.
- **Ветки кода** (а не символы) компилятор НЕ ловит — такие места
  выявлять анализом всех вызывающих, **перечислять пользователю
  списком до удаления**.
- Проверять **косвенные вызовы** (указатели на функции/диспетчеризация)
  — компилятор их не ловит.
- Проверять использование под **другими `#ifdef`/платформами**
  (macOS-сборка и пр.), которые в нашем билде «молчат».
- Кандидаты на удаление: легаси per-face POLY-путь персонажей,
  `FIGURE_rpoint`/reflection CPU-путь (после миграции), любые
  CPU-трансформ/лайтинг хелперы фигуры, ставшие мёртвыми.
