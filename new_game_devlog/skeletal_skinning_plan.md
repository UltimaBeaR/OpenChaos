## 1. Главное открытие (по коду, не по KB)

Текущая система персонажей — **уже скелетная**, не «болванки без скелета».
Проверено в `figure.cpp` / `polypage.cpp`:

- **Кости** — каждая часть тела = слот матрицы; в коде так и зовётся
  *skeleton slot* (`FIGURE_draw_prim_tween`, `part_number`).
- **Иерархия** — родитель→ребёнок (`parent_base_mat`/`parent_curr_mat`/
  `parent_*_pos`), прямая кинематика.
- **Покостная анимация** — кейфреймы на кость (`GameKeyFrameElement`:
  offset + rotation), межкадровый слёрп кватернионами
  (`CQuaternion::BuildTween`, `figure_morph_matrix`/`figure_morph_offset`).
- **Покостный кэш поз уже есть** — `render_interp_get_cached_pose`,
  `BoneInterpTransform`.
- **Привязка вершина→кость уже в данных** — у каждой вершины индекс кости
  (`bMatIndex`, байт по смещению 12), читается в `ge_draw_multi_matrix`.

Не хватает ровно **одного**: весов (мульти-кость на вершину). Сейчас
каждая вершина жёстко привязана к **одной** кости (rigid skinning) —
поэтому щели на стыках (шея/голова) при движении частей тела.

**Следствие для плана:** авто-риггер в итерации 1 **НЕ нужен** — привязка
уже есть в данных. Авто-риг (генерация новых весов) — только в итерации
сглаживания.

## 2. Где настоящая тяжесть (не риггинг)

Сейчас в матрицы костей **запечена проекция** (World×Projection вместе).
Отсюда костыли-компенсации в `ge_draw_multi_matrix`:

- спец-формула z `pTLVert.z = 1 - POLY_ZCLIP_PLANE / z`
  (`BUGFIX-OC-ZBUF-MISMATCH` — совпадение z-пространства с POLY-путём);
- per-vertex туман считается отдельно из view-space Z
  (`g_mm_fog_view_z`, `POLY_FADEOUT_*`), т.к. в вершины приходит уже
  спроецированное;
- per-vertex half-Lambert освещение на CPU (dot(normal, bone-local light)
  → 64-рамп, `fNormScale = 251`).

Ядро работы и главный риск — **расцепить** это: VBO = модельные вершины
(статичные), матрицы костей = модель→вид (без проекции), а
проекция/освещение/туман — в шейдере, с воспроизведением вида **1:1**.
Риггинг тут ни при чём.

## 3. Цель

Постоянный VBO модели персонажа (создаётся один раз, не пересоздаётся
каждый кадр; переиспользуется между полностью одинаковыми моделями).
Каждый кадр меняются только матрицы костей. Скиннинг + трансформ + свет +
туман — в вершинном шейдере. Вид — пиксель-в-пиксель как сейчас.
Архитектура с первого дня поддерживает мульти-костные веса (потом меняем
только данные весов, не конвейер).

## 4. Этапность

### Итерация 1 — конвейер скиннинга, вид 1:1

Объединяет «жёсткую привязку» и «постоянный VBO + GPU-скиннинг» в одну
сборку (раздельно смысла не имеют).

- **Постоянный VBO в модельном пространстве**: создаётся один раз
  (загрузка / первое использование), переиспользуется между одинаковыми
  моделями. Ключ — id prim-object / модели.
- **Формат вершины — сразу под веса** (см. §5). Итерация 1 пишет
  `[существующая кость, вес 1.0, пусто, пусто, пусто]`. Шейдер делает
  взвешенную сумму; при одном влиянии результат **идентичен** жёсткой
  привязке.
- **Палитра матриц костей** — на инстанс, на кадр (UBO/юниформы).
  Источник — существующий покостный кэш поз (`render_interp`).
  Анимацию/FK/tween **не переписываем**, переиспользуем.
- **Шейдер воспроизводит 1:1**: модель→вид по палитре костей, затем
  чистая проекция; half-Lambert по 64-рамп с `fNormScale 251`; туман по
  view-Z. Убираются костыли z/fog (становятся честной математикой в
  шейдере).
- **Критерий приёмки:** визуально неотличимо от текущего билда на
  контрольных сценах (RTA, катсцены, ближний план Дарси, ночь/туман,
  прозрачные персонажи) **И** геометрия не перезаливается каждый кадр
  **И** трансформ на GPU.

### Итерация 2 — эксперимент со сглаживанием швов

- Меняются **только данные весов**. Конвейер не трогаем (он уже
  мульти-костный с итерации 1).
- Bind-поза: явной T-позы нет → берём подходящий **нейтральный кадр** как
  референс (в нейтральной позе щели минимальны; вылезают при движении —
  значит сглаживать надо переходы в суставах).
- Авто-риг: синтез весов для **граничных вершин в суставах**. Алгоритм
  подбираем итеративно, оцениваем глазами. Система готова — это
  data-only.

## 5. Формат данных вершины (постоянный VBO)

Всё нужное уже есть в исходных данных модели:

- позиция (модельное пространство)
- нормаль (модельное пространство; `GEVertex.nx/ny/nz`)
- UV
- до 4 индексов кости + 4 веса
  - итерация 1: `idx0 = bMatIndex, w0 = 1.0`, остальные нули
  - итерация 2: те же поля, заполняются авто-ригом — **формат не
    меняется**

Решить (открытый вопрос): максимум костей на персонажа → размер палитры
(UBO). Посчитать по фактической иерархии частей тела.

## 6. Чек-лист «1:1» (риски итерации 1)

Самый визуально-чувствительный код в игре, требование 1:1 жёсткое:

- **z-пространство** — воспроизвести то, что давал
  `BUGFIX-OC-ZBUF-MISMATCH`, иначе персонажи начнут конфликтовать по
  глубине со стенами/машинами (это был отдельный тяжёлый баг — см.
  [stage5_bug_zbuffer_mismatch.md](stage5_bug_zbuffer_mismatch.md)).
- **Освещение** — half-Lambert `cos*0.5+0.5` → 64-рамп, `fNormScale 251`,
  tint-таблица (`MM_pcFadeTableTint`). Нормаль трансформируется костью.
  Ориентир — [stage12_character_lighting_investigation.md](../new_game_planning/stage12_character_lighting_investigation.md).
- **Туман** — per-vertex по view-Z (`POLY_FADEOUT_*`), тот же вход в
  шейдер.
- **Текстурные варианты по скиллу** — jacket/face (`TEXTURE_PAGE_FLAG_*`,
  `figure.cpp`): геометрия одна, текстура на инстанс. Меньше draw-вызовов,
  чем Этап 1, это **не даст** — отдельный рычаг (атлас/массив текстур),
  вне этого плана.
- **Прозрачность/сортировка** — `FIGURE_alpha`, порядок прозрачных.
- **Тени персонажей рисуются ОТДЕЛЬНЫМ путём** (не `ge_draw_multi_matrix`).
  Прежнее предположение «тот же путь» опровергнуто эмпирикой A/B 1A
  (2026-05-19): тело ломалось на GPU-пути, тени — нет. Тени остаются
  рабочими сами по себе, их перевод на скелетную систему — отдельная
  работа (см. Майлстон 1E). Отражения — проверить аналогично.
- **near-clip** — `POLY_ZCLIP_PLANE = 1/512`; чистая проекция в шейдере
  должна дать ту же глубину/precision (см. недавний фикс depth-bias).
- **Winding/стрип** — even/odd флип сохранить (как в Этапе 1).

## 7. Что переиспользуем (не переписываем)

- Анимация костей: FK-иерархия, кейфреймы, кватернионный tween — как есть.
- Покостный кэш поз `render_interp` — источник палитры костей для GPU.
- Этап 1 (батчинг) — остаётся; GPU-скиннинг его естественно поглощает
  (один draw на модель/текстуру, но геометрия теперь статична).

## 8. Что НЕ делаем в этом плане

- Авто-риг алгоритм (итерация 1) — привязка уже в данных.
- Атлас/массив текстур ради меньшего числа draw-вызовов — отдельный
  рычаг, ортогонален.
- Гладкие веса — итерация 2, после подтверждения что конвейер работает.
- Vulkan — отдельная далёкая история.

## 9. Открытые вопросы

- Bind-поза для итерации 2 — какой нейтральный кадр.
- Макс. костей/персонаж → размер UBO палитры.
- Ключ переиспользования VBO между одинаковыми моделями (prim-object id?).
- Тени/отражения персонажей: порт вместе или временно старый путь.
- Скиннованные нормали для освещения — трансформ нормали костью в шейдере
  (без неравномерного масштаба — проверить, что в матрицах костей нет
  scale, иначе нужна inverse-transpose).

## 10. Процесс

Крупная переделка самого чувствительного к визуалу пути. Делать:

- отдельной веткой / worktree;
- по-этапная **визуальная сверка** с текущим билдом на контрольных
  сценах после каждого ощутимого куска;
- до кода — этот план; реализацию начинать по явной отмашке.

## 11. Исполнительная декомпозиция (зафиксировано по коду)

### Интеграционный дизайн (определён разбором кода)

- **Источник палитры костей — `render_interp_get_cached_pose(Thing*)`**
  → `BoneInterpTransform[POSE_MAX_BONES]` (15 костей: pelvis + 14), уже
  **чистые мировые** pos+rot. Де-бейкать `MM_pMatrix` (там запечено
  World×Projection×Viewport) **не нужно**.
- **Граница нового пути:** используется когда (a) есть валидная кэш-поза
  **и** (b) A/B debug-тумблер = «новый». Иначе — старый
  `ge_draw_multi_matrix`. Фолбэк уже естественный: `compute_pose`
  возвращает false → старый путь.
- **Постоянный VBO:** модельные (bone-local) вершины из исходного
  `TomsPrimObject` (`pD3DVertices`/материалы/стрип-индексы) —
  раскладываются один раз. `boneIndex = bMatIndex` (как читает старый
  путь). Нормаль — из тех же данных.
- **Формат вершины (сразу под веса):** pos(vec3, модель), normal(vec3,
  модель), uv(vec2), 4×boneIndex + 4×weight. Итерация 1: `idx0=bMatIndex,
  w0=1`, остальное 0 → при одном влиянии результат идентичен жёсткой
  привязке.
- **Шейдер:** skin (Σ weight·(boneRot·pos+bonePos)) → view → proj →
  viewport (воспроизвести `g_matProjection` + viewport-математику из
  `MM_pMatrix[0]`-сборки и `tl_vert.glsl`); нормаль скиннится boneRot;
  свет — half-Lambert по 64-рамп (`fNormScale 251`, tint-таблица); туман
  — per-vertex по view-Z (`POLY_FADEOUT_*`).
- **Палитра в шейдер:** 15× (rot mat3 + pos vec3) юниформом/UBO, на
  инстанс на кадр.

### Майлстоны (каждый = сборка + твоя A/B визуальная сверка)

- **1A — трансформ+z, цвет проброшен.** Новый шейдер+путь за тумблером;
  GPU-скиннинг позиции, чистая проекция/viewport; **per-vertex цвет пока
  берём из старого CPU-расчёта** (лайтинг/туман идентичны *по
  построению*). VBO пока строим per-frame (изолируем корректность
  трансформа/z от вопроса времени жизни буфера). Гейт: силуэт/позиция/
  **перекрытия со стенами** 1:1 (тумблером).
- **1B — лайтинг в шейдер.** half-Lambert+рамп+tint в шейдере. Гейт:
  тона 1:1 (день/ночь/tint).
- **1C — туман в шейдер.** per-vertex по view-Z. Гейт: ночь/туман 1:1.
- **1D — постоянный + общий VBO.** Раскладка модели один раз,
  переиспользование между одинаковыми моделями; per-frame только палитра
  костей. Гейт: перф вверх + всё ещё 1:1. После подписания — новый путь
  дефолт, старый/тумблер удаляем.
- **1E — тени/отражения персонажей.** До 1D идут по старому пути; портим
  на новый (тот же шейдер) и сверяем. Решить: в 1D или отдельно.
  - ⚠️ **Эмпирика A/B 1A (2026-05-19):** при включённом GPU-пути тело
    персонажей местами ломалось, **а тени НЕ ломались**. Значит наше
    прежнее предположение «тени персонажей идут через тот же
    `ge_draw_multi_matrix`» **неверно** — тени рисуются отдельным путём
    (иначе ломались бы синхронно с телом). Тени тоже надо перевести на
    эту же скелетную/скиннинг-систему, но это **отдельная работа** со
    своим путём — найти где именно строится геометрия тени персонажа
    (проекция силуэта) и применить тот же подход. Не «тот же шейдер
    автоматически» — сначала разобрать реальный shadow-путь.

### Что НЕ в итерации 1

Авто-риг/веса (это итерация 2), атлас текстур (отдельный рычаг),
удаление старого пути (только после полного 1:1-подписания).
