## Зачем

Обработка ввода сейчас размазана по коду — часть в physics-тике
(`process_controls`, vehicle handlers, state-handler'ы персонажа), часть в
render-loop'е (`FRONTEND_input` для меню, `gamemenu.cpp`, `check_debug_timing_keys`).
У каждого места — своя логика edge-detect / auto-repeat / hold-detect. Из-за этого:

- **Latency-баги при низкой physics Hz** — input опрашивается раз в 50ms (на 20 Hz),
  быстрые нажатия проскакивают между тиками. Пример: #5 (toggle сирены), #15 (общая
  отзывчивость управления).
- **FPS-bound поведение при render-side обработке** — кнопка опрашивается per render
  frame без edge-detect, на unlimited FPS action триггерится несколько раз за нажатие.
  Пример: #20 (смена оружия).
- **Дублирование логики** — клавиатурный и геймпадный пути обрабатываются раздельно
  даже когда смысл нажатия идентичен. Меню имеет свою input-логику, игра — свою,
  debug-клавиши — третью.
- **Тайминги нажатий разные в разных местах** — фикс одного места не переносится
  в другое автоматически. Каждый раз когда всплывает похожий баг — его чинят
  отдельно, дублируя код.

Также сейчас нет нормальной поддержки переназначения управления — оно требует чтобы
игра не работала с конкретными кнопками напрямую, а это значит нужен слой actions.

## Что предлагается (общая идея)

Отдельный модуль управления (условно `input/`), работающий **на render-tick'е до
физики**. Готовит высокоуровневое состояние ввода и отдаёт его всем потребителям
— игре, меню, debug-handler'ам — единым API.

**Главный принцип:** input-сэмплинг идёт на wall-clock cadence (= render-frame),
не на physics-tick'е. Это согласуется с [`CORE_PRINCIPLE.md`](CORE_PRINCIPLE.md)
— физика читает уже готовое состояние или очередь событий, не сэмплит сама.

## Состав модуля

### 1. High-level state

Текущее состояние всех источников ввода:

- `is_pressed` для каждой логической кнопки/клавиши.
- float-значения adaptive triggers (L2/R2) с deadzone.
- vec2 для аналоговых стиков с deadzone и нормализацией.
- Активный источник ввода (Xbox controller / DualSense / keyboard) — определяется
  gamepad-системой по последнему активному устройству.

Обновляется каждый render-frame из raw HID/SDL событий, **до** физического тика.

### 2. Edge-detect (press registration)

Модуль сам решает когда считать клавишу "только что нажатой"
(`just_pressed`/`just_released`), хранит prev-state. Потребители читают готовый
флаг, не дублируют edge-detect логику.

Это закрывает класс багов где разные места хранят свои static prev-state'ы и
расходятся между собой (или забывают сбросить prev-state на смену контекста).

### 3. Press consumption

Потребитель может пометить press как отработанный (`consume()` / `mark_handled()`),
чтобы другие подписчики на ту же кнопку не сработали на тот же edge.

Закрывает классические баги типа "нажал ESC — и меню открылось, и пауза снялась
одновременно" — потребитель меню после открытия consume'ит ESC, и обработчик
паузы его уже не видит.

### 4. State reset

API сбросить is_pressed всех клавиш по событию:
- Переход между загрузочными экранами.
- Смена game state.
- После катсцены / окончания миссии.
- Любое место где "залипшее" нажатие не должно протекать в новый контекст.

Предотвращает баги когда игрок зажал кнопку перед сменой контекста и нажатие
ложно срабатывает в новом контексте.

### 5. Action abstraction

Логические actions (например `ACTION_JUMP`, `ACTION_FIRE`, `ACTION_MENU_CONFIRM`,
`ACTION_TOGGLE_SIREN`) с маппингом на физические кнопки клавы/геймпада.

**Игровой код всегда работает с actions, а не с конкретными `KB_SPACE` /
`GAMEPAD_BUTTON_X`** — это правило, а не приятный бонус. Конкретные клавиши
читаются только в исключениях (debug-клавиши `check_debug_timing_keys` и
подобное), не в основной игровой логике.

#### 5.1. Ремап централизован в модуле

Вся переназначаемость управления делается здесь: пользовательский конфиг → таблица
маппинга `action → physical input`. Никакого `if (Keys[KB_SPACE])` в игровом коде
— это сразу ломает ремап. Если игре что-то нужно от ввода — она спрашивает
по action.

#### 5.2. Дефолты

Для каждого action — по одному дефолту на каждый источник ввода:
- `ACTION_JUMP`: `Space` на клаве, `Cross/A` на геймпаде.
- И т.д. для остальных actions.

Дефолты подбираются под классическую схему оригинала.

#### 5.3. Один action — несколько источников

`ACTION_JUMP` отрабатывает одинаково при нажатии Space или Cross — игровая
обработка не различает. Это работает out-of-the-box без специальной логики
у потребителя.

#### 5.4. Аналоговые actions (только геймпад)

Для входов которых физически нет на клавиатуре:

- `ACTION_MOVE` — vec2 от левого стика, для аналогового перемещения персонажа.
- `ACTION_LOOK` — vec2 от правого стика, для камеры.
- `ACTION_ACCELERATE` / `ACTION_BRAKE` — float от R2/L2, для плавного газа/тормоза
  машины. На клаве этого нет — там только бинарное "нажато/нет".

В коде потребитель проверяет тип action и понимает что для клавиатурного
fallback'а (если он осмыслен) надо использовать boolean-аналог. Например, для
управления машиной с клавиатуры — Up/Down дают `ACTION_ACCELERATE = 1.0` /
`ACTION_BRAKE = 1.0` (без плавности), как в оригинале.

#### 5.5. Reverse lookup для подсказок UI

По action можно запросить какая физическая кнопка/клавиша на него сейчас
замаплена для активного источника ввода — и нарисовать корректную подсказку
в HUD/туториалах:

- "Нажмите [Cross] чтобы прыгнуть" — если активен DualSense.
- "Нажмите [A] чтобы прыгнуть" — если активен Xbox controller.
- "Нажмите [Space] чтобы прыгнуть" — если активна клавиатура.

Активный источник определяется gamepad-системой (есть понятие активного
устройства), модуль ввода держит этот контекст и отдаёт соответствующий
glyph/имя клавиши. Без этого пришлось бы хардкодить две версии подсказок и
переключать вручную.

#### 5.6. Контекстные группы actions

Actions делятся на контексты:

- **"На ходу"**: `ACTION_PUNCH`, `ACTION_KICK`, `ACTION_JUMP`, `ACTION_USE`,
  `ACTION_CRAWL`, `ACTION_SPRINT`, ...
- **"В машине"**: `ACTION_ACCELERATE`, `ACTION_BRAKE`, `ACTION_SIREN`,
  `ACTION_HORN`, `ACTION_EXIT_VEHICLE`, ...
- (возможно ещё контексты — меню, катсцены — определятся по необходимости)

Одна и та же физическая кнопка может быть замаплена на разные actions из
разных контекстов: R2 → `ACTION_PUNCH` на ходу и → `ACTION_ACCELERATE` в машине.
Это **не конфликт**, а нормальная конфигурация.

##### Переназначение делается per-action, не per-кнопка

Игрок ремапит уникальные игровые actions, не сырые кнопки. То что R2
одновременно триггерит и `ACTION_PUNCH` и `ACTION_ACCELERATE` — детальная
конфигурация которую игрок настраивает независимо в двух разных местах UI ремапа.

##### Чтение по активному контексту

Все actions могут обрабатываться параллельно (модуль ввода **не знает** в каком
состоянии игрок), но потребитель читает только релевантный для своего контекста:

- Код управления машиной читает `ACTION_ACCELERATE`.
- Код движения на ходу читает `ACTION_PUNCH`.

Какой контекст активен — определяет игровой код (он знает в машине игрок или
нет), не модуль ввода. Модуль просто отдаёт состояние всех actions, а игра
выбирает что слушать.

##### UI ремапа группирует actions по контекстам

Отдельные секции "На ходу" / "В машине" / ... в экране настроек управления.
Пользователь видит что R2 = удар на ходу и газ в машине, и ремапит каждый
action независимо. Это понятнее чем плоский список где одна кнопка вдруг
оказывается замаплена на несколько действий и непонятно почему.

#### 5.7. Split vs combined actions для контекстных кнопок

В оригинале одна кнопка обслуживает несколько разных действий — например
**use / crawl / sprint** делятся одной кнопкой, а приоритет какое именно
сработает определяется игровой логикой (по контексту: рядом с дверью →
`USE`, лежит → `CRAWL`, бежит и зажат → `SPRINT`).

Хочется дать игроку выбор:

- Завести **3 отдельных action**: `ACTION_USE`, `ACTION_CRAWL`, `ACTION_SPRINT`
  — каждый ремапится на свою кнопку.
- Плюс **один combined action**: `ACTION_USE_CRAWL_SPRINT` — поведение как в
  оригинале (одна кнопка, приоритет внутри игровой логики).

В настройках управления — переключатель:

- **Классическая схема** (один combined bind на одну кнопку, поведение оригинала).
- **Раздельная схема** (три отдельных bind'а на три кнопки, для тех кто хочет
  явный контроль).

Выбирает сам игрок, а не игра решает за него. Сейчас в оригинале это всё одна
кнопка с приоритетом — игрок не может разделить.

**Технически это значит:** игровой код проверяет либо combined action, либо
сразу три раздельных — в зависимости от того что у пользователя в настройках.
Альтернатива: модуль ввода это абстрагирует и эмитит правильный набор
`is_pressed` actions на основе схемы (тогда игровой код всегда читает три
отдельных, а combined-режим внутри модуля просто триггерит все три при
нажатии combined-кнопки + полагается на приоритет в игре).

### 6. (опционально) Event queue для физики

Между physics-тиками копится список input-событий (`just_pressed`/`released`
с таймстампом). Физика на следующем тике может прочитать всю очередь и
независимо отработать каждое событие, не теряя нажатий проскочивших между
тиками. Очередь чистится после physics-тика.

Нужно только если выяснится что какие-то gameplay-actions реально требуют
per-event обработки в физике (а не просто "состояния на момент тика"). Для
большинства входов достаточно "is_pressed на момент physics-тика" + edge-detect.

## Универсальность

Один и тот же модуль обслуживает меню и игру. Тайминги нажатий, edge-detect,
auto-repeat — всё определяется в одном месте. Не будет "в меню пролистывание
через свой `dir_next_fire`, в игре через свой счётчик".

То же касается debug-клавиш — они идут через тот же модуль (но с пометкой
"debug-only", чтобы не светились в UI ремапа и не считались игровыми actions).

## Что закрывает напрямую

- **#5** (toggle сирены) — edge-detect и/или event queue решает регистрацию
  между physics-тиками независимо от Hz. Помечено как won't-fix для случая
  низких physics Hz (это debug-режим), но архитектурно решается тут же.
- **#15** (отзывчивость на 20 Hz) — state читается на каждом render-frame,
  физика видит свежее состояние или очередь событий, а не последний sample
  раз в 50ms.
- **#20** (смена оружия FPS-bound) — edge-detect в input-слое триггерит ровно
  одно событие на нажатие независимо от render rate.
- **Будущие input-баги аналогичной природы** — фиксятся в одном месте, а не
  размазываются точечными правками по `vehicle.cpp` / `panel.cpp` /
  `gamemenu.cpp`.

## Где сейчас обработка ввода (для понимания scope миграции)

- `process_controls` в physics-тике — player movement, weapon, vehicle controls.
- `FRONTEND_input` в render-loop'е — меню навигация, auto-repeat (`dir_next_fire`).
- `gamemenu.cpp` — in-game pause menu.
- `check_debug_timing_keys` — debug-клавиши с edge-detect через static prev-state.
- Распределённые места в state-handler'ах персонажа — combo, weapon-switch.
- Vehicle controls — отдельная обработка для машины.

## Подход к миграции

Поэтапная. Сначала ввести модуль и actions для самых проблемных входов:
1. Movement (стик / WASD) — критично для отзывчивости.
2. Weapon switch — закрывает #20.
3. Siren toggle — закрывает архитектурную часть #5.

Потом постепенно мигрировать остальное:
- Combat actions (punch / kick / jump / use).
- Меню (FRONTEND_input).
- In-game pause menu.
- Debug-клавиши (последние, можно оставить как есть на время).

**Не пытаться перевернуть всю обработку ввода одним коммитом** — слишком
большой scope, легко наломать. Каждая фаза — отдельный коммит с проверкой
что ничего не сломалось.

## Связь с принципом задачи

[`CORE_PRINCIPLE.md`](CORE_PRINCIPLE.md) — "вся логика и отрисовка зависят
только от wall-clock". Это применимо и к input-сэмплингу:

- Ввод опрашивается на render-frame cadence (= wall-clock).
- Edge-detect и состояние тоже на render-frame.
- Физика читает уже готовое состояние или очередь событий — не сэмплит сама.

Этот модуль — естественное расширение принципа на input-слой.

## Приоритет

**Высокий, нужно до 1.0.** Закрывает класс багов разом, без него точечные
фиксы продолжат всплывать в новых местах. Также блокирует фичу
переназначения управления (без actions её делать неудобно — пришлось бы
ремапить сырые кнопки и подкладывать костыли в каждое место чтения).
