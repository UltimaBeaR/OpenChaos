## [2026-03-15] Первый запуск — настройка билда и найденные проблемы

**Стадия:** между Этапом 1 (форматирование, завершён) и Этапом 2 (удаление мёртвого кода).
Не планировалось заранее — возникло при первой попытке собрать и запустить проект из командной строки.

### Что сделано: Makefile в корне проекта

Добавлен `Makefile` для запуска сборки и игры без открытия Visual Studio.

Команды:
- `make build-debug` / `make build-release` — полный ребилд соответствующей конфигурации
- `make run-debug` / `make run-release` — запуск игры

Всегда используется `-t:Rebuild` (полный ребилд). Инкрементальный билд намеренно не используется
— при массовом рефакторинге он глючит.

Технические нюансы при настройке:

- **Флаги MSBuild через bash:** нужно `-p:` (дефис), а не `/p:` — bash интерпретирует `/p:` как путь к файлу.
- **vcpkg DLL-ки не копировались:** VS при сборке через IDE автоматически копирует `SDL2.dll`,
  `OpenAL32.dll`, `fmt.dll` через своё расширение vcpkg. MSBuild из командной строки этого не делает,
  потому что интеграция прописана в `%LOCALAPPDATA%\vcpkg\vcpkg.user.targets` — файл подгружается
  VS как IDE-расширение, а не через стандартный MSBuild user.props (которого на машине нет).
  Решение: передать vcpkg targets напрямую через `-p:CustomAfterMicrosoftCppTargets`.
- **Запуск exe из bash:** `./Fallen.exe` не работает — bash пытается запустить Windows PE как ELF.
  Используется `Start-Process` через PowerShell с явным `WorkingDirectory`.

### Что найдено: проблемы в Fallen.vcxproj

#### Release не собирался — C2338 в winnt.h (ИСПРАВЛЕНО)

Ошибка у каждого файла:
```
static_assert(__alignof(LARGE_INTEGER) == 8,
    "Windows headers require the default packing option...")
```

Причина: оба конфига используют `/Zp1` (выравнивание структур по 1 байту — нужно для чтения
бинарных форматов ресурсов). Windows SDK 10.0.26100 добавил `static_assert` который это запрещает.
В Debug конфиге `WINDOWS_IGNORE_PACKING_MISMATCH` был прописан (подавляет assert), в Release — нет.
Это просто несоответствие между конфигами, не намеренное решение.

Фикс: добавлен `WINDOWS_IGNORE_PACKING_MISMATCH` в Release PreprocessorDefinitions в `Fallen.vcxproj`.

**Известный риск `/Zp1` (не исправлялся, отложен на Этап 4+):** WinAPI-структуры
(`LARGE_INTEGER`, `FILETIME`, `OVERLAPPED` и др.) при `/Zp1` упакованы по 1 байту вместо
ожидаемых 4/8 — потенциальный memory corruption при WinAPI вызовах. Этот риск одинаков
в обеих конфигурациях и существовал до наших правок. Правильный долгосрочный фикс —
отделить пакованные структуры ресурсов от WinAPI-кода через `#pragma pack`.

#### Подозрительные флаги — расследование завершено (2026-03-15)

**`/MTd` в Release (`MultiThreadedDebug`) — подтверждённая проблема:**

`MultiThreadedDebug` (строка 63 в `Fallen.vcxproj`) автоматически определяет `_DEBUG` компилятором.
Из-за этого в Release-сборке активны все `#ifdef _DEBUG` блоки, в т.ч. отладочная надпись
`"music mode %d vol %f %s"` в `DDEngine/Source/panel.cpp:5441`.

Все `/D /NODEFAULTLIB:libcmtd.lib"` в `<AdditionalOptions>` линкера (строки 303–447 vcxproj) —
костыли, добавленные чтобы компенсировать конфликт: Release-код линкуется с отладочным CRT
(`libcmtd.lib`), что вызывает ошибки линковки. Артефакт кривой конвертации `.dsp` → `.vcxproj`.

**Что нужно исправить в `Fallen.vcxproj`:**
- `RuntimeLibrary`: `MultiThreadedDebug` → `MultiThreaded` в Release конфиге (строка 63)
- Убрать все `/D /NODEFAULTLIB:libcmtd.lib"` из `<AdditionalOptions>` линкера Release конфига

**Отдельный баг — `FARFACET squares drawn` (не связан с `/MTd`):**

В `DDEngine/Source/panel.cpp:5408` блок с `"FARFACET squares drawn: %d"` обёрнут только в
`#ifndef TARGET_DC` (активен на PC), без какой-либо защиты `#ifdef _DEBUG` или `#ifndef FINAL`.
Показывается в **любой** сборке — и Debug, и Release.
Это оригинальный пре-релизный баг: в финальной игре этой надписи нет.
Фикс: добавить `#ifndef FINAL` вокруг блока.

## Находки из внешних источников (dizzy2003/MuckyFoot-UrbanChaos issues#5)

Источник: https://github.com/dizzy2003/MuckyFoot-UrbanChaos/issues/5

### Альтернативный подход к `/Zp` — отключение вместо WINDOWS_IGNORE_PACKING_MISMATCH

TracerDS (март 2025) решил проблему сборки Release иначе: полностью отключил `/Zp`
(Struct Member Alignment → Default в настройках проекта).

**Почему наш подход лучше:** отключение `/Zp` сломало бы чтение бинарных форматов ресурсов —
весь код парсинга `.ucm`, `.spd` и т.п. завязан на структуры с `/Zp1`. Мы оставили `/Zp1`
и добавили `WINDOWS_IGNORE_PACKING_MISMATCH` только для подавления `static_assert` в Windows SDK.

### "Unable to open display" при первом запуске — НЕ ошибка сборки

При первом запуске игра может упасть с сообщением **"Unable to open display"**.

Причина: игра ищет разрешение из `config.ini` в захардкоженном списке поддерживаемых разрешений,
и при несовпадении нет fallback. Если `config.ini` отсутствует, он создаётся автоматически
с `video_res=1` — слишком низкое разрешение которого нет в списке.

**Фикс:** создать/отредактировать `config.ini` в директории с `Fallen.exe`, выставить `video_res=4`.

**Дополнительный риск:** если путь к exe превышает 255 символов, `config.ini` не читается вообще.
При разработке держать проект в коротком пути.

### `do_only_game_intro` — функция без тела

PieroZ обнаружил, что функция `do_only_game_intro` объявлена, но тела нигде нет.
При компиляции с `EDITOR` директивой это проявляется как ошибка линковки.
В обычной игровой сборке (без `EDITOR`) не мешает, но стоит иметь в виду при анализе кода.
