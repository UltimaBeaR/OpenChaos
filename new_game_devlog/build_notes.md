### Что сделано: Makefile в корне проекта

Добавлен `Makefile` для запуска сборки и игры без открытия Visual Studio.

Команды:
- `make build-debug` / `make build-release` — полный ребилд соответствующей конфигурации
- `make run-debug` / `make run-release` — запуск игры

Всегда используется `-t:Rebuild` (полный ребилд). Инкрементальный билд намеренно не используется
— при массовом рефакторинге он глючит.

Технические нюансы при настройке:

- **Флаги MSBuild через bash:** нужно `-p:` (дефис), а не `/p:` — bash интерпретирует `/p:` как путь к файлу.
- **vcpkg DLL-ки не копировались:** VS при сборке через IDE автоматически копирует `SDL2.dll`,
  `OpenAL32.dll`, `fmt.dll` через своё расширение vcpkg. MSBuild из командной строки этого не делает,
  потому что интеграция прописана в `%LOCALAPPDATA%\vcpkg\vcpkg.user.targets` — файл подгружается
  VS как IDE-расширение, а не через стандартный MSBuild user.props (которого на машине нет).
  Решение: передать vcpkg targets напрямую через `-p:CustomAfterMicrosoftCppTargets`.
- **Запуск exe из bash:** `./Fallen.exe` не работает — bash пытается запустить Windows PE как ELF.
  Используется `Start-Process` через PowerShell с явным `WorkingDirectory`.

### Что найдено: проблемы в Fallen.vcxproj

#### Release не собирался — C2338 в winnt.h (ИСПРАВЛЕНО)

Ошибка у каждого файла:
```
static_assert(__alignof(LARGE_INTEGER) == 8,
    "Windows headers require the default packing option...")
```

Причина: оба конфига используют `/Zp1` (выравнивание структур по 1 байту — нужно для чтения
бинарных форматов ресурсов). Windows SDK 10.0.26100 добавил `static_assert` который это запрещает.
В Debug конфиге `WINDOWS_IGNORE_PACKING_MISMATCH` был прописан (подавляет assert), в Release — нет.
Это просто несоответствие между конфигами, не намеренное решение.

Фикс: добавлен `WINDOWS_IGNORE_PACKING_MISMATCH` в Release PreprocessorDefinitions в `Fallen.vcxproj`.

**Известный риск `/Zp1` (не исправлялся, отложен на Этап 4+):** WinAPI-структуры
(`LARGE_INTEGER`, `FILETIME`, `OVERLAPPED` и др.) при `/Zp1` упакованы по 1 байту вместо
ожидаемых 4/8 — потенциальный memory corruption при WinAPI вызовах. Этот риск одинаков
в обеих конфигурациях и существовал до наших правок. Правильный долгосрочный фикс —
отделить пакованные структуры ресурсов от WinAPI-кода через `#pragma pack`.

#### Подозрительные флаги (зафиксированы, не трогались)

- `/MTd` в Release — Debug CRT в релизной сборке. Должно быть `/MT`.
- `/D /NODEFAULTLIB:libcmtd.lib"` в обоих конфигах — `/NODEFAULTLIB` это линкерный флаг,
  случайно предварён `/D` (флаг дефайна). Артефакт кривой конвертации старого `.dsp` → `.vcxproj`.
  Фактического эффекта скорее всего нет — линкер получает его отдельно.
