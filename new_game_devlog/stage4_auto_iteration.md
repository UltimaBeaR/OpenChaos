# Stage 4 — Автономный процесс итераций миграции

## Это ручной процесс, запускаемый явной командой

Этот процесс **не запускается автоматически**. Ты запускаешь его только когда я явно прошу —
примерно так: *"запусти автономный процесс итераций этапа 4"* или *"глянь stage4_auto_iteration.md
и запустись по этому процессу"*.

Процесс предназначен для многочасовой непрерывной работы без участия пользователя.
Каждая итерация миграции выполняется в отдельном субагенте (свежий контекст),
между итерациями делается git commit. Процесс идёт до исчерпания DAG или до нерешаемого блокера.

---

## Предусловие: проверить права на git commit

**До запуска первой итерации** — обязательно проверь, есть ли у тебя права на `git commit`.

Как проверить: попробуй выполнить `git status` и посмотри есть ли в настройках проекта
разрешение на git-операции. Практически — попробуй запустить тестовый коммит или просто
выполни `git commit --dry-run` и посмотри не заблокировано ли это.

Если прав нет — **не начинай процесс**, а сообщи пользователю:

> "Для запуска автономного процесса итераций нужны права на git commit.
> Включи их в настройках проекта Claude Code, после этого скажи мне и я начну."

Права нужны потому что коммит между итерациями — не опциональная вещь: ревью следующей
итерации работает по git working changes, и они должны содержать только изменения текущей
итерации, а не накопленные изменения всех предыдущих.

---

## Архитектура процесса

```
Основной агент (ты, в текущем чате)
│
├── Проверяет права на git commit
├── Цикл:
│   ├── Запускает субагент (свежий контекст)
│   │   └── Субагент делает одну полную итерацию (см. ниже)
│   │
│   ├── Получает результат от субагента:
│   │   ├── SUCCESS  → git commit → следующая итерация
│   │   ├── DONE     → нечего мигрировать, процесс завершён
│   │   └── BLOCKED  → не коммитить, ждать пользователя
│   │
│   └── (повторить)
```

---

## Что делает субагент (одна итерация)

Субагент запускается с полным самодостаточным промптом (см. раздел ниже).
Он выполняет следующее по порядку:

### Шаг 1 — Ориентация
- Читает `CLAUDE.md`
- Читает `new_game_planning/stages.md` — понять текущий этап
- Читает `new_game_planning/stage4.md` (или аналогичный) — детали Этапа 4
- Опционально: `new_game_devlog/stage4_log.md` — если нужен контекст прошлых итераций

### Шаг 2 — Миграция
- Вызывает скилл `stage4-migrate` (он сам определяет по DAG что мигрировать следующим)
- Выполняет перенос кода согласно правилам скилла:
  - `// uc_orig:` комментарии для каждой перенесённой сущности
  - Запись в `entity_mapping.json`
  - Правильные include guards, CMake, удаление оригинальных комментариев в `new/`

### Шаг 3 — Компиляция
- Запускает `make build-release` в корне проекта
- Если ошибка:
  - Пробует исправить самостоятельно
  - Если исправил — компилирует снова, убеждается что всё чисто
  - **Обязательно пишет в `new_game_devlog/stage4_log.md`** что была ошибка и как исправил
  - Если исправить не получается — переходит к состоянию BLOCKED (см. ниже)

### Шаг 4 — Ревью
- Вызывает скилл `review` (общая самопроверка по git working changes)
- Вызывает скилл `stage4-review` (специфичные проверки Этапа 4)
- Если ревью выявило проблемы — исправляет их, перекомпилирует
- Если исправить не получается — переходит к состоянию BLOCKED

### Шаг 5 — Отчёт
Субагент возвращает один из трёх статусов:

**SUCCESS:**
```
STATUS: SUCCESS
Перенесено: <список файлов/сущностей>
Компиляция: OK
Ревью: OK
Примечания: <если были исправления по ходу>
```

**DONE (нечего мигрировать):**
```
STATUS: DONE
Все сущности в DAG уже перенесены. Этап 4 завершён.
```

**BLOCKED:**
```
STATUS: BLOCKED
Причина: <подробное описание блокера>
Что сделано: <что успело перенестись до блокера, если что-то было>
Что не закоммичено: <список изменённых файлов>
Требуется: <что нужно от пользователя>
```

---

## Что делает основной агент после получения результата субагента

### При SUCCESS:
```bash
git add -A
git commit -m "stage 4 - iteration (auto commit by ai)"
```
Затем сразу запускает следующий субагент.

### При DONE:
Сообщает пользователю: "Этап 4 завершён, все сущности перенесены. Последняя итерация закоммичена."

### При BLOCKED:
- **Не делает git commit** (изменения остаются в working tree)
- Сообщает пользователю подробности блокера (из отчёта субагента)
- Ждёт пользователя

---

## Промпт для субагента

Основной агент запускает каждый субагент с примерно таким промптом
(адаптируй по ситуации, но суть должна оставаться):

```
Ты выполняешь одну итерацию миграции Stage 4 проекта OpenChaos.
Рабочая директория: c:/WORK/OpenChaos

Прочитай CLAUDE.md чтобы понять проект и правила.
Затем выполни полную итерацию:

1. Ориентируйся: прочитай new_game_planning/stages.md и new_game_devlog/stage4_log.md (опционально)
2. Мигрируй: вызови скилл stage4-migrate (он определит что переносить)
3. Скомпилируй: make build-release. При ошибке — исправь и запиши в stage4_log.md
4. Ревью: вызови скиллы review и stage4-review
5. Верни отчёт в формате:

STATUS: SUCCESS | DONE | BLOCKED
(и детали согласно формату из stage4_auto_iteration.md)

Правила:
- Ничего не спрашивай у пользователя в процессе
- Принимай решения самостоятельно
- Если блокер нерешаем — верни BLOCKED с подробным описанием
- Все важные события (ошибки компиляции, нестандартные решения) пиши в stage4_log.md
```

---

## Важные ограничения и нюансы

- **Коммит делает только основной агент**, не субагент. Субагент только выполняет работу.
- **Ревью работает по git working changes** — поэтому коммит между итерациями обязателен.
  Без коммита ревью следующей итерации будет видеть накопленные изменения всех итераций,
  а не только текущей.
- **Субагент не должен ничего спрашивать** у пользователя в процессе работы. Все решения —
  самостоятельно. Вопросы к пользователю только через статус BLOCKED в финальном отчёте.
- **Нерешаемый блокер** — это ситуация когда субагент попробовал несколько подходов и все
  провалились, или когда требуется архитектурное решение выходящее за рамки текущей итерации.
  Обычные ошибки компиляции субагент должен уметь исправлять сам.
- **Девлог обязателен** при ошибках компиляции — даже если исправил сам. Это нужно чтобы
  пользователь мог ретроспективно понять что происходило.
