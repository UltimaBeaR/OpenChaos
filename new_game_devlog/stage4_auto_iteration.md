# Stage 4 — Автономный процесс итераций миграции заголовков

## Это ручной процесс, запускаемый явной командой

Этот процесс **не запускается автоматически**. Ты запускаешь его только когда пользователь явно
просит — примерно так: *"запусти автономный процесс"* или *"глянь stage4_auto_iteration.md
и запустись по этому процессу"*.

Процесс предназначен для многочасовой непрерывной работы без участия пользователя.
Каждая итерация выполняется в отдельном субагенте (свежий контекст),
между итерациями делается git commit. Процесс идёт до исчерпания работы или до нерешаемого блокера.

---

## Текущая задача (Этап 4.0, фаза заголовков)

Из `src/fallen/` и `src/MFStdLib/` нужно устранить все `// Temporary:` включения.

**Что делает каждая итерация:**
1. Берёт 1–3 тематически связанных legacy-заголовка из `src/fallen/` или `src/MFStdLib/`
   которые ещё содержат настоящий контент (не редиректы)
2. Переносит их сущности в правильные заголовки новой структуры
3. Конвертирует legacy-заголовок в редирект `#include "new/path/file.h"`
4. Обновляет все потребители — заменяет `// Temporary:` включение на прямое новое
5. Компилирует, делает ревью

**Критерий DONE:** `grep -r "// Temporary:" src/` возвращает 0 результатов
И нет legacy-заголовков с настоящим контентом (только редиректы или пустые).

---

## Предусловие: проверить права на git commit

**До запуска первой итерации** — проверь, есть ли права на `git commit`.
Выполни `git status`. Если git-операции заблокированы — сообщи пользователю:

> "Для запуска автономного процесса нужны права на git commit.
> Включи их в настройках проекта Claude Code, после этого скажи мне и я начну."

Права нужны потому что ревью следующей итерации работает по git working changes — они должны
содержать только изменения текущей итерации, не накопленные.

---

## Архитектура процесса

```
Основной агент (ты, в текущем чате)
│
├── Проверяет права на git commit
├── Цикл:
│   ├── Запускает субагент (свежий контекст, изолированная итерация)
│   │
│   ├── Получает результат от субагента:
│   │   ├── SUCCESS  → git commit → следующая итерация
│   │   ├── DONE     → нечего мигрировать, процесс завершён
│   │   └── BLOCKED  → не коммитить, ждать пользователя
│   │
│   └── (повторить)
```

## Что делает субагент (одна итерация)

### Шаг 1 — Ориентация

- Читает `CLAUDE.md`
- Читает `.claude/skills/stage4-migrate/SKILL.md` — правила миграции (обязательно!)
- Читает `new_game_devlog/stage4_log.md` — последний номер итерации и что делали
- Выясняет текущее состояние:
  ```bash
  # Сколько Temporary осталось
  grep -r "// Temporary:" new_game/src/ --include="*.cpp" --include="*.h" | wc -l

  # Какие legacy-заголовки ещё с контентом (не редиректы)
  # (заголовок — редирект если в нём после #ifndef/#define/#endif осталось только #include)
  ```

### Шаг 2 — Выбор батча

Выбрать 1–3 legacy-заголовка для этой итерации. Критерии выбора:
- Небольшие (100–300 строк предпочтительнее)
- Тематически связанные (например, все про AI, или все про environment)
- Не умбреллы (`Game.h` — отдельная итерация, см. ниже)
- DAG-порядок: сначала то, от чего ничто не зависит

**Особый случай — `Game.h`:** это умбрелла-заголовок (24 потребителя). Её не надо мигрировать
как обычный заголовок — нужно каждому потребителю прописать конкретные новые заголовки вместо
неё, и убрать Game.h из includes. Это отдельная итерация только для Game.h.

### Шаг 3 — Миграция

Следовать правилам из `.claude/skills/stage4-migrate/SKILL.md` строго:

1. Прочитать legacy-заголовок — выписать все сущности
2. Для каждой сущности решить куда она идёт (см. критерии размещения в SKILL.md)
3. Проверить `entity_map.py find --name X` — не мигрирована ли уже
4. Перенести сущности в новые заголовки с `// uc_orig:` комментариями
5. Конвертировать legacy-заголовок в редирект
6. Обновить потребителей (grep перед заменой!)
7. Добавить записи в entity mapping (bulk python script если сущностей много)

**Обязательно проверять транзитивные зависимости** после batch-замены:
если файл терял `game.h` через цепочку legacy-заголовков — после замены сломается.
Для каждого сломавшегося файла: открыть оригинал в `original_game/` и сравнить includes.

### Шаг 4 — Компиляция

```bash
make build-release
```

Если ошибка:
- Исправить самостоятельно
- Скомпилировать снова до чистого результата
- Записать в `stage4_log.md` что была ошибка и как исправил
- Если исправить не получается → BLOCKED

### Шаг 5 — Ревью

- Применить общие проверки из `.claude/skills/review/SKILL.md`
- Применить проверки A–H из `.claude/skills/stage4-review/SKILL.md`
- Если нашлись проблемы — исправить, перекомпилировать
- Если исправить не получается → BLOCKED

### Шаг 6 — Обновить лог

Добавить запись в `new_game_devlog/stage4_log.md`:
- Номер итерации (предыдущий + 1)
- Только нетривиальные заметки (ошибки компиляции, нестандартные решения, конфликты имён)
- **Не писать:** списки перенесённых сущностей, статус компиляции, размеры файлов

### Шаг 7 — Отчёт

**SUCCESS:**
```
STATUS: SUCCESS
Перенесено: <какие заголовки, коротко>
Temporary до/после: <было N, стало M>
Примечания: <если были исправления, нестандартные решения>
```

**DONE:**
```
STATUS: DONE
Все // Temporary: включения устранены. Этап 4.0 завершён.
```

**BLOCKED:**
```
STATUS: BLOCKED
Причина: <подробное описание блокера>
Что сделано: <что успело перенестись до блокера>
Что не закоммичено: <список изменённых файлов>
Требуется: <что нужно от пользователя>
```

## Что делает основной агент после получения результата

### При SUCCESS:
```bash
git add new_game/src/ new_game_devlog/ new_game_planning/
git commit -m "stage 4 - iteration (auto commit by ai)"
```
Затем сразу запускает следующий субагент.

### При DONE:
Сообщает пользователю: "Этап 4.0 завершён — все // Temporary: включения устранены."

### При BLOCKED:
- **Не делает git commit**
- Сообщает пользователю подробности блокера
- Ждёт пользователя

## Промпт для субагента

Основной агент запускает каждый субагент с таким промптом:

```
Ты выполняешь одну итерацию миграции заголовков Stage 4 проекта OpenChaos.
Рабочая директория: c:/WORK/OpenChaos

Задача: устранить // Temporary: включения путём переноса контента legacy-заголовков
из src/fallen/ и src/MFStdLib/ в новую структуру src/.

Порядок работы:
1. Прочитай CLAUDE.md
2. Прочитай .claude/skills/stage4-migrate/SKILL.md — это твои правила миграции
3. Прочитай new_game_devlog/stage4_log.md — последний номер итерации
4. Выясни текущее состояние:
   - grep -r "// Temporary:" new_game/src/ --include="*.cpp" --include="*.h" | wc -l
   - Какие заголовки в src/fallen/ ещё с настоящим контентом
5. Выбери батч (1–3 связанных заголовка, не Game.h — она отдельно)
6. Выполни миграцию по правилам SKILL.md (uc_orig, entity mapping, редиректы, обновление потребителей)
7. make build-release — при ошибке исправь, запиши в stage4_log.md
8. Проверь по .claude/skills/review/SKILL.md и .claude/skills/stage4-review/SKILL.md
9. Запиши итерацию в stage4_log.md
10. Верни отчёт:

Перенесено: <заголовки>
Temporary до/после: <N → M>
Примечания: <нетривиальные события>

Правила:
- Ничего не спрашивай у пользователя в процессе
- Принимай решения самостоятельно
- Если блокер нерешаем — верни BLOCKED с подробным описанием
- Game.h — умбрелла с 24 потребителями, обрабатывается отдельной итерацией
```

## Важные ограничения

- **Коммит делает только основной агент**, не субагент
- **Ревью работает по git working changes** — коммит между итерациями обязателен
- **Субагент не спрашивает пользователя** — все решения самостоятельно, вопросы только через BLOCKED
- **Девлог обязателен при ошибках** — даже если исправил сам
- **Game.h обрабатывается отдельно** — не как обычный заголовок, а как замена умбреллы
  на конкретные включения в каждом из 24 потребителей
