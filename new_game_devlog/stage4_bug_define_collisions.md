# Пересечения `#define` макросов проекта со стандартной библиотекой

**Обнаружено:** 2026-03-22 (при расследовании бага с кнопкой C).

---

## Баг: кнопка C — анимация подбора предмета вместо приседания (конфликт `#define INFINITY`)

**Обнаружен:** 2026-03-22. **Коммит появления:** `92d90a1` (миграция dirt.cpp из `old/` в `new/`).
**Git-тэг:** `stage4_bug_C_button_stealth_mode_broken`.

**Симптом:** При нажатии кнопки C стоя на месте Дарси выполняет анимацию поднятия
предмета с пола вместо того чтобы присесть (crouch/crawl).
Воспроизводится **только в Release**, в Debug — работает корректно.

### Как обнаружили причину

Через `git bisect` нашли коммит `92d90a1` — миграция `dirt.cpp` из `old/` в `new/`.
Код функций идентичен оригиналу — логических изменений нет. Добавили отладочные
`PANEL_new_text` в `do_an_action()` (interfac.cpp) — подтвердили что срабатывает
путь подбора банки (coke can): `DIRT_get_nearest_can_or_head_dist()` возвращает
`dist=0` (банка якобы прямо под ногами) хотя банки рядом нет.

Проверили в Debug — баг не воспроизводится. Добавили `MSG_add` внутрь функции —
баг пропал и в Release (вызов функции не даёт оптимизатору агрессивно оптимизировать
цикл). Это классический признак **undefined behavior**.

Через препроцессор (`clang++ -E`) обнаружили что `INFINITY` раскрывается не в
`0x7fffffff` (наше определение из `MFStdLib.h` / `core/macros.h`), а в
`((float)(1e+300))` — определение из `<math.h>` стандартной библиотеки C.

### Причина: конфликт макроса `INFINITY`

Оригинальный код (`MFStdLib.h`) определяет:
```cpp
#define INFINITY 0x7fffffff   // целое число, INT_MAX — используется как "бесконечное расстояние"
```

Стандартная библиотека C (`<math.h>`) тоже определяет `INFINITY`:
```cpp
#define INFINITY ((float)(1e+300))   // float положительная бесконечность
```

В `core/macros.h` есть `#undef INFINITY` + `#define INFINITY 0x7fffffff` — попытка
переопределить стандартный макрос. Но это работает только если `macros.h` включается
ПОСЛЕ `<math.h>`. Если **любой** хедер в цепочке include подтягивает `<math.h>` ПОЗЖЕ —
float-версия затирает нашу.

**Почему проявилось при миграции dirt.cpp:** в `old/` файлах цепочка include другая и
`<math.h>` либо не включается, либо включается до `MFStdLib.h`. При переносе в `new/`
порядок include изменился — `<math.h>` стал включаться через другие хедеры ПОСЛЕ
`core/macros.h`, и float-версия `INFINITY` перезатёрла нашу.

**Результат в коде:**
```cpp
SLONG best_dist = INFINITY;
// препроцессор раскрывает в:
SLONG best_dist = ((float)(1e+300));   // float +inf → SLONG = undefined behavior
```

Присвоение float +infinity в SLONG — **UB** по стандарту C++:
- **Debug:** компилятор генерирует честную конвертацию, x86 инструкция `cvttss2si`
  даёт INT_MIN или насыщение → функция случайно работает корректно
- **Release:** оптимизатор имеет право предполагать что UB не происходит →
  `best_dist` инициализируется в произвольное значение (наблюдалось 0) →
  функция `DIRT_get_nearest_can_or_head_dist()` возвращает 0 →
  `do_an_action()` считает что банка прямо под ногами →
  вызывает `set_person_can_pickup()` вместо `set_person_croutch()`

**Проблема с оптимизатором — это НЕ отдельный баг.** Это следствие UB. После
устранения UB (правильный `#define`) оптимизатор будет работать корректно,
разницы между Debug и Release не будет.

### Масштаб проблемы

`INFINITY` используется в **~100 местах** в `new/`. Все файлы где `<math.h>` попадает
в цепочку include — **потенциально сломаны** в Release (UB при инициализации).

Это **не единственный** возможный конфликт: `MFStdLib.h` определяет и другие макросы
с короткими общими именами которые могут совпадать с именами из стандартной библиотеки.
→ Аудит ниже.

**Статус:** ⚠️ причина найдена и подтверждена, фикс ожидает выполнения аудита и переименования.

## Аудит: все пересечения `#define` между `original_game/` и стандартными хедерами

**Суть проблемы:** Оригинальная кодовая база определяет макросы с именами,
совпадающими с именами из стандартной библиотеки C/C++ и Windows SDK. При компиляции
с современным clang + стандартными хедерами макросы могут перезатираться, вызывая UB
и сломанную логику. Проблема проявляется при миграции файлов из `old/` в `new/`,
т.к. в `new/` другой порядок include.

**Решение:** все пересекающиеся макросы переименовываются с префиксом `UC_`
(Urban Chaos). Пример: `INFINITY` → `UC_INFINITY`. Префикс `UC_` не используется
нигде в оригинальной кодовой базе (проверено).

### Методика

1. Вытащили все `#define ИМЕНА` из `original_game/` через `grep` — **8806** уникальных имён.
   Из поиска исключены vendored хедеры (`original_game/fallen/vcpkg_installed/`) —
   они являются частью сторонних библиотек (SDL2 и т.п.), а не кодом проекта.
   Без этого исключения в список попадали ложные пересечения,
   которые определены в SDL2 хедерах и естественно совпадают со стандартом.
   Исключённые vcpkg-макросы проверены отдельно → Группа D ниже.
2. Вытащили все макросы из стандартных хедеров через `clang++ -dM -E` (включили
   `<math.h>`, `<stdlib.h>`, `<stdio.h>`, `<limits.h>`, `<float.h>`, `<errno.h>`,
   `<stdint.h>`, `<stddef.h>`, `<stdbool.h>`, `<signal.h>`, `<time.h>`, `<locale.h>`,
   `<setjmp.h>`, `<stdarg.h>`, `<wchar.h>`, `<wctype.h>`, `<windows.h>` и C++
   обёртки `<cmath>`, `<cstdlib>` и т.д.) — **35092** имени.
3. Пересекли два списка — **18 совпадений** (ниже).

#### Группа C — `#define` только в PSX-хедерах (но макросы используются и в PC-коде)

`#define` этих макросов стоит только в PSX-хедерах (`PSXENG/`, `psxlib1/`), потому что
на PSX нет Windows SDK. PC-код получает эти макросы из Windows SDK (`<windef.h>` и т.п.).
Значения совпадают с SDK (кроме `MAX_PATH`: PSX определяет `128`, SDK — `260`).
Сами макросы активно используются в PC-коде: `HIWORD`/`LOWORD` — в GMouse, DIManager;
`MAX_PATH` — в AutoRun, Director, io.cpp и др.

| # | Макрос | `#define` в | Значение в проекте | Источник в стандарте/SDK |
|---|--------|------------|-------------------|------------------------|
| 15 | `HIWORD` | `fallen/PSXENG/headers/psxeng.h` | `((l)>>16)` (совпадает с SDK) | `<windef.h>` |
| 16 | `LOWORD` | `fallen/PSXENG/headers/psxeng.h` | `((l)&0xffff)` (совпадает с SDK) | `<windef.h>` |
| 17 | `MAX_PATH` | PSX-код (4 файла) | `128` (SDK: `260`) | `<windef.h>` |
| 18 | `_WIN32` | `MFStdLib.h`, `MFHeader.h` (PSX-хедеры) | Пустой define (флаг, совпадает) | Compiler-defined |

#### Группа D — vcpkg/SDL2 (исключены из основного поиска, проверены отдельно)

Эти макросы определяются в vendored SDL2 хедерах (`original_game/fallen/vcpkg_installed/`),
а не в коде проекта. Исключены из основного списка. Проверено, используются ли в коде игры:

| Макрос | Используется в игре? | Примечание |
|--------|---------------------|-----------|
| `NULL` | Да, ~3956 раз | Значение везде одинаковое (`0` / `((void*)0)`). SDL2 не меняет. Безопасно. |
| `APIENTRY` | Да, 4 места: `WinMain()`, `DllMain()`, `UninstInitialize()`, `UninstUnInitialize()` | Всё Windows-специфичный код (entry points, деинсталлятор). SDL2 переопределяет `APIENTRY` для OpenGL на не-Windows платформах (`GLAPIENTRY`), но при переходе на мультиплатформу эти функции будут полностью заменены (SDL2 берёт entry point через `SDL_main`). В игровом коде (рендерер, логика, физика) `APIENTRY` не используется. Безопасно. |

Также проверено пересечение `#define` между vcpkg и собственным кодом игры — нашлись только
`WIN32_LEAN_AND_MEAN` (одинаковый флаг) и `main` (SDL2 `#define main SDL_main` —
стандартный SDL2 трюк). Ничего опасного.

#### Группа E — Compiler internals (определяются компилятором, не кодом проекта)

При поиске по `original_game/` без исключения vcpkg попадались макросы определяемые
компилятором (`__MMX__`, `__SSE__`, `__SSE2__`, `__PRFCHWINTRIN_H`, `_UINTPTR_T_DEFINED`).
Все они оказались из vendored vcpkg хедеров — собственный код проекта их не определяет,
только читает через `#ifdef` (проверка CPU capabilities и т.п.). Трогать нечего.
