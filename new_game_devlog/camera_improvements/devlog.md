# Сводка для следующего агента

## Что это за задача

Третья попытка переделать поведение камеры при контакте со стенами/окружением (трекалось в [`known_issues_and_bugs.md` секция «Камера»](../../new_game_planning/known_issues_and_bugs.md)). Первые две попытки провалились — Claude не понимал систему достаточно глубоко, правил наобум, ломал работающую логику.

**Новая стратегия (нерушимая):** старая физическая логика камеры (`FC_process` в [`fc.cpp`](../../new_game/src/camera/fc.cpp)) **НЕ трогается**. Вся новая логика живёт в отдельном модуле [`vis_cam.h`/`.cpp`](../../new_game/src/camera/) поверх результата FC_process. FC_process пишет в `FC_cam[*]` как раньше; `VC_process` после него читает FC_cam и пишет в `VC_state[*]`; render-interp снэпшотит и lerp'ает VC_state.

Это даёт защиту: если новая логика ошибётся — старая система остаётся целой, rollback тривиален. Сломать накапливаемый accumulator FC_process физически невозможно (его поля не трогаются).

## Pipeline (одна строчка)

```
Physics tick:  FC_process()  →  VC_process()  →  render_interp_capture_camera(&FC_cam[0])
Render frame:  RenderInterpFrame ctor (substitute fc->x/y/z = lerp моих snap'ов)  →  draw  →  dtor restore
```

## Что уже сделано

**Подготовка:**
- Удалены старые системы wall-vs-camera / wall-vs-ground / L1-R1 rotate lock / emergency toonear / `FC_setup_initial_camera` LOS. Детали удалённого — в [`legacy_wall_collision_reference.md`](legacy_wall_collision_reference.md).
- Починены 2 бага по ходу: вертикальное кручение правым стиком триггерило get-behind (Запись 6); F5/F6/F7/End/Del/PgDn теряли клики при FPS > physics rate (Запись 7).

**Инфраструктура и алгоритм `vis_cam`:**
- [`vis_cam.h`/`.cpp`](../../new_game/src/camera/) — `VC_State`, `VC_state[FC_MAX_CAMS]`, `VC_process()`, реализация коллизии в `vc_process_one`.
- [`render_interp.cpp`](../../new_game/src/engine/graphics/render_interp.cpp) — `render_interp_capture_camera` снэпшотит из `VC_state[idx]`, ctor/dtor не тронуты.
- [`game.cpp`](../../new_game/src/game/game.cpp) — `VC_process()` вызывается сразу после `FC_process()`.

**Алгоритм коллизии (вошёл в коммит):**
- Sphere-model: камера трактуется как сфера `VC_CAMERA_RADIUS = 0x4000` (~0.5m).
- 5 рейкастов из focus: центр + ±right + ±up.
- Каждый рейкаст = 32-sample walk с `MAV_inside(s>>8, s>>8, s>>8)` + фильтр `MAVHEIGHT(sample_cell) > MAVHEIGHT(focus_cell) + 4`. Фильтр нужен потому что `MAV_inside` семантически «ниже MAVHEIGHT surface», и без фильтра на flat ground срабатывает на каждом sample.
- `min_d_hit` среди fail рейкастов → vc смещается к focus с `d_new = min_d_hit - VC_WALL_OFFSET (0x5000)`.
- Gates: `EWAY_cam_active || PLAYCUTS_playing || fc->focus == NULL` → identity copy.
- Детали — Запись 12 ниже.

**Точка фокуса (raycast origin):**
- `VC_FocusPoint focus = {fc->focus_x, fc->focus_y + fc->lookabove, fc->focus_z}`.
- Look-at want-target (приблизительно голова Дарси / центр машины).
- Focus НЕ в центре кадра визуально — это by design из-за smoothing камеры. Готовой точки «центр кадра в мире» в коде нет архитектурно.

**Debug helper для будущей диагностики:**
- Нажатие `\` (KB_BACKSLASH) → в stderr.log записывается per-sample лог центрального рейкаста (32 строки: координаты, MAV_inside, MAVHEIGHT, verdict). Used для разбора проблем 1/2/5 ниже.

# Договорённости по алгоритму коллизии (перед стартом реализации)

Собраны из обсуждения с пользователем 2026-05-14. Когда начинаешь Шаг 2, это твой контракт.

## Базовый алгоритм (формулировка пользователя)

1. Рейкаст(ы) от **точки фокуса** (`focus` в `vc_process_one`) до **текущей позиции камеры** (`vc.x/y/z` — это копия `fc->x/y/z`).
2. Цель — определить нет ли на пути препятствий (стены, неровный пол, и т.д. — то что в старой системе ловилось).
3. Проверять **не только центр** камеры, но и **весь её видимый объём** (рейкаст по нескольким точкам — например 4 угла near-плоскости frustum'а, или другой sampling).
4. Если есть пересечение — **переместить камеру по существующей нормали** (по линии «камера → фокус») в сторону фокуса до точки пересечения **плюс некий offset** (вглубь, к фокусу — это убирает мелкие декоративные элементы со стен типа bump mapping в кадре).
5. **Только вдоль этой нормали**, не сбоку. Камера остаётся на той же оси «view→focus», просто становится ближе к фокусу.

## Уточнения от пользователя

**A) Камера может «пролететь» точку фокуса — это нормально.**
Если Дарси прижата к стенке вплотную, offset может оказаться больше расстояния до фокуса → камера пройдёт через focus и окажется с другой стороны. Не пытаться clamp'ить на focus point, это by design.

**B) Дарси может стоять внутри стенки (лестница, hug-wall, и т.п.).**
Тогда focus point может оказаться **внутри** геометрии и рейкаст «от focus к камере» начнётся уже внутри стены. Предположительный способ обработки (не в первой итерации, **только записать**):
- Если origin рейкаста внутри геометрии — пускать дополнительный рейкаст от Дарси **в противоположную сторону** от камеры; найти точку выхода из стены.
- Использовать эту точку выхода как новый origin рейкаста к камере.
- Точное решение пока неизвестно — оставить как открытый вопрос, не реализовывать в первой версии.

**C) LOS API.** Пользователь не уверен. Цель — игнорировать мелкие объекты (фонари, мусорки), машины тоже игнорировать (на первой итерации). Картинка должна быть «красивой», не должно быть возможности заглянуть на другой конец карты через стенку. Возможно нужен микс API. Действие для тебя: **разобраться** какой LOS API что покрывает (стены/заборы/прозрачные стены/полы/etc), и решить — возможно `there_is_a_los` с конкретным набором `LOS_FLAG_*` подойдёт. Опции:
- `there_is_a_los(x1,y1,z1, x2,y2,z2, flags)` — точная геометрия. Флаги: `LOS_FLAG_IGNORE_PRIMS` (мелкие объекты → нам нужно их игнорить), `LOS_FLAG_IGNORE_SEETHROUGH_FENCE_FLAG`, `LOS_FLAG_IGNORE_UNDERGROUND_CHECK`.
- `MAV_height_los_slow(warehouse_id, x1,y1,z1, x2,y2,z2)` — height-aware MAV-навмеш; заполняет глобалы `MAV_height_los_fail_x/y/z` точкой пересечения.

Принципиально для рейкаста нужна **точка пересечения** (чтобы переместить камеру к ней). У `MAV_height_los_slow` это в готовом виде. У `there_is_a_los` — придётся итеративно сужать (binary search вдоль луча). Возможно стоит начать с MAV_height_los_slow + флаг для игнора мелких; если он не игнорит мелочёвку — переключиться на there_is_a_los или комбинацию. **Решение — за тобой, в коде после анализа API.**

**D) Многоточечный sampling.** Без разницы какой именно — пользователь сказал «бери тот что дешевле по перфу». Варианты:
- (a) Одна точка центра + safe-radius offset в сторону focus (минимум вычислений).
- (b) 4 угла near-плоскости frustum'а (4 рейкаста).
- (c) Дискретная сетка 3×3 / 4×4 (9-16 рейкастов, дороже).
Я склоняюсь к (b) — 4 рейкаста, симметричный покрывает корнер-кейсы. **Перф: помни про целевое железо** (Steam Deck, слабые ПК) — не делать больше чем достаточно.

**E) Машина (CLASS_VEHICLE).** Тот же алгоритм без особенностей. Скрытие машины (когда камера близко) — позже.

**F) Offset «вплотную к стене + немного».** «Полметра» по словам пользователя. В world units — Дарси ростом ~`0x10000` (≈ 1.7m), значит полметра ≈ `0x4800-0x5000`. Стартовое — берём ~`0x4000–0x5000`. Подкрутим по факту визуально.

**G) Скрытие модели Дарси / машины** при близкой камере — **НЕ в этой итерации**. Сначала только collision. Скрытие — отдельный шаг.

**H) Y-составляющая.** Не обсуждалось напрямую. По смыслу «полы / уступы» тоже могут быть препятствием — алгоритм должен работать в 3D, не только XZ. Возможно пол под камерой будет ловиться при низком pitch. Если станет проблемой — обсудить отдельно.

**I) Кинематики (EWAY/PLAYCUTS) — не трогать.** Сценарные камеры идут своим pipeline через EWAY_grab_camera + render_interp_apply_eway_camera. Применять collision-логику только когда **не** активна cutscene-камера (по флагу `EWAY_cam_active` или результату `EWAY_grab_camera`).

### История — почему третья попытка

Это **третья попытка** переписать поведение камеры начиная с текущего коммита.

Первые две попытки провалились. Главная причина (формулировка пользователя):

> «У тебя то ли контекст заканчивался то ли ты всю систему понять не мог и ты не проверял то что надо проверять. Делал наобум. Чето правил — ломалось в другом месте. Как будто архитектуру не понимал.»

И:

> «Я просил тебя полностью подробно проанализировать как СЕЙЧАС работает перед тем как что то делать, чтобы ты понял систему. Но это тебе не помогло.»

То есть глубинная причина провалов — **я не понимал систему**, при этом всё равно правил, фиксил одно — ломалось другое, по кругу.

Дополнительно: в обеих попытках я каждый раз **переписывал всю систему работы с камерой** (зачем — пользователь уже не помнит, видимо чтобы «правильно сделать»). Это усугубляло: ломалась рабочая логика, и при этом своя — не работала как надо.

### Новая стратегия — старая логика НЕ трогается

Принципиальное изменение подхода:

1. **Существующая физическая логика камеры — НЕ трогается вообще.** Это нерушимое правило этой попытки.
2. Заводим **отдельный набор переменных** для новой системы камеры.
3. Наша логика встраивается как **новый этап**, идущий ПОСЛЕ всей старой физики камеры и ПЕРЕД фазой интерполяции (которая сейчас уже работает).
4. **Интерполяция переключается** на чтение наших новых переменных вместо старых.
5. Между кадрами можно хранить накапливающийся state в наших переменных.
6. **Кинематики (cutscenes) — не трогаем**, пусть работают как раньше; интерполяция нужна только для геймплейной камеры.

Ключевая защита: даже если я не до конца понимаю старую систему — я её не ломаю. Худший сценарий — наша новая логика считает ерунду, интерполяция читает ерунду. Старая система при этом остаётся целой; rollback = «вернуть интерполяцию на старые переменные».

## Запись 6 — фикс бага «get-behind активируется при вертикальном кручении правым стиком»

### Симптом

В состоянии покоя (Дарси стоит) если правым стиком отклонить камеру от дефолтного положения «за спиной»:
- **Горизонтальное кручение стика** (`stick_x`) — система get-behind не активируется, камера держится в той ориентации куда её отклонили. ✅
- **Вертикальное кручение стика** (`stick_y` — height adjust) — система get-behind **активируется прямо во время кручения** и тянет камеру обратно за спину Дарси. ❌

Гипотеза пользователя: горизонтальная ветка ставит какой-то «активность управления» флаг, который подавляет get-behind, а вертикальная — нет.

### Причина (подтверждена кодом)

Флаг — `fc->nobehind` (это и означает «не пытайся развернуть камеру за спину»). Используется в [`FC_process` get-behind блоке](../../new_game/src/camera/fc.cpp): пока `fc->nobehind > 0`, get-behind пропускается и nobehind decay'ется (`-= 0x80 * TICK_RATIO >> TICK_SHIFT` каждый тик).

Места где `nobehind = 0x2000` выставляется (т.е. «пользователь сейчас управляет камерой, get-behind не лезь»):
- L1/R1 rotate decay-блок ([fc.cpp:790](../../new_game/src/camera/fc.cpp#L790))
- Right stick **horizontal** orbit ([fc.cpp:819](../../new_game/src/camera/fc.cpp#L819))
- Right stick **vertical** height adjust — **отсутствовало**, в этом и был баг

При горизонтальном кручении nobehind постоянно «обновлялся» до 0x2000 → get-behind подавлялся. При вертикальном — nobehind decay'ался до 0 → get-behind включался даже пока стик отклонён.

### Фикс

Добавлена одна строка `fc->nobehind = 0x2000;` в ветку `if (abs(stick_y) > 8000)` (после Y clamp), [fc.cpp:835](../../new_game/src/camera/fc.cpp#L835).

- `make build-release`: passed.

## Запись 7 — фикс класса бага «edge-triggered camera keys теряются на FPS > physics rate»

### Симптом

F5/F6/F7 (set camera type 1/2/3 + behind) на 60+ FPS с интерполяцией срабатывали 1 из 10–15 нажатий. На 30 FPS cap без интерполяции — стабильно. То же касается Delete/PgDn (rotate left/right на клаве) и End (force behind на клаве) — но они нажимаются часто, потеря одного клика не замечалась.

### Причина (architecture-level)

Rate mismatch между render frame и physics tick:

- [`input_frame_update`](../../new_game/src/engine/input/input_frame.cpp#L169) копирует `prev = curr` и обновляет `curr` **каждый render frame** (через `LibShellActive` в [host.cpp:241](../../new_game/src/engine/platform/host.cpp#L241)).
- [`input_key_just_pressed(k)`](../../new_game/src/engine/input/input_frame.cpp#L264) = `curr[k] && !prev[k]` — true только на одном snapshot'е.
- F5/F6/F7 опрашиваются внутри [`get_hardware_input`](../../new_game/src/game/input_actions.cpp#L3338), которая вызывается из `do_packets` ([thing.cpp:506](../../new_game/src/things/core/thing.cpp#L506)) **только когда physics accumulator дотягивает до tick** ([game.cpp:1008-1015](../../new_game/src/game/game.cpp#L1008-L1015)).

Сценарий потери: нажал F5 на render frame N → `prev=0, curr=1`. Physics accumulator < tick → process_things пропущен → клик не прочитан. Render frame N+1 → `input_frame_update` → `prev=1, curr=1` или `0`. Edge стёрт. Когда physics tick наконец отработает — `just_pressed = false`. Клик потерян.

При physics_hz=30, render=60+ → ~50% кадров без physics tick → ~50% кликов теряется.

### Решение

Заменено `input_key_just_pressed` на `input_key_press_pending` + `input_key_consume` (известный паттерн в проекте — уже используется в [siren](../../new_game/src/game/input_actions.cpp#L2979-L2981), [JUMP](../../new_game/src/game/input_actions.cpp#L3808) в той же функции, в [widget.cpp](../../new_game/src/ui/menus/widget.cpp#L69-L70), [gamemenu.cpp](../../new_game/src/ui/menus/gamemenu.cpp#L206), [playcuts.cpp](../../new_game/src/missions/playcuts.cpp#L511)).

`s_keys_press_pending` латчит rising edge и **переживает** frame_update'ы, сбрасывается только явным `input_key_consume`. Подходит для consumer'а, работающего на ритме отличном от frame_update.

## Запись 9 — решение по архитектуре, план шага 1 (скаффолд)

### Решение пользователя

**Вариант A в строгом виде. Никаких перетираний fc->x/y/z (или других полей старой системы).**

Ключевое уточнение пользователя:
> «там может быть накапливаемый эффект когда прошлая итерация зависит от результатов предыдущей»

Это важный нюанс который я не сформулировал явно в Варианте A: FC_process **использует свои fc->x/y/z как accumulator между tick'ами** (smoothing `actual += (want - actual) >> 2` читает actual из прошлого тика). Если перетереть fc->x/y/z своими значениями — следующий тик FC_process получит мои значения как стартовую точку smoothing'а, и его накапливаемая траектория сломается.

**Следствие:** fc->x/y/z/yaw/pitch/roll/lens — это **read-only для моей системы**, и **read-only для всех read'еров на render-вне-substitution** (handle_sfx, MFX_CAMERA voices, etc) тоже должны постепенно мигрировать на мои поля — но не в первом шаге.

### Архитектура (финал)

**Новые поля** — добавляю в `FC_Cam` (или nested struct внутри неё). Рабочее имя — `fc->vis.*` («visible / output camera», то что реально видно игроку):

```cpp
struct FC_Cam {
    // ... existing fields (не трогаем) ...
    struct {
        SLONG x, y, z;
        SLONG yaw, pitch, roll;
        SLONG lens;
    } vis;  // computed by FC_collision_process, snapshotted/lerped by render-interp
};
```

(Имя `vis` обсуждаемо. Альтернативы: `out`, `final`, `coll`, отдельный массив `g_vis_cam[FC_MAX_CAMS]`. Я склоняюсь к nested struct внутри FC_Cam для удобства передачи.)

**Pipeline новой системы:**

1. **Physics tick:**
   - FC_process отрабатывает как раньше → пишет в fc->x/y/z и т.д. Никаких изменений в fc.cpp кроме добавления полей в struct.
   - **`FC_collision_process()`** (новая функция, чистая, без побочных эффектов на FC_process state) читает fc->x/y/z и focus + другие данные, считает collision-aware значения, пишет в fc->vis.x/y/z и т.д.
   - На шаге 1 (скаффолд) — просто копирует fc->x/y/z → fc->vis.x/y/z. Алгоритм коллизии — отдельным шагом потом.
   - `render_interp_capture_camera` модифицируется: снэпшотит fc->vis.* (не fc->x/y/z).

2. **Render frame:**
   - `RenderInterpFrame` ctor — substitutes interpolated моих snap'ов в **fc->x/y/z** (там где их читают render-time consumers).
   - **`RenderInterpFrame` dtor — НЕ restores к старым FC_process значениям.** Restore к **моим non-interp** (fc->vis.x/y/z). Это «правильное» состояние «снаружи» — мои значения.
   - **ВАЖНО — это всё ещё не нарушает накапливание FC_process**, потому что FC_process читает **только fc->x/y/z**, и **на следующем physics tick'е** перед FC_process'ом старые значения нужны.
   - Подожди. Перечитываю свой план — это **противоречие**. dtor restores к моим = fc->x/y/z = mine. Следующий physics tick FC_process читает fc->x/y/z = mine = НЕ его accumulator. Сломал.

### Re-thinking

Чтобы НЕ перетирать FC_process accumulator, fc->x/y/z **должны** оставаться значениями FC_process даже после render. То есть:
- В RenderInterpFrame ctor: **save** старые fc->x/y/z в temp, **substitute** fc->x/y/z = lerp(my_prev, my_curr).
- В RenderInterpFrame dtor: **restore** fc->x/y/z = saved (старые FC_process значения).

Это **ровно как сейчас работает RenderInterpFrame** — только snap'ы теперь мои.

После dtor: fc->x/y/z = FC_process значения (нетронуты, accumulator работает).
Между ctor и dtor (внутри draw_screen): fc->x/y/z = lerp(my) — render видит мои.

**Что с handle_sfx, MFX и др.?** Они вне substitution (после dtor) → читают FC_process значения, не мои. Это **расхождение** между визуальной камерой (мои) и audio/HUD-yaw (FC_process).

Для шага 1 (скаффолд) — расхождения нет (мои = копия FC_process). Тестируется идентичное поведение.

Для будущих шагов (с реальной collision logic) — расхождение появится. **Решение** — на тот момент менять отдельных читателей чтобы читали fc->vis.* вместо fc->x/y/z. Это правка существующего кода, но это правка **читателей-адаптеров** (audio/HUD), не FC_process логики. Уточню с пользователем в момент когда понадобится.

### Финальный алгоритм (вошёл в коммит)

Файл [`vis_cam.cpp`](../../new_game/src/camera/vis_cam.cpp):

- **5 рейкастов** sphere-model: центр + ±right + ±up на радиусе `VC_CAMERA_RADIUS = 0x4000` (~0.5m).
- **Каждый рейкаст** = 32-sample walk от focus к corner. Step по XYZ интерполируется.
- **На каждом sample:** `MAV_inside(s>>8, s>>8, s>>8)`. Если TRUE — кандидат на hit; но дополнительно проверяется `MAVHEIGHT(sample_cell) > MAVHEIGHT(focus_cell) + 4`. Без этого фильтра на ровной улице `MAV_inside` возвращает TRUE на каждом sample (он семантически «ниже MAVHEIGHT поверхности», и на ровной поверхности это так всюду).
- **На hit:** `d_hit = full_len * step / N`. min по всем 5 fail рейкастам → `min_d_hit`.
- **Применение:** `d_new = min_d_hit - VC_WALL_OFFSET` (0x5000, ~0.5m). vc сдвигается к focus по линии focus→vc с масштабом `d_new / d_now`. Если `d_new <= 0` — vc прыгает прямо на focus (overshoot by design, см. договорённость A).
- **Gates:** `EWAY_cam_active || PLAYCUTS_playing || fc->focus == NULL` → identity copy, никакой коллизии.

В `VC_process` поллится `KB_BACKSLASH` через `input_key_press_pending + input_key_consume` (тот же паттерн что фикс edge-keys из Записи 7). На press — следующий тик `vc_process_one` пишет в stderr детальный per-sample лог центрального рейкаста (32 строки: координаты sample, MAV_inside, MAVHEIGHT, verdict HIT/inside-but-filtered/outside).

### Известные проблемы (открытые вопросы для следующей итерации)

Пользователь нашёл играя:

1. **Не работает против небольших выступов.** Возможно это другой тип геометрии (col_vects? OB prims-стены?), либо delta `MAVHEIGHT(sample) - MAVHEIGHT(focus)` < threshold = 4 и фильтр их отсекает. Стартовая точка диагностики — дамп по `\` рядом с таким выступом, посмотреть `mavh` per-sample.

2. **Не работает против земли (горы, склоны).** `MAV_inside` про MAVHEIGHT — это эффективно «потолок», а не «пол». Падения камеры под terrain (на крутом склоне камера может уйти под землю) не детектится этой системой. Возможно нужен **второй sample-проход** через `PAP_calc_map_height_at` — если sample.y < ground_height, это hit. Аналог underground-check у `there_is_a_los`.

3. **Резкость при поворотах когда камера уже прижата к стене.** Без коллизии → identity copy = старая физика FC_process даёт плавность. С коллизией → vc вычисляется заново каждый тик от текущей vc.x/y/z (а fc-state не аккумулирует мой shift, потому что мы пишем в VC_state, не в fc->x/y/z — это by design). При повороте камеры вдоль стены `min_d_hit` прыгает от тика к тику, и vc прыгает соответственно. **Пользователь явно сказал:** сглаживание нужно ТОЛЬКО **между collision-tick'ами когда камера уже прижата к стене**. Сглаживание между нормальной траекторией и moment-of-impact НЕ нужно (резкое прижатие к стене — норма). То есть: если в предыдущем тике fail_count > 0 И в текущем тоже fail_count > 0 — lerp между prev_vc и computed_vc.

4. **Near-clip-plane не учтён.** `POLY_ZCLIP_PLANE = 1/64` unshifted (~1cm) — камера визуально видит на этом смещении ВПЕРЁД от позиции vc. Моя коллизия двигает саму vc, но **не учитывает** что near-plane корнеры выступают на ~1cm впереди. На некоторых углах в кадр может попасть «кусок внутренностей стенки» через near-clip-отсечку. Фикс — двигать **точку проверки** на near-clip distance forward вдоль forward-вектора камеры (нужно правильно посчитать forward из yaw/pitch) ИЛИ просто увеличить `VC_WALL_OFFSET` чтобы покрыть near-plane запас.

5. **Стена перестаёт работать когда Дарси на выступе** (1-я миссия, машинка, забирается на бордюр). Та же стена в других ситуациях работает. **Причина почти наверняка — MAVHEIGHT filter:** когда Дарси на возвышении, `focus_mavh` поднимается до уровня выступа. Соседние стены, которые раньше превышали `focus_mavh + 4`, теперь могут не превышать. Threshold-фильтр их отсекает. Это **архитектурное ограничение текущего подхода** — алгоритм слепнет если focus находится на чём-то поднятом. Нужен другой эвристический фильтр, например absolute MAVHEIGHT threshold (например > некого минимального уровня «строения»), либо учёт focus.y вместо `MAVHEIGHT(focus_cell)`.

### Фикс проблемы №6 — first-contact jump

Через несколько минут после фиксации проблем 1-5 пользователь нашёл ещё одну: при медленном (миллиметровом) сближении со стеной — в момент первого контакта камера резко прыгает к персонажу на ~half-offset. Пользователь предложил правильную гипотезу: рейкаст не учитывает, что цель — не «дистанция до камеры», а «дистанция до камеры **+ offset**».

**Причина:** рейкаст шёл от focus ровно до vc.x/y/z. В момент когда стена впервые становится достаточно близкой чтобы `d_hit < d_now` (т.е. касается камеры), `d_new = d_hit - WALL_OFFSET` сразу становится `≈ d_now - WALL_OFFSET` — это **discrete jump на WALL_OFFSET**, видимое как скачок ~0.5m.

**Фикс:** `vc_probe_mav` продлевает рейкаст за corner на `VC_WALL_OFFSET + VC_NEAR_CLIP_PAD` (отдельная константа `0x800` ~0.05m, как safety margin от near-clip-plane). Стена детектится **до** того как она доходит до камеры. В момент первого detection `d_hit ≈ d_now + WALL_OFFSET` → `d_new ≈ d_now` → нет прыжка. По мере дальнейшего приближения стены `d_new` плавно убывает.

## Запись 14 — 2026-05-15 — переделан EXIT-режим: плавный fade без velocity-разрыва

### Симптом

Пользователь нашёл играя: при отрыве камеры от стены — видимый дёргающий рывок длиной 1-2 кадра. Прошлый агент пытался это починить «крутя туда-сюда» offset taper параметры и не смог; смотрел логи и не увидел рывка в абсолютных значениях, начал убеждать пользователя что «показалось» из-за разницы скоростей. Контекст рассыпался, задача передана новому агенту.

### Корневая причина

Прошлый агент реализовал EXIT-режим как **статичный offset taper**: на тике выхода из COLLISION захватывался `exit_off = prev_vc - fc`, далее N тиков `vc = fc + offset * (ticks_left / N)`. Это **другая математическая модель**, не плавное продолжение COLLISION lerp'а.

На анализе stderr.log оказалось наглядно — Y-скорость камеры на переходе COL→EXIT:
- последний COL тик: delta out.y = **+5256/тик** (lerp prev→computed догоняет fc по мере уменьшения wall offset);
- первый EXIT тик: delta out.y = **+1538/тик** (теперь движение определяется только fc-движением + равномерным сдувом offset'а).

Это разрыв первой производной траектории — видимый глазу скачок ускорения на 1-2 кадра. В абсолютных значениях не заметен (позиции рядом), но визуально читается как «дёрганул».

### Фикс

EXIT-режим теперь использует **тот же lerp механизм** что и COLLISION:

```cpp
// COLLISION:    vc = prev * (DEN-NUM)/DEN + computed_vc * NUM/DEN
// EXIT:         vc = prev * (DEN-NUM)*tl/(EXIT_TICKS*DEN) + fc * (...)
//                                       ^----- keep shrinks t→0 with ticks
```

На переходе COL→EXIT lerp механизм идентичен, меняется только:
1. **Target** — `computed_vc` → `fc` (т.к. `vc_process_one` вернул false и оставил vc как identity copy).
2. **Keep вес** начинает линейно убывать от `(DEN-NUM)/DEN` (т.е. ~0.4 при NUM=6, DEN=10) до 0 за `VC_EXIT_TICKS = 10` тиков (~500мс @ 20Hz).

По мере keep→0 lerp вырождается в `vc = fc`, наше сглаживание тает естественно. Скорость непрерывна потому что уравнение движения то же самое.

### Параметры (вошли в коммит)

| Константа | Значение | Что значит |
|-----------|----------|------------|
| `VC_SMOOTH_ALPHA_NUM / VC_SMOOTH_ALPHA_DEN` | `6/10` | alpha=0.6 (keep=0.4) — в 2 раза меньше плавности чем была у прошлого агента (3/10) по просьбе пользователя |
| `VC_EXIT_TICKS` | `10` (~500мс @ 20Hz) | Длина fade'а после ухода от стены — keep линейно идёт от 0.4 до 0 |

**Я выдвинул гипотезу про кривизну алгоритма** (что углы камеры после shift'а могут оказаться за hit point, потому что рейкасты идут «веером» из focus, а shift идёт по центральной оси). При тесте пользователь зафиксировал:
1. **Тени с offset** — shadow projection использует константы калиброванные под `1/64`.

На PC с float z-buffer мы можем позволить себе **значительно меньший** near plane — это улучшение поверх оригинала, не только фикс wall-collision. Глобально снижать near — **рабочий путь**, нужно только перекалибровать зависимости.

## Запись 16 — 2026-05-15 — снижение POLY_ZCLIP_PLANE 1/64 → 1/512 + калибровка зависимостей

### Решение

Глобально снизили `POLY_ZCLIP_PLANE` с `1/64` до `1/512` (×8 ближе к камере). Это:
1. Закрывает «кишки на углах стен» (главная цель — обнаружено диагностикой в Записи 15).
2. Само по себе улучшение поверх оригинала: оригинальные 1/64 — наследие PSX z-precision ограничений; на PC с float reverse-z buffer'ом мы можем позволить себе значительно меньший near.

Пользователь визуально не наблюдал z-fighting на дальних объектах при тесте 1/4096 (потолок), так что 1/512 — со значительным запасом по precision.

### Бинарный поиск значения

- **1/256 (×4):** обычные стены чистые, но при «Дарси на выступе» (близкая позиция focus к стене) ещё видны кишки.
- **1/512 (×8):** все известные углы чистые. ← **выбрано**
- 1/4096 (×64) был тестовым потолком, не нужен.

### Калибровка зависимостей

**Прямые usages `POLY_ZCLIP_PLANE`** (grep по всему `new_game/src`) — все 30+ мест уже derived: `aeng.cpp` (sz clamp), `polypage.cpp` (z-buffer encode), `glow.cpp` (glow z clamp), `facet.cpp` (near clip), `poly.cpp` (projection / screen mul / sprite scale / fog uniforms / clip vertex / dist buffer). **Автоматически масштабируются.**

**Hardcoded `1/64` или `64.0` в формулах** — найден один прямой:

**Far facets fade (lit-path):**

**Косвенные зависимости через z-precision (не lockstep с POLY_ZCLIP_PLANE):**

`ge_set_depth_bias` в [core.cpp:1039](../../new_game/src/engine/graphics/graphics_engine/backend_opengl/game/core.cpp#L1039) — `glPolygonOffset(-1.0f, (float)(-bias * 512))`. Empirical mapping D3D6 ZBIAS (16-bit Z) → GL polygon offset (24-bit Z). При изменении near plane z-precision GL buffer перераспределяется, тот же polygon offset даёт другой визуальный сдвиг. На 1/4096 пользователь видел shadow offset — на 1/512 визуально OK, не трогали. Если регрессии появятся, надо подбирать `512`-множитель эмпирически.

## Запись 17 — 2026-05-15 — расширение детекции, регрессии от near plane

### Что сделано

**1. Подъём focus point на ~0.2м (`VC_FOCUS_LIFT = 0x2000`).**

Зачем: когда Дарси висит на уступе (pre-pull-up animation) и игрок крутит камеру к ней лицом, неподнятый focus оказывался на уровне головы Дарси — на или ниже верхней грани уступа. Рейкасты focus→camera пересекали угол уступа, wall-collision стягивала камеру на focus → кадр оказывался внутри тела Дарси. Подъём focus чуть выше уступа убирает артефакт. Значение `0x2000` — минимальный empirically достаточный лифт.

**2. Снижение MAVHEIGHT threshold с 4 до 1 (`VC_OBSTACLE_MAVH_THRESHOLD = 1`).**

Зачем: старый threshold = 4 единицы MAVHEIGHT ≈ 1.7м (рост Дарси) — фильтровал всё что ниже её роста. Здания (sentinel 127) проходили, но крупные декоративные платформы (~1м, MAVHEIGHT ≈ 2-3) — нет. Снижение до 1 ловит платформы. Плоская земля по-прежнему фильтруется (там разница 0 < 1).

Побочный эффект: **проблема №3 «стена не работает когда Дарси на выступе»** (Запись 13) — тоже закрылась. Когда focus_mavh поднимался до уровня выступа, старый threshold `+ 4` отсекал соседние стены; новый `+ 1` пропускает.

**3. Ground/slope detection через `PAP_calc_map_height_at`.**

Зачем: MAVHEIGHT хранит «верх препятствия» (стены, потолок), не пол. Камера могла нырнуть под terrain на склонах/холмах. Решение — параллельная проверка в `vc_probe_mav`: если `sample_y < PAP_calc_map_height_at(sx>>8, sz>>8) << 8` → hit. Срабатывает или MAVHEIGHT-условие или ground-условие, любое = hit.

**Indoor edge case (известная открытая проблема, в этой итерации НЕ фиксится).**

Внутри зданий (warehouse'ы или клетки с `PAP_FLAG_HIDDEN`):
- Ground check срабатывает на каждом сэмпле — `PAP_calc_map_height_at` для HIDDEN-клеток возвращает `MAVHEIGHT(x,z) << 6` (≈ 2M shifted), что выше любого sample sy. Каждый сэмпл = «под землёй» → камера прижимается к focus.
- MAVHEIGHT-фильтр indoor ведёт себя непредсказуемо: focus_mavh для interior cells может быть sentinel-значением (-127 / 127), relative threshold срабатывает на случайных сэмплах → периодический zoom-in.

В этой сессии было два неудачных подхода (per-sample skip ground check для `PAP_FLAG_HIDDEN`; гейт по `fc->focus_in_warehouse || PAP_FLAG_HIDDEN`), оба сломали outdoor — оказалось, `PAP_FLAG_HIDDEN` ставится и на некоторые outdoor raised cells (декоративные платформы / ступени), и любой гейт по этому флагу ошибочно отключает wall-collision на outdoor сценах.

**Решение отложено** — indoor wall-collision как класс это отдельная задача (нужен другой sampler — WARE_inside для warehouse'ов, и/или другой источник «есть ли пол» для PAP_FLAG_HIDDEN-клеток). Пользователь явно сказал «indoor отдельно, не трогай в этой задаче».

### Регрессии от снижения POLY_ZCLIP_PLANE (Запись 16) — открытые

После снижения `POLY_ZCLIP_PLANE` с `1/64` до `1/512` появились визуальные баги. Их корень — в формулах рендера/эффектов которые либо явно зависят от near plane (а мы не нашли все hardcoded зависимости), либо изменились косвенно через z-precision distribution.

**Подлежат фиксу в этой задаче перед коммитом:**

1. **Outro: чёрный экран вместо видеоролика.**
   - До снижения POLY_ZCLIP_PLANE работало корректно. Сейчас просто чёрный экран.
   - Гипотеза: какая-то часть outro pipeline'а зависит от старого значения near plane — возможно camera setup в outro, или 2D/3D-проекция полностью клипается ближним планом.
   - Где смотреть: `outro/` модуль, проверить какие POLY_*/projection функции дёргаются на outro path, есть ли где hardcoded numerator под near plane.

2. ~~**Провода между домов — в 1 пиксель толщиной.**~~ **Закрыто 2026-05-15.**
   - Корневая причина: `POLY_create_cylinder_points` (используется только `cable_draw`) считает on-screen half-width как `width * POLY_cam_over_view_dist * pt->Z`, где `pt->Z = POLY_ZCLIP_PLANE / view_z`. То есть результат **линейно пропорционален POLY_ZCLIP_PLANE**. При снижении near plane 1/64 → 1/512 (×8 меньше) экранная толщина стала ×8 меньше.
   - Sibling line APIs (`POLY_add_line` / `POLY_add_line_tex_uv`) этой проблемы не имеют — они умножают на `POLY_sprite_scale`, в котором уже есть `1/POLY_ZCLIP_PLANE`, и он компенсирует `POLY_ZCLIP_PLANE` из `pt->Z`. У `POLY_create_cylinder_points` этой компенсации не было.
   - Фикс: добавлен компенсирующий множитель `(POLY_LINE_LEGACY_NEAR / POLY_ZCLIP_PLANE)` где `POLY_LINE_LEGACY_NEAR = 1/64`. Сохраняет текущий вид независимо от любых будущих тюнингов POLY_ZCLIP_PLANE.

3. **Тёмный полупрозрачный силуэт вокруг светящихся вывесок.**
   - На RTA (и других миссиях) светящиеся круглые вывески рисуются с тёмным полупрозрачным ореолом вокруг. В релизной PC-версии такого нет. В PieroZ — либо нет, либо очень слабо.
   - Гипотеза: alpha-blended sprite с тёмным бордюром — при определённом near plane / depth расчёте бордюр начинает виднеться. Возможно sprite z-bias (мы поменяли `1/64` на `POLY_ZCLIP_PLANE` в `sprite.cpp:279-280`) сделал его меньше, и теперь sprite пишет z ближе к настоящему wall'у → блендится с темнотой стены за ним.
   - Где смотреть: `sprite.cpp`, конкретно SPRITE_SORT_NORMAL z-bias и куда эти вывески идут. Возможно надо вернуть для них старый bias или другой sort.
