# libDualsense — скоп либы (what belongs here vs. in the consumer)

Доктринальная памятка: что имеет право лежать внутри `libDualsense/`
и что нет.

## Правило

libDualsense — **функциональный wire-level DualSense API**. В ней
лежит только то, что потребитель (игра, движок, приложение)
**действительно вызвал бы в продакшене** чтобы драйвить DualSense
через HID.

**Внутри либы:**
- Открытие/закрытие устройства, input/output reports, adaptive
  triggers, feature reports, `test_command` primitive, factory-data
  геттеры, IMU calibration helpers.
- Нормализация raw-байтов которые **любой** потребитель иначе
  реализовал бы заново (stick axis → ±1.0, trigger → 0..1). Это
  функционал, а не диагностика — он делает API user/code-friendly
  без потери гибкости (raw fields тоже остаются доступны).

**Снаружи либы (в потребителе):**
- Диагностические обёртки ради обёрток.
- Всё, что существует только чтобы **проверить саму либу** или
  подготовить test scenario.

## Практические примеры того что НЕ идёт в либу

- Проигрывание 1 kHz тестового тона через спикер / jack контроллера —
  реализовано в игровом тестере через публичный `test_command(Audio,
  ...)`, файл
  [`../../new_game/src/engine/debug/input_debug/input_debug_dualsense.cpp`](../../new_game/src/engine/debug/input_debug/input_debug_dualsense.cpp)
  (`send_audio_test_tone` / `audio_test_tone_force_off`).
- Probe-хелперы «работает ли сейчас вибрация на этом коннекшене»
  (потребитель сам может выставить `OutputState::rumble_*`).
- Debug-принты, hex-dump хелперы, pretty-print для `InputState`.
- Любой convenience-wrapper, единственная мотивация которого —
  «так проще вызвать из тестера».

## Практические примеры того что ИДЁТ в либу

- Всё что кодирует реальную HID-операцию: open device, send output
  report, parse input report, получить firmware info, применить IMU
  calibration к raw sample.
- `normalize_stick_axis()` / `normalize_trigger()` в `ds_input.h` —
  формула одна и та же у каждого потребителя, смысла её дублировать
  нет.

## Правило принятия решения

Перед тем как добавить что-то в `libDualsense/`, спросить себя:

> **Будет ли это вызывать реальный потребитель в продакшене, или
> только чтобы убедиться что либа ведёт себя правильно?**

Если второе — это в тестер / игру / диагностическое приложение,
а не в либу.

## Почему

- Либа должна быть минимальной и переиспользуемой между потребителями.
- Тестер-специфичные обёртки раздувают API и размазывают границу между
  «что делает игра» и «что проверяет тестер».
- Потребители всегда могут скомпозировать публичные примитивы
  (`test_command`, `device_send_output`, …) для любого диагностического
  сценария — и именно так это и делается в нашем in-game тестере.

## История

- **2026-04-18** — принцип зафиксирован после того как предложил
  добавить `ds_audio_test.h` (1 kHz test tone wrappers) в саму либу;
  пользователь отклонил с формулировкой «либа только для реализации
  функционала, дополнительные обёртки и прочее говно там не нужно —
  пусть это игра имплементит». Test tone переехал в in-game тестер,
  собирается из существующего публичного `test_command()`.
