# perf_diag — инструкция по панели (что меряет, как пользоваться)

Памятка для чтения панели во время отладки перфа. Базовая версия панели
готова. Архитектура/история — [`design.md`](design.md).

## Как включить

Флаги в `new_game/src/debug_config.h` (dev-only, в коммит/релиз — `false`):

- `OC_DEBUG_PERF` — сама панель + хоткеи.
- `OC_DEBUG_PERF_LOG` — запись CSV в файл (можно включать отдельно, ловит
  фазу загрузки с самого старта процесса).

Размер шрифта панели — **не флаг**, а кодовая константа
`DBG_PANEL_TEXT_SCALE` в `new_game/src/engine/debug/debug_panel_text_scale.h`
(общая с dbglog). 1 = мелкий 5×9, 2 = крупный 2×.

Пересобрать. Панель — слева, поверх HUD. Работает в геймплее, в меню и в
катсценах одинаково (рисуется отдельным пиксельным слоем).

## Хоткеи (геймплей, без bangunsnotgames)

| Клавиша | Действие |
|---------|----------|
| `4` | Показать / скрыть панель |
| `5` | Переключить окно усреднения: 10 → 30 → 60 → 120 кадров (по кругу). Это же значение становится окном усреднения файл-лога. |

> **GPU всегда меряется через `glFinish`** (тоглера нет). Поэтому FPS на
> этой панели **синтетически занижен** (CPU/GPU сериализованы) — это
> норма дебаг-режима, цена за честное разделение CPU/GPU. Реальный FPS
> смотри без перф-панели.

Шапка панели — **одна строка**: `FPS NNN (N last frames)` (N это окно
усреднения, клавиша `5`). Разбивка CPU/GPU больше не в шапке — она в
строке-футере `split` внизу таблицы (см. ниже).

Под шапкой — строка-заголовок колонок: `stages  ge_* CPU  non-ge_* CPU  GPU`.

## Модель таблицы

- **Порядок строк = реальный порядок вызова в кадре** (физика → рендер
  → idle …; внутри render: scene → hud → post → ui.* → present). Стадия,
  которая давно не выполнялась (например игровые строки после выхода в
  меню), **скрывается** — мёртвые пустые строки не висят.
- **Имя — иерархия через точки** (`render.scene`, `render.ui.dbglog`).
  Строки таблицы идут сплошняком, без пустых строк между группами
  (уровней мало, разделители только мешали). Имя задаёт `.other` по
  уровню; визуальной группировки пробелами нет.
- **Единый принцип. Один макрос `PERF_SCOPE(имя)` меряет И CPU, И GPU.**
  Отдельного «GPU-скоупа» нет — забыть нельзя, у **каждой** стадии-
  строки **всегда обе колонки** (GPU = 0, если стадия ничего не шлёт
  видяхе — physics/idle/present так и читаются `00.00%`, не пусто).
  Кусок может быть **обёрткой** над под-кусками — тогда автоматически
  рисуется `<кусок>.other = обёртка − Σ(детей)`, **по той же формуле
  для CPU и для GPU** (обычная строка со всеми данными). `.other`
  появляется всегда, если уровень не покрыт детьми полностью. Внизу
  единый глобальный `(other)` = кадр − Σ(корней) — щели между стадиями
  (ввод, звук, склейка); GPU у него `00.00%` (там нет работы видяхе, и
  «целого GPU-кадра» для остатка нет). Точки в имени = иерархия
  (`.other` по уровню); пустых строк-разделителей в таблице нет.
- **GPU мерится через `glFinish`, всегда (тоглера нет).** Таймстемп-
  запросы убраны — они путали (считали простой GPU как работу). Теперь:
  на выходе каждой стадии CPU делает `glFinish` (ждёт пока GPU домолотит
  именно её — предыдущая стадия уже слита), и это ожидание = реальный
  GPU стадии. Это **намеренно сериализует CPU↔GPU → FPS синтетически
  проседает** (примерно вдвое в GPU-bound) — осознанная цена за то, что
  чётко видно, где CPU, а где GPU. Реальный FPS смотри без перф-панели.
- **Колонки: `ge_* CPU | non-ge_* CPU | GPU`.** Оба `*CPU` — это
  CPU-wall-время, разделённое по признаку «внутри граф-API или нет»;
  `GPU` — реальный per-stage GPU-busy (`glFinish`, всегда вкл).
  - `ge_* CPU` = CPU **внутри** графического API (`ge_*` draws +
    composition): сабмит/драйвер + редкое **ожидание GPU**, случившееся
    внутри самих этих вызовов (back-pressure). НЕ исполнение GPU (это
    отдельная `GPU`). Своп в `ge_*` не входит — его ожидание уходит в
    строку `GPU wait (glFinish)`.
  - `non-ge_* CPU` = CPU **вне** API. ⚠️ Не строго «логика игры»: для
    `physics` — да логика; для **рендер-стадий** сюда входит движковая
    CPU-подготовка отрисовки (сборка геометрии, построение глиф-квадов,
    циклы). Поэтому полный dbglog раздувает `non-ge_* CPU` (построение
    ~тысяч квадов), `ge_* CPU` (тысячи API-вызовов) и `GPU` одновременно
    — норма, не путаница колонок.
  - **КАЖДАЯ колонка сама по себе = 100%** (доля от своего ресурса):
    `CPU%` = стадия ÷ весь CPU; `ge_*%` = стадия ÷ весь ge_*; `GPU%`
    = стадия ÷ весь GPU. Складывать две колонки **не надо** — каждая
    читается отдельно «куда уходит этот ресурс». `(other)` входит в
    сумму `non-ge_*` как обычная строка. **GPU-ожидание (`GPU wait
    (glFinish)`) в проценты НЕ входит вообще** — оно отдельной строкой
    ПОСЛЕ `total`, без процента, только мс (см. ниже). Так проценты
    показывают только реальную работу.
  - **Формат ячейки = `процент% (мс)`** (как в шапке у FPS): и доля
    ресурса, и абсолютные миллисекунды стадии. Цвет ячейки градуируется
    по проценту нагрузки. Счётчики (секция `counters`) — только сырое
    число, без %/мс.
  - **Строка `total`**: по каждой колонке всегда `100.00%` + в скобках
    сумма миллисекунд этой колонки (`ge_* CPU` реальный сабмит,
    `non-ge_* CPU`, `GPU`) — **БЕЗ GPU-ожидания**. Ярко-янтарная.
  - **Строка `split` — ПОСЛЕ `total`** (имя ярко-янтарное, как у
    `total`). Распределение замеренного кадра по 3 колонкам: для каждой
    её тотал-мс ÷ (`ge_total` + `cpu_total` + `gpu_total`). Слева
    направо три процента **в сумме = 100%** — видно, на что ушёл кадр
    (CPU-сабмит / игровой CPU / GPU). Сами проценты красно-белые по
    нагрузке (как ячейки).
  - **Строка `GPU wait (glFinish)` — ПОСЛЕ `split`**, приглушённый
    мягко-янтарный, **только мс, без процента и без скобок**. Не входит
    ни в один 100%. Это синтетика замера (CPU сливал видяху) — контекст
    «сколько от кадра съелось → почему такой FPS», не для оптимизации.
  - `GPU` = реальный per-stage GPU-busy (через `glFinish`, всегда).
  - Как читать: большой `non-ge_*` = упёрлись в логику/вычисления;
    большой `ge_*` = упёрлись в отрисовку на стороне CPU (сабмит/
    драйвер); большой `GPU` = в саму видяху. `GPU wait` на боттлнек не
    указывает — это цена режима замера.

Замечание про точность `ge_*`: меряется обёртыванием горячих `ge_*`
(draws/composition). Считается только графика через `ge_*` (по
правилам проекта весь графический код игры идёт через него). Накладные:
2 чтения часов на ge-вызов; при тысячах draw (много строк dbglog) —
доли мс и лёгкое самоискажение. Под debug-флагом, для диагностики ок.

> ⚠️ **CPU vs GPU.** `CPU`/`ge_*`/`GPU` достоверны (wall-clock потока +
> per-stage `glFinish`), но ценой искажённого FPS (CPU/GPU
> сериализованы — всегда, тоглера нет). Таймстемп-замер считал бы
> простой GPU как работу — поэтому убран в пользу `glFinish`. Реальный
> FPS смотри без перф-панели; на панели FPS — синтетически заниженный.

| Строка | Что это | CPU / GPU |
|--------|---------|-----------|
| `physics` | Весь игровой тик за кадр (0..N catch-up тиков): `process_things`, частицы, OB/TRIP/DOOR, скрипт миссии, DIRT, гранаты, MAP/POW/FC + интерп-снапшоты. | **100% CPU**, GPU нет. |
| `render.scene` | Отрисовка 3D-сцены (мир, персонажи, транспорт). | CPU: куллинг/сборка/сабмит. GPU: растеризация сцены. |
| `render.hud` | Игровой 2D-интерфейс (HUD) — `OVERLAY_handle`. | CPU+GPU 2D. |
| `render.post` | Пост-обработка: FXAA + апскейл (composition). | В основном **GPU** (полноэкранные проходы). |
| `render.ui.dbglog` | Правая debug-панель: подложка + накопление текста (буквы — в textflush). | CPU+GPU мелкие. |
| `render.ui.perfpanel` | Эта (левая/FPS) панель: подложка + накопление текста. | CPU+GPU мелкие. |
| `render.ui.textflush` | **Общий сброс всего debug-текста** на GPU (dbglog+perf+FPS+help вместе — один буфер, по панелям не делится). **Сюда падает цена «много строк dbglog → тормоза».** В релизе нет — мысленно вычитай. | CPU+GPU. |
| `render.ui.other` | Остаток `render.ui` (= обёртка − dbglog/perfpanel/textflush): консоль, меню, help, fadeout, radio-сообщения. | CPU+GPU (обе по формуле обёртка−дети). |
| `render.present` | Своп буферов + ожидание GPU/vsync. | CPU тут **блокируется** ждать GPU/vsync. |
| `render.other` | Остаток `render` (= обёртка − scene/hud/post/ui/present): склейка — AENG_blit, сброс GL-стейта, скриншот, pre-flip/diag-колбэки. | CPU+GPU. |
| `render.frontend` | Только в главном меню: `FRONTEND_loop()` (логика+отрисовка меню). | CPU+GPU. |
| `idle` | `lock_frame_rate()` — намеренный сон под FPS-кап. | CPU% = доля сна в кадре; GPU `00.00%` (видяхе ничего не шлёт). Большое = есть запас, кадр успевает раньше кэпа. |
| `io.levelload` | Разовая загрузка уровня (`game_init`). В установившемся геймплее ≈0; всплеск в момент загрузки. | CPU/диск + GPU-upload. |
| `(other)` | Глобальный остаток кадра = кадр − Σ(корней) − GPU-wait: щели между стадиями (ввод, звук, геймпад). | CPU. |
| `total` | Итоговая строка-футер (ярко-янтарная). По каждой колонке `100.00%` + в скобках сумма мс колонки (БЕЗ GPU-wait). | `ge_* CPU` / `non-ge_* CPU` / `GPU` тоталы. |
| `split` | **ПОСЛЕ `total`** (имя ярко-янтарное). Распределение кадра по 3 колонкам: тотал-мс колонки ÷ (`ge_total`+`cpu_total`+`gpu_total`). Три % слева направо = 100%. Проценты красно-белые по нагрузке. | % по каждой из 3 колонок. |
| `GPU wait (glFinish)` | **ПОСЛЕ `split`.** Всё CPU-side ожидание GPU в одном ряду (per-stage glFinish-сливы + дренаж перед свопом). Приглушённый, **только мс — без процента и без скобок**, НЕ входит ни в один 100%. Цена режима замера, не работа игры; контекст «почему такой FPS». | вне процентов. |

`render` / `render.ui` — **обёртки**: своей строкой не печатаются, вместо
них дети + `<имя>.other` (`CPU`/`ge_*`/`GPU` по одной формуле обёртка−дети).
**Каждая колонка (`non-ge_*`, `ge_*`, `GPU`) суммируется в 100%
независимо** (доля стадии от своего ресурса). Всё CPU-side ожидание GPU
(per-stage glFinish-сливы + явный дренаж перед свопом) собрано в
**отдельную приглушённую строку `GPU wait (glFinish)` ПОСЛЕ `total`/
`split`, без процента (только мс)** и **вне всех 100%** — вычтено из
wall стадий и из `cpu_total`/`(other)`, чтобы проценты показывали
только реальную работу. FPS в шапке синтетически занижен на её величину
(glFinish всегда вкл — реальный FPS смотри без перф-панели). Строка
`split` после `total` показывает раскладку кадра по 3 колонкам в % (в
сумме 100%) — куда ушёл кадр: CPU-сабмит / игровой CPU / GPU.

## Список `counters` (отдельной секцией внизу)

Под таблицей стадий — отдельный список с заголовком `counters`: имя +
сырое значение (без %, без CPU/GPU; своя гибкая колонка под значение).
Это не время кадра, а просто числа:

| Счётчик | Что |
|---------|-----|
| `physics.ticks` | Сколько тиков физики в кадре. 1 — норма; >1 — машина не успевает (просадка). |
| `render.drawcalls` | Draw-вызовов за кадр (все `glDrawElements`). Резкий рост = много мелких батчей (типичный убийца перфа, напр. строки dbglog). |

Новый счётчик добавляется одной строкой `PERF_COUNT("имя", значение)` —
сам появляется в этом списке.

## Цвета и линии

- Колонка % красится по нагрузке: бело-серый → красный, нелинейно
  (от ~10% уже тонировано, 100% — красный, 50 vs 80 различимы). Боковым
  зрением видно где «горит». Колонки CPU и GPU красятся независимо.
- Под каждой строкой одинаковая тонкая линия — чтобы не путать имя с
  далеко стоящим значением. «Жирных» линий нет; единственная выделенная
  строка — `total` внизу (ярко-янтарный текст, всегда 100% + сумма мс
  по каждой колонке).

## Файл-лог (`OC_DEBUG_PERF_LOG`)

- Пишется `perf_log.csv` рядом с рабочей директорией процесса.
- Строка раз в N кадров (N = окно усреднения, по умолчанию 60; следует
  за клавишей `5`), значения усреднены за это окно. Per-frame не пишем —
  не успевает и тяжело анализировать.
- Колонки = имена метрик; заголовок перевыпускается если появились новые
  метрики (ленивая регистрация).
- Ловит фазу загрузки (флаг активен со старта процесса, до появления
  панели/хоткеев).
- Перед анализом конкретной проблемы режим усреднения подбирается под
  неё (для спайков — маленькое окно, для трендов — большое).

## Тюнинг (одним числом, в коде панели)

- `DBG_PANEL_TEXT_SCALE` (engine/debug/debug_panel_text_scale.h) — размер
  шрифта (общий с dbglog).
- `PERF_COL_GAP_PX` (perf_diag.cpp) — зазор имя → колонка CPU%.
- `PERF_COL_GAP2_PX` (perf_diag.cpp) — зазор CPU% → колонка GPU%.
- `PERF_TOP_PX` / `PERF_LINE_PX` — отступ сверху / шаг строки.

Ширина панели и колонка значения — НЕ хардкод: меряются по факту от
самого длинного имени каждый кадр (панель сама обтягивает контент).
