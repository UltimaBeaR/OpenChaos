# perf_diag — архитектура и план

Встроенный debug-режим диагностики перформанса. Цель: ловить **плавающие**
просадки которые по одному только FPS не видны — резкое изменение в каком-то
из множества per-stage параметров сразу выдаёт где проблема.

## Зачем (мотивация)

Перф просаживается в определённых местах/ситуациях, но без инструментария
непонятно **что именно** тормозит. По общему FPS не видно деградации
отдельной подсистемы (одна стадия может удвоиться по времени, а FPS просесть
чуть-чуть). Нужен режим где видно много per-stage метрик усреднённо + резкие
изменения подсвечиваются цветом, чтобы боковым зрением заметить аномалию.

## Принятые решения

| Вопрос | Решение |
|--------|---------|
| Гейтинг | Compile-time мастер-флаг `OC_DEBUG_PERF` в `debug_config.h`. Off → все макросы замера = `((void)0)`, 0 оверхеда (как `OC_DEBUG_LOG`/dbglog). |
| Файл-лог | Отдельный флаг `OC_DEBUG_PERF_LOG`. Отдельный — чтобы ловить **фазу загрузки с самого старта процесса**, до появления панели/хоткеев. |
| Реестр метрик | Не хардкод полей. Именованные пробы с иерархическим путём (`render.poly.flush`). Имя видится первый раз → слот авто-регистрируется. Добавить метрику = одна строка макроса, без правок в центральном месте. |
| Виды метрик | (1) тайминг — RAII scoped-таймер, за кадр суммируется; (2) счётчик/значение — число (партиклы, МБ, draw calls). |
| Усреднение | Per-metric ring buffer последних N кадров, на экране среднее по окну (тот же механизм что у FPS-оверлея). FPS — одна из метрик. Окно регулируется хоткеем. |
| Подсветка | Цвет = градиент дефолт→красный по относительному отклонению «короткого среднего» от «нормы» (длинного среднего). Peak-hold: резкий всплеск залипает красным ~1–2 сек, чтобы транзиент не промелькнул. |
| Scene-baseline (сравнение со средним по типу сцены) | **Фаза 2.** Сначала только self-relative (отклонение от собственной недавней нормы) — даёт ~90% пользы без ручной калибровки. |
| Расположение панели | Слева, прижата к левому краю (закрывает игровой HUD/радар снизу). Тёмная подложка + текст сверху — точь-в-точь как dbglog, но левая сторона (dbglog держит правую). |
| ⚠️ Сама панель | Отрисовка перф-панели **тоже инструментируется** отдельной метрикой — вдруг панель сама тормозит. |
| Формат файл-лога | CSV, **усреднённо по N кадров** (per-frame может не успевать писаться и тяжело анализировать). Окно усреднения файл-лога регулируемое; перед анализом конкретной проблемы режим усреднения подбирается под неё. |

## Архитектура

Модуль `new_game/src/engine/debug/perf_diag/`:

- `perf_diag.h` — публичный API + макросы. Всё под `#if OC_DEBUG_PERF`;
  иначе макросы → `((void)0)` (аргументы не вычисляются, символы не
  эмитятся). Образец гейтинга — `engine/debug/dbglog/dbglog.h`.
- `perf_diag.cpp` — реестр, ring buffers, усреднение, цвет/аномалии,
  отрисовка панели, файл-логгер. Всё под `#if OC_DEBUG_PERF`.

### API (эскиз)

- `PERF_SCOPE("path.name")` — RAII-зона, мерит wall-clock, суммирует в
  слот за текущий кадр (зона может срабатывать много раз/кадр).
- `PERF_COUNT("path.name", value)` — записать/прибавить числовую метрику.
- `perf_frame_begin()` / `perf_frame_end()` — в главном цикле: перекат
  аккумуляторов в историю, сброс per-frame, total кадра.
- `perf_diag_draw()` — из верхнего UI-слоя (рядом с `DBGLOG_draw`), своя
  `PANEL_start/finish` подложка + `FONT_buffer_add` (фикс-пиксели).
- Хоткеи (только при включённом режиме, геймплей): тогл панели, размер
  окна усреднения, сброс baseline. По образцу `OC_DEBUG_PHYSICS_TIMING`.

### Реестр

Слот метрики: имя, вид (timing/value), per-frame аккумулятор, ring buffer
истории (фикс-размер static, без аллокаций), короткое+длинное средние,
peak-hold таймер. Имя интернится один раз (макрос кэширует указатель на
слот в static local) → за кадр только указатель + сложение. Точечные имена
→ панель группирует/строит дерево по категориям.

### Усреднение и цвет

- Короткое среднее (~30 кадров) = «сейчас», длинное (~неск. секунд) = «норма».
- `deviation = (now - norm) / norm`. Цвет линейно дефолт→красный по
  модулю deviation, с порогом «почти не поменялось → дефолт».
- Peak-hold: при превышении сильного порога метрика латчится красной,
  таймер ~1–2 сек, затем плавный спад цвета.

### Файл-лог (`OC_DEBUG_PERF_LOG`)

CSV рядом с exe (как `crash_log.txt`/`stderr.log`). Колонки = метрики,
строка = усреднение за N кадров. Буферизованная запись. Ловит фазу
загрузки (флаг активен со старта процесса). Окно усреднения настраиваемое.

## Что мерить (грубо → детально, дробить инкрементально)

- **Кадр/цикл:** total, FPS, present/swap, ticks/frame (catch-up физики),
  idle от FPS-кэпа.
- **Физика (per tick, 0..N за кадр):** весь тик, коллизии, PCOM/AI,
  pathfinding, физика машины, projectile/particle sim, EWAY/скрипт.
- **Графика:** весь AENG_draw, culling/build, мир/город, персонажи/figure,
  POLY flush (по page-типам), тени, отражения (wibble),
  bloom/lensflare/postprocess, UI/HUD, composition, текст, **сама
  perf-панель**. Счётчики: draw calls, поли/верты, живые партиклы,
  dlight'ы, texture uploads/кадр.
- **IO/загрузка:** total загрузки уровня + подфазы (map/textures/prims/
  sounds), стриминг на лету, clump-загрузки.
- **Логика:** mission/EWAY, звук, ввод.

Старт грубый (frame total, physics-total, render-total, present,
load-total), углубляем добавлением проб — реестр это позволяет инкрементально.

## План реализации (база, фаза 1)

1. Флаги `OC_DEBUG_PERF` / `OC_DEBUG_PERF_LOG` в `debug_config.h`.
2. Модуль `perf_diag.{h,cpp}` + CMake: реестр, ring buffers, усреднение,
   цвет/peak-hold, API/макросы, no-op ветка.
3. Развязка в главном цикле: `perf_frame_begin/end`; `perf_diag_draw` в
   ui_render рядом с `DBGLOG_draw` (своя метрика на саму панель).
4. Грубые пробы: frame total, physics tick, render (AENG_draw), present,
   level load total.
5. Хоткеи: тогл панели, окно усреднения, сброс baseline.
6. Файл-логгер (`OC_DEBUG_PERF_LOG`), CSV усреднённо.
7. Углубление проб по подсистемам (графика/физика/IO/логика) —
   инкрементально по мере надобности.

Фаза 2 (позже): scene-baseline (снять средние по типу сцены, режим
сравнения, подсветка отклонений от снятого baseline).
