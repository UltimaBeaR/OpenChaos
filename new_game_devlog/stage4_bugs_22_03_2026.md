## Баг 1: lock_frame_rate — вечный spin-loop (игра не запускается)

**Файл:** `new_game/src/new/missions/game.cpp` — `lock_frame_rate()`

**Суть:** Функция ограничивает FPS через busy-wait на `GetTickCount()`:

```cpp
static SLONG tick1 = 0;
SLONG tick2;
SLONG timet;

while (1) {
    tick2 = GetTickCount();
    timet = tick2 - tick1;
    if (timet > (1000 / fps)) break;
}
tick1 = tick2;
```

`GetTickCount()` возвращает `DWORD` (unsigned 32-bit, миллисекунды с загрузки системы).
При записи в `SLONG` (signed 32-bit) значение переполняется в отрицательное когда счётчик
превышает 2^31 ≈ 2 147 483 648 мс ≈ **24.8 дня**.

При первом вызове `tick1 = 0`. Если `GetTickCount()` уже за порогом 24.8 дней:
- `tick2` = большое отрицательное число (например, −2 141 133 921)
- `timet = tick2 − tick1` = −2 141 133 921 − 0 = −2 141 133 921
- Условие `timet > 16` → FALSE → цикл никогда не выходит

**Фикс:**
```cpp
static DWORD tick1 = 0;
DWORD tick2;
DWORD timet;
```
DWORD-арифметика корректно обрабатывает и переполнение, и wraparound через 49.7 дней.

## Баг 2: PANEL_sign_time — знак разворота всегда на экране

**Файл:** `new_game/src/old/fallen/DDEngine/Source/panel.cpp`

**Суть:** `PANEL_sign_time` — глобальная переменная, хранит момент вызова `PANEL_flash_sign()`
(показать знак направления, знак разворота и т.п.). По умолчанию = 0 (глобал, zero-init).

Код рисовки:
```cpp
SLONG dtime = GetTickCount() - PANEL_sign_time;
if (dtime < 3000) {  // показывать знак 3 секунды
    // рисуем знак
}
```

Если `PANEL_flash_sign()` не вызывался: `PANEL_sign_time = 0`.
При `GetTickCount()` > 24.8 дней: `dtime = (отрицательное) − 0 = большое отрицательное`.
`dtime < 3000` → TRUE → знак рисуется постоянно с момента загрузки любого уровня.

**Фикс:** `SLONG PANEL_sign_time` → `DWORD PANEL_sign_time`,
`SLONG dtime` → `DWORD dtime`.

## Про "24.8 дня": нюанс с гибернацией Windows

Пользователь перезагружал систему ~2 дня назад, но GetTickCount() показывал ~25 дней.

Причина: **Windows Fast Startup (быстрый запуск)** — режим по умолчанию с Windows 8+.
При "выключении" ПК Windows не делает полную перезагрузку: сохраняет состояние ядра
в hiberfil.sys (по сути гибернация сессии ядра). При следующем включении ядро
**восстанавливается** из hibernation, и `GetTickCount()` продолжает отсчёт с момента
предыдущей загрузки, **не сбрасывается**.

Сон (Sleep/Hibernate вручную) тоже не сбрасывает счётчик.
Сбросить `GetTickCount()` можно только полным выключением с отключённым Fast Startup
или через "Перезагрузка" (а не "Завершение работы").

Итог: реальный аптайм GetTickCount() может сильно расходиться с тем что показывает
Task Manager в поле "Время работы" — который считает время с последнего resume, а не
с последнего реального сброса счётчика.

**Практически это означает:** баг может проявиться у любого пользователя с Fast Startup
уже через ~25 дней с момента последней полной перезагрузки. Для разработчика — ещё раньше,
т.к. девмашина обычно работает дольше без полной перезагрузки.

## Баг 4: кнопка C — анимация подбора предмета вместо приседания (конфликт `#define INFINITY`)

**Обнаружен:** 2026-03-22. **Коммит появления:** `92d90a1` (миграция dirt.cpp из `old/` в `new/`).
**Git-тэг:** `stage4_bug_C_button_stealth_mode_broken`.

**Симптом:** При нажатии кнопки C стоя на месте Дарси выполняет анимацию поднятия
предмета с пола вместо того чтобы присесть (crouch/crawl).
Воспроизводится **только в Release**, в Debug — работает корректно.

### Как обнаружили причину

Через `git bisect` нашли коммит `92d90a1` — миграция `dirt.cpp` из `old/` в `new/`.
Код функций идентичен оригиналу — логических изменений нет. Добавили отладочные
`PANEL_new_text` в `do_an_action()` (interfac.cpp) — подтвердили что срабатывает
путь подбора банки (coke can): `DIRT_get_nearest_can_or_head_dist()` возвращает
`dist=0` (банка якобы прямо под ногами) хотя банки рядом нет.

Проверили в Debug — баг не воспроизводится. Добавили `MSG_add` внутрь функции —
баг пропал и в Release (вызов функции не даёт оптимизатору агрессивно оптимизировать
цикл). Это классический признак **undefined behavior**.

Через препроцессор (`clang++ -E`) обнаружили что `INFINITY` раскрывается не в
`0x7fffffff` (наше определение из `MFStdLib.h` / `core/macros.h`), а в
`((float)(1e+300))` — определение из `<math.h>` стандартной библиотеки C.

### Причина: конфликт макроса `INFINITY`

Оригинальный код (`MFStdLib.h`) определяет:
```cpp
#define INFINITY 0x7fffffff   // целое число, INT_MAX — используется как "бесконечное расстояние"
```

Стандартная библиотека C (`<math.h>`) тоже определяет `INFINITY`:
```cpp
#define INFINITY ((float)(1e+300))   // float положительная бесконечность
```

В `core/macros.h` есть `#undef INFINITY` + `#define INFINITY 0x7fffffff` — попытка
переопределить стандартный макрос. Но это работает только если `macros.h` включается
ПОСЛЕ `<math.h>`. Если **любой** хедер в цепочке include подтягивает `<math.h>` ПОЗЖЕ —
float-версия затирает нашу.

**Почему проявилось при миграции dirt.cpp:** в `old/` файлах цепочка include другая и
`<math.h>` либо не включается, либо включается до `MFStdLib.h`. При переносе в `new/`
порядок include изменился — `<math.h>` стал включаться через другие хедеры ПОСЛЕ
`core/macros.h`, и float-версия `INFINITY` перезатёрла нашу.

**Результат в коде:**
```cpp
SLONG best_dist = INFINITY;
// препроцессор раскрывает в:
SLONG best_dist = ((float)(1e+300));   // float +inf → SLONG = undefined behavior
```

Присвоение float +infinity в SLONG — **UB** по стандарту C++:
- **Debug:** компилятор генерирует честную конвертацию, x86 инструкция `cvttss2si`
  даёт INT_MIN или насыщение → функция случайно работает корректно
- **Release:** оптимизатор имеет право предполагать что UB не происходит →
  `best_dist` инициализируется в произвольное значение (наблюдалось 0) →
  функция `DIRT_get_nearest_can_or_head_dist()` возвращает 0 →
  `do_an_action()` считает что банка прямо под ногами →
  вызывает `set_person_can_pickup()` вместо `set_person_croutch()`

**Проблема с оптимизатором — это НЕ отдельный баг.** Это следствие UB. После
устранения UB (правильный `#define`) оптимизатор будет работать корректно,
разницы между Debug и Release не будет.

### Масштаб проблемы

`INFINITY` используется в **~100 местах** в `new/`. Все файлы где `<math.h>` попадает
в цепочку include — **потенциально сломаны** в Release (UB при инициализации).

Это **не единственный** возможный конфликт: `MFStdLib.h` определяет и другие макросы
с короткими общими именами которые могут совпадать с именами из стандартной библиотеки
(`NAN`, `TRUE`, `FALSE`, `NULL`, `MIN`, `MAX` и т.п.). → Баг 5.

### Фикс

Переименование `INFINITY` → `UC_INFINITY` (и других конфликтующих макросов) во всей
кодовой базе. Подробный план → Баг 5.

**Статус:** ⚠️ причина найдена и подтверждена, фикс ожидает выполнения Бага 5.

## Баг 5: аудит и переименование `#define` макросов конфликтующих со стандартной библиотекой

**Обнаружен:** 2026-03-22 (при расследовании Бага 4).

**Суть:** Оригинальная кодовая база (MFStdLib и др.) определяет макросы с именами,
совпадающими с именами из стандартной библиотеки C/C++. При компиляции с современным
clang + стандартными хедерами (`<math.h>`, `<cmath>`, `<stdlib.h>`) макросы
перезатираются, вызывая UB и сломанную логику. Проблема проявляется при миграции
файлов из `old/` в `new/`, т.к. в `new/` другой порядок include.

**Подтверждённый конфликт:** `INFINITY` (`MFStdLib.h`: `0x7fffffff` vs `<math.h>`:
`((float)(1e+300))`).
**Нужна проверка:** `NAN`, `TRUE`, `FALSE`, `NULL`, `MIN`, `MAX` и другие короткие
общие имена.

**Решение:** все конфликтующие макросы переименовываются с префиксом `UC_`
(Urban Chaos). Пример: `INFINITY` → `UC_INFINITY`. Префикс `UC_` не используется
нигде в оригинальной кодовой базе (проверено).
